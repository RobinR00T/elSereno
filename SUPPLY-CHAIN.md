# Supply-chain policy

The set of controls that protect ElSereno's build + distribution
chain. Every release artefact MUST satisfy every control here
before the tag is cut; the supply-chain workflow enforces this
automatically.

## SLSA level

Target: **SLSA L3**.

Controls that justify L3:

| Requirement | Evidence |
|-------------|----------|
| Build integrity — hermetic, reproducible | `CGO_ENABLED=0`, `-trimpath`, `-buildvcs=false`, `GOFLAGS=-mod=readonly` in `.goreleaser.yml` |
| Isolated build environment | GitHub-hosted `ubuntu-latest` runner (single-use VM) |
| Provenance exists | `actions/attest-build-provenance@v2` attests every archive + `checksums.txt` with a SLSA v1.0 predicate (v1.2+) |
| Provenance is authenticated | GitHub OIDC identity signs the attestation via Sigstore; no long-lived key |
| Provenance is service-generated | Attestation minted by the Attestations API action in the same job, not from a developer laptop |
| Provenance is non-falsifiable | Sigstore transparency log (Rekor) backs every signature |

Operators verify via GitHub CLI (easiest):

```sh
gh attestation verify elsereno_v1.2.0_linux_amd64.tar.gz \
    --repo RobinR00T/elSereno
```

Or with cosign against the Sigstore transparency log directly:

```sh
cosign verify-attestation \
    --type slsaprovenance1 \
    --certificate-identity-regexp 'https://github.com/RobinR00T/elSereno/.*' \
    --certificate-oidc-issuer     'https://token.actions.githubusercontent.com' \
    elsereno_v1.2.0_linux_amd64.tar.gz
```

**v1.2 change**: dropped the `slsa-framework/slsa-github-generator`
reusable workflow. That workflow's `final` step had a long-standing
exit-27 bug (upstream issue 2610) that forced v1.0.1 and v1.1.0 to
wrap the job in a non-blocking gate. The GitHub Attestations API
path is maintained by GitHub itself, produces the same SLSA v1.0
predicate, and doesn't hit the upstream bug. `slsa-verifier` still
works against the attestation — you just fetch it via `gh
attestation download` instead of it being a release asset.

## Dependency policy

Enforced by `.github/dependency-review-config.yml` + the
supply-chain workflow's `dependency-review` job.

### License allow-list

Implicit allow: **MIT, BSD-2-Clause, BSD-3-Clause, Apache-2.0,
ISC, MPL-2.0, Unlicense**.

Explicit deny:
- GPL family (v2, v3, AGPL, LGPL) — copyleft conflicts with the
  MIT licence we ship under.
- SSPL-1.0, Commons-Clause, Elastic-2.0 — commercial restrictions
  that block pentesting / consultancy use.

Adding a dep under a listed-allow licence requires no action;
anything else fails PR CI and must be replaced or vendored.

### Vulnerability threshold

`dep-review` fails on any advisory at severity `high` or
`critical`. Transitive advisories count. Operators wishing to
merge anyway must either:

1. Wait for the upstream fix.
2. Vendor the dep and patch locally with a rationale recorded in
   an ADR.
3. Swap to an alternative.

The `osv-scanner` job additionally runs against the full tree
nightly; its output uploads as an artefact.

## SBOM distribution

Every release carries a CycloneDX 1.6 SBOM per archive,
generated by `cyclonedx-gomod` inside goreleaser. The local
equivalent (syft against the binary) is documented in
`RELEASING.md`.

Receivers can diff the SBOM between tags to detect a supply-chain
surprise:

```
jq -r '.components[] | "\(.name) \(.version)"' \
  elsereno_0.0.9_sbom.cyclonedx.json | sort > old.txt
jq -r '.components[] | "\(.name) \(.version)"' \
  elsereno_0.1.0_sbom.cyclonedx.json | sort > new.txt
diff old.txt new.txt
```

## Scorecard

[OpenSSF Scorecard](https://securityscorecards.dev) runs nightly
via the supply-chain workflow; results upload to GitHub Security
as SARIF. Target score: **≥ 8.0**. Issues with lower scores
become blocker tickets for 1.0.

## Secrets rotation

| Secret | Owner | Rotation policy |
|--------|-------|-----------------|
| Vault passphrase | operator | operator-driven; no scheduled rotation |
| Canary webhook HMAC key | `creds.Derive(info="elsereno/canary/webhook/v1")` — tied to vault master | rotates when vault re-keyed |
| CSRF key | `creds.Derive(info="elsereno/csrf/v1")` | rotates when vault re-keyed |
| cosign keyless identity | GitHub OIDC; per-build, short-lived | n/a |
| JIRA / GitHub Issues / Shodan / Censys tokens | vault entries | operator rotates via `elsereno creds rotate <name>` |

## Incident response

See `SECURITY.md` for the disclosure pipeline. Supply-chain
incidents (a compromised upstream dep, a leaked OIDC identity,
etc.) trigger an immediate `RELEASING.md` rollback + a public
advisory on the GitHub Security tab.
