# Snapshot — v2.4.0: schedule tags/labels

Single-chunk cycle. Operators with N schedules want to
group them by environment / criticality / owner +
filter the list endpoint accordingly.

## Schema (migration 00016)

```sql
ALTER TABLE scan_schedules
    ADD COLUMN tags TEXT[] NOT NULL DEFAULT ARRAY[]::TEXT[];

CREATE INDEX idx_scan_schedules_tags
    ON scan_schedules USING GIN (tags);
```

GIN index accelerates `WHERE tags && ARRAY[$1]::text[]`
overlap filters. DEFAULT empty-array means existing rows
come up tagless without a backfill.

## Go

`ScanSchedule.Tags []string`. Validation Go-side:

  - 1..10 tags per schedule.
  - Each tag 1..32 chars.
  - Each tag matches `/^[a-z0-9_-]+$/` (lowercase only;
    keeps URL param parsing simple + sidesteps Unicode
    normalisation).
  - Duplicates within a schedule deduped silently.
  - Result sorted (canonical form).

Helpers: `canonicaliseTags(in) ([]string, error)` +
`isValidTag(t string) bool` + constants
`scheduleMaxTags=10`, `scheduleMaxTagLen=32`.

Sentinels: `ErrScheduleInvalidTag`,
`ErrScheduleTooManyTags`.

## REST

  - Create + Update accept `tags []string`.
  - List supports `?tag=<value>` filter (new
    `Store.ListByTag(ctx, tag)` method; Memory + PG impls).
  - PG variant uses the GIN index via
    `tags && ARRAY[$1]::text[]`.
  - Validation errors map: 400 with explicit message.

## OpenAPI

  - `CreateScheduleRequest` / `UpdateScheduleRequest` /
    `ScanSchedule` schemas gain `tags` array with
    `pattern: "^[a-z0-9_-]{1,32}$"` + `maxItems: 10`.
  - List operation description mentions `?tag=` filter.

## Tests (+3)

  - TestCreateSchedule_Tags: input
    `["prod","net-team","prod"]` round-trips to
    `["net-team","prod"]` (deduped + sorted).
  - TestCreateSchedule_TagsInvalidShape: `"Bad Tag"` →
    400.
  - TestListSchedules_ByTag: 3 schedules with mixed
    tags → `?tag=critical` returns only the 1 match.

## Honest scope

  - Tag-shape validation is Go-side only; the DB column
    stays shape-agnostic so a future cycle can relax
    the regex without a migration.
  - No "list distinct tags" endpoint yet (operators
    wanting a tag-autocomplete UI need to compose
    schedule.tags client-side). Future cycle.
  - Filter is single-tag-only (`?tag=critical`).
    Multi-tag AND/OR filters can land later via
    `?tag=a&tag=b` semantics.
  - INSTALL.md unchanged. Build sizes unchanged.
