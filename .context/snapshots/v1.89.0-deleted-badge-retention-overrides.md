# Snapshot — v1.89.0: Deleted badge + per-schedule retention overrides

Single-chunk cycle. Closes two carryover items from v1.87 +
v1.88: dedicated rendering for `event_type='delete'` in the
audit-history dashboard view (previously rendered as a
"every field changed to —" diff that obscured operator
intent), and per-schedule audit-retention overrides so
"critical" schedules can opt into a longer retention window
than the global `--audit-retention-days` setting.

## Schema (migration 00013)

```sql
ALTER TABLE scan_schedules
    ADD COLUMN audit_retention_days INTEGER NULL
        CHECK (audit_retention_days IS NULL OR audit_retention_days >= 0);
```

NULL → inherit global retention. >0 → per-schedule cutoff.
Negative values rejected by CHECK (defence in depth alongside
Go-side `validateAuditRetention`).

## Go model

`ScanSchedule.AuditRetentionDays int` (JSON
`audit_retention_days,omitempty`). 0 maps to NULL on the wire
+ DB. Clamped to `scheduleMaxAuditRetentionDays = 365*10`
(10 years).

New sentinel: `ErrScheduleInvalidAuditRetentionDays`
(< 0 input → 400 in REST).

## Pruner per-schedule logic

`ScheduleAuditStore.PruneWithOverrides(ctx, globalCutoff,
overrides map[string]time.Time) (int64, error)` — new
interface method. Memory + PG impls. PG splits into 2
passes:

  1. Per-schedule DELETE for each (id → cutoff) in overrides.
  2. Global DELETE for everything NOT in overrides (including
     orphaned schedule_id IS NULL rows from v1.88 tombstones).

`AuditPruner` gets optional `ScheduleStore` field. Each tick
walks live schedules → builds overrides map →
PruneWithOverrides. Nil ScheduleStore → falls back to v1.87
PruneOlderThan semantics.

`cmd_serve.buildScanAndSchedule` now passes scheduleStore to
`startAuditPruner` alongside auditStore.

## REST

`CreateScheduleRequest` + `UpdateScheduleRequest` both accept
`audit_retention_days`. `writeScheduleValidationError` maps
the new sentinel to 400. `getSchedule` + `listSchedules`
serialise the field (omitempty when 0).

## Dashboard

  - Schedule form: new `audit days` input (0 = inherit; max
    3650). Plumbed through `submitSchedule` + edit-mode load.
  - Audit history view: `event_type='delete'` now renders a
    red "DELETED" badge in the Event column + a "pre-delete
    snapshot: name / input=X" line in the Changes column
    (sourced from payload_before). Other event types still
    use the v1.85 field-by-field diff.

## Honest scope

  - Pruner reads ScheduleStore.List on every tick. Cheap for
    typical fleets (<100 schedules); a deployment with
    thousands of schedules would want a cache. Out of scope.
  - PG PruneWithOverrides uses `<>ALL($1)` with a string
    array. Postgres handles this fine but for an array of
    100k IDs it'd be worth a temp table. Operational reality
    is <1k.
  - The CHECK constraint allows audit_retention_days = 0
    explicitly (NOT NULL but 0). The Go-side maps 0 → NULL
    so the column never stores literal zero — keeps NULL as
    the single source of truth for "inherit".
  - No backfill of pre-v1.89 schedules' audit_retention_days
    — they default to NULL (inherit) naturally.
  - INSTALL.md unchanged (no CLI/install/feature-matrix
    impact).

## Linter side-quest

surfaced when the audit.yml CI started running golangci-lint
on Linux (the main ci.yml is gated to workflow_dispatch
since the billing event). Surgical excludes added to
`.golangci.yml`:

  - `syscalls_linux.go`: revive (Linux underscore convention)
    + dupl (intentional per-arch maps) + gofmt + goimports.
  - `bpf_linux.go`: gosec G103/G115 (BPF imm-field) + goconst.
  - `bpf_linux_test.go`: gosec + errorlint.
  - `sandbox_integration_test.go`: gosec G702 + noctx
    (test re-exec via exec.Command).

These were always bug-shaped but invisible because no CI
gate exercised them. v1.89 doesn't introduce them.
