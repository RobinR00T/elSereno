# Snapshot — v1.63.0: scan_state_change SSE event

Single-chunk cycle. Continues the dashboard scan-
orchestration feature opened in v1.58 and extended
through v1.62 (panel + 2-tier polling). v1.63 closes
the polling latency gap: the dashboard now sees state
transitions in real time via SSE instead of waiting
for the next poll tick.

## Why a separate cycle

The v1.62 polling timer is fine for "is anything
running?" but bad for the user-facing "I just clicked
Cancel" moment — operators stare at the row for up to
2s before it flips to cancelled. SSE drops that
latency to ~10ms. The two-tier polling timer stays as
a safety net for SSE reconnect gaps.

## Architecture

```
operator clicks Cancel ─── POST /scans/{id}/cancel ──┐
                                                     ↓
                                   BroadcastingStore (NEW)
                                   ↓                ↓
                      inner store call         publish on b
                      (Memory or DB)               ↓
                                           SSE EventScanState
                                                   ↓
                                        Subscribers (browsers)
                                                   ↓
                                        renderScans() in JS
```

Worker drives the same path:

```
Worker.Process(jobID)
  ↓
  Store.Transition(running)  → BroadcastingStore publishes
  ↓
  runner.Run
  ↓
  Store.Transition(completed) → BroadcastingStore publishes
```

Same wrapper handles both call sites — the REST
handler's transitions and the worker's transitions
both flow through one Publish surface.

## Composition rationale

The decorator pattern keeps scanorch package free of
stream coupling. scanorch.MemoryStore + scanorch.DBStore
are unchanged; they don't reach for the broadcaster.
cmd_serve composes:

```
broadcaster := stream.New(128)
inner := scanorch.NewDBStore(pool)
wrapped := stream.NewBroadcastingStore(inner, broadcaster)
worker := &scanorch.Worker{Store: wrapped, ...}
srvOpts.ScanStore = wrapped  // same wrapper to REST
srvOpts.Broadcaster = broadcaster // same broadcaster to /api/v1/stream
```

This preserves the chunk-by-chunk separability the
v1.58/59/60/61 cycles built up: each layer can be
swapped or tested independently.

## Honest scope

  - **No SSE event throttling** for high-frequency
    transitions. A pathological worker that flips
    states 1000 times/second would flood the SSE
    bus. Not a real concern at v1.63 (state
    transitions are operator-paced or scanner-
    duration paced); a future cycle could add a
    debouncer if needed.
  - **No "stats progress" mid-scan event.** The
    state-change event fires on transitions only;
    in-progress Stats updates would need a new
    event kind + a Worker hook to push them. v1.64+.
  - **Reconnect "since-id" replay** not added in
    this chunk. The Broadcaster has the nextID
    counter ready (see broadcaster.go), but
    EventSource clients can't ask "give me events
    after ID 42" without a small server-side ring
    buffer. The polling timer covers this gap.
