# Snapshot — v1.73.0: cron expressions as alternative cadence

Single-chunk cycle. Adds `CronExpr` to `ScanSchedule`
alongside `IntervalSeconds`. Operators choose interval
("every N seconds") or cron ("0 9 * * 1-5") per
schedule; the two are mutually exclusive.

## Why both modes

v1.70-v1.72 shipped interval-only because continuous
security scanning wants cadence, not wall-clock anchors.
The follow-up "0 2 * * *" use-case (daily-at-02:00
maintenance window) doesn't fit the interval model
cleanly — interval=86400 fires roughly daily but drifts
relative to the wall clock. Cron expressions anchor the
fire time. Both are useful; this cycle keeps both.

## Cron parser

`internal/scanorch/cron.go` (~270 LoC). Custom impl
rather than a dep — the supported syntax is small
enough that an external library would be overkill.

```
*           every value (full mask)
N           single numeric
N,M,...     comma list
N-M         inclusive range
*/S         step from minimum
N-M/S       stepped range
```

Implementation: each field becomes a 64-bit bitmask of
allowed values. `Match(t)` tests `bit(field-of-t) & mask
!= 0` per field; `Next(after)` walks minute-by-minute
from `after+1m` up to 366 days. Walk caps at 1 year so
impossible expressions ("0 0 30 2 *" — Feb 30 doesn't
exist) return `ErrCronNoMatch` instead of spinning
forever.

DOM + DOW use Unix-cron **OR semantics** when both are
restricted: "0 0 1 * 1" matches every 1st OR every
Monday. When one is `*` the other dominates.

Not supported (deferred): `@daily` / `@hourly` shortcuts;
JAN..DEC / SUN..SAT names; `last-of-month`. Operators
write canonical 5-field expressions.

## ScanSchedule contract change

```go
type ScanSchedule struct {
    ...
    IntervalSeconds int    `json:",omitempty"`
    CronExpr        string `json:",omitempty"`  // NEW
}
```

Exactly one is set. `CreateScheduleRequest` validation
order:
  1. Name non-empty.
  2. Template input non-empty.
  3. Cadence specified (exactly one of interval/cron).
  4. Cron parses (only if hasCron — defer until cadence
     settled to avoid noisy errors).

Two new sentinels: `ErrScheduleCadenceRequired`,
`ErrScheduleCadenceConflict`.

`buildScheduleFromRequest` helper extracted so
MemoryScheduleStore + DBScheduleStore share the same
validation. Single source of truth across both stores.

## IsDue routing

```go
func (s ScanSchedule) IsDue(now time.Time) bool {
    if !s.Enabled { return false }
    if s.CronExpr != "" { return s.cronIsDue(now) }
    // interval path unchanged from v1.70
}
```

Cron path: `CronExpr.Next(anchor) <= now`. Anchor is
`LastFiredAt` (or `CreatedAt` if never fired). A cron
parse error here returns `false` (defence in depth —
should have been caught at Create).

**Never-fired difference**: interval-based fires
eagerly on the next tick; cron-based waits for the
next-match-after-CreatedAt. Operator creating
`"0 2 * * *"` at 13:00 doesn't get an immediate
fire — they wait for tomorrow 02:00.

## DB migration 00008

```sql
ALTER TABLE scan_schedules
    ADD COLUMN cron_expr TEXT NOT NULL DEFAULT '';
```

Drops the v1.71 `CHECK (interval_seconds BETWEEN 60
AND 604800)` constraint by walking `pg_constraint` to
find the existing CHECK on the column (PG doesn't
expose a stable name across versions). Adds a
replacement combined CHECK:

```sql
CHECK (
    (interval_seconds = 0 AND cron_expr <> '')
    OR (interval_seconds BETWEEN 60 AND 604800 AND cron_expr = '')
)
```

Pre-v1.73 rows satisfy without data migration:
interval is in `[60, 604800]`; cron_expr defaults to
`''` from the new column DEFAULT.

## Dashboard form

Cadence-mode dropdown toggles interval/cron inputs:

```
[ name ] [ input ] [ plugin(s) ]
[ cadence: interval ▼ ]
  [ interval (s) 3600 ]   ← visible when interval
  [ cron 0 2 * * * ]      ← visible when cron
[ Create ]
```

`onScheduleCadenceModeChange` shows/hides the
relevant input. Submit reads only the visible one,
so the backend cadence-XOR check is never tripped
from the UI happy path.

`renderSchedules` shows `"cron: 0 2 * * *"` in the
Interval column for cron schedules; interval-based
keep `humanInterval` rendering ("6h", "1d").

## Honest scope

  - **No named shortcuts** (`@daily`, `@hourly`).
    Trivially layer-able via a pre-process pass on
    the input string before ParseCron; deferred.
  - **No named months/days** (JAN, MON). Same — small
    pre-process expansion. Deferred.
  - **No timezone-aware cron**. Expressions evaluate
    in UTC (the Scheduler's `time.Now().UTC()` clock).
    Operators in non-UTC timezones write expressions
    relative to UTC. Future cycle could add a
    per-schedule timezone field.
  - **No DST handling**. Tied to UTC-only — DST is
    a wall-clock concept and the Scheduler never
    sees wall-clock-local time. This is correct
    behaviour for security-monitoring schedules but
    documented for clarity.
  - **No "next fire" preview** in the dashboard.
    The form accepts a cron expression but doesn't
    show "next fire: 2026-05-09 02:00 UTC" as
    feedback. Could add via a `?dry=1` query param
    on Create returning the parsed expression's
    Next(now); deferred.
