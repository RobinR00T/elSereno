# Snapshot — v1.61.0: real scan runner + serve --scan-store

Single-chunk cycle. Connects the scan-orchestration
shell (v1.58) + worker (v1.59) + DB store (v1.60) to
the existing scanner + plugin registry. Submitted
jobs now execute when `serve` is started with
`--scan-store=memory` or `--scan-store=db`. The
orchestration cycle (v1.58 → v1.61) is structurally
complete: an operator can curl POST /api/v1/scans/,
the worker picks the job up, scans, and the result
is persisted (in DB mode) or in-memory (in memory
mode).

## Why this needed its own cycle

v1.58 → v1.60 produced a complete contract + storage
+ worker stack but the runner was deliberately left
abstract (JobRunner interface) so the chunk
boundaries stayed inside `internal/scanorch/` (a
package that doesn't import `internal/scanner` or
`internal/inputs`). v1.61 lives in `cmd/elsereno`
where the imports already exist:

  - `internal/inputs/{list,nmapxml,stdin}` for
    Job.Input parsing
  - `internal/core.RegisteredPlugins` for plugin
    lookup
  - `internal/scanner` for the actual probe dispatch

Single-plugin contract for v1.61: each Job names
exactly one plugin in Plugins. Multi-plugin parallel
dispatch (one job → N plugins) is deferred; keeps
the Stats accounting unambiguous and the cycle scope
tight.

## Architecture (now complete)

```
Dashboard ──── POST /api/v1/scans (v1.58) ─── Submit() ──┐
       ↑                                                 ↓
       │                                              [Store]
       │                                               ↑   ↑
       │                              memory (v1.58) ──┘   └── db (v1.60)
       │                                                        ↑
       │                                                        DBStore atomic
       │                                                        UPDATE-RETURNING
       │
       │           Worker pool (v1.59) ←── claims queued jobs
       │                ↓
       │     defaultScanRunner (v1.61, NEW)
       │                ↓
       │       parseInput → core.RegisteredPlugins → scanner.Run
       │                ↓
       │           drainScanRun → Stats
       │                ↓
       │        Worker.terminate
       └─────── State transition (running → completed | failed)
```

## Chunks

### Chunk 1 — runner + flag wiring (`08238d1`)

`cmd/elsereno/scan_runner.go` (new, ~100 lines):

  - **defaultScanRunner** struct with concurrency knob.
  - **Run(ctx, job)** validates exactly-one-plugin,
    looks up the plugin, parses input via the shared
    parseInput dispatcher, builds a `scanner.Scanner`
    with `MaxConcurrentTargets` from the runner's
    knob (default 100), runs the plugin's Probe.
  - **drainScanRun** consumes findings + errs to
    completion, accumulating
    `TargetsScanned` (every target that produced a
    finding-or-error) and `FindingsCount` (findings
    only).
  - **lookupPluginByName** walks the registry with
    a (Plugin, ok) shape — local helper since
    cmd_fingerprint.go's similar helper has a
    (Plugin, error) shape.
  - 3 sentinels: ErrRunnerNoPlugin,
    ErrRunnerTooManyPlugins, ErrRunnerUnknownPlugin.

`cmd/elsereno/cmd_serve.go` (modified):

  - **--scan-store {off|memory|db}** flag (default off).
  - **--scan-pool N** flag (default 2, clamped [1, 64]).
  - **buildScanOrchestrator** helper picks the Store
    + Worker Pool based on --scan-store. db mode
    requires a non-nil pgxpool.Pool from
    DATABASE_URL.
  - **buildWebOptions** + **startAuditTail** extracted
    so runServe stays under the gocyclo 15-branch
    ceiling.

`internal/web/server.go` (modified):

  - web.Options gains ScanStore field threaded
    through to APIV1Deps.

## Operator-facing contract

Memory mode (dev):

```sh
elsereno serve --scan-store memory --scan-pool 4
```

DB mode (production, jobs survive restart):

```sh
DATABASE_URL=postgres://elsereno:****@127.0.0.1:5432/elsereno \
  elsereno db migrate          # apply 00005_scan_jobs
elsereno serve --scan-store db --scan-pool 4
```

Submit:

```sh
curl -X POST http://127.0.0.1:8787/api/v1/scans \
  -H "Content-Type: application/json" \
  -H "X-Operator: alice" \
  -d '{"input":"list:targets.txt","plugins":["modbus"],"default_port":502}'
```

`db` mode: two `serve` instances pointing at the
same DB safely race on the same job queue. Atomic
UPDATE-RETURNING (v1.60) guarantees at most one
worker claims each job — no advisory locks needed.

## Honest scope

  - **Multi-plugin per job** NOT in this chunk. A
    Job names exactly one plugin. Multi-plugin
    parallel dispatch (one Job → N plugins per
    target) needs careful Stats semantics — defer
    until field requests it.
  - **Live progress reporting** NOT in this chunk.
    Run returns final Stats; mid-scan polling shows
    the queued / running state but Stats stays at
    zero until the job completes. A future chunk
    can add a `progress chan<- Stats` arg or a
    callback that the Worker forwards into a
    Store.UpdateStats helper.
  - **Auth scoping**: scan-trigger sits behind
    whatever auth middleware the operator has wired
    upstream of /api/v1/. A separate "scan-trigger
    scope" (so a viewer-only operator can list but
    not submit) is deferred.
  - **Cancel-to-runner-stop signal** still soft.
    POST /scans/{id}/cancel transitions state but
    doesn't reach into the running goroutine; the
    runner's ctx propagation via the worker is the
    only path.
