# Snapshot — v1.69.0: bulk submit

Single-chunk cycle. Adds POST /api/v1/scans/bulk +
a dashboard textarea panel. Operators submit many
jobs in one call instead of looping the
single-submit endpoint.

## Why

The v1.61 single-submit endpoint requires one HTTP
round-trip per job. For "scan these 50 /24s" the
operator either loops curl in a shell or scripts
50 dashboard form submits. Bulk submit collapses
that to one request — same plugins, same default
port, many inputs.

## API contract

```
POST /api/v1/scans/bulk
Content-Type: application/json
X-Operator: alice
{
  "inputs":       ["list:t1.txt", "list:t2.txt", ...],
  "plugins":      ["modbus", "s7"],   // optional, shared
  "default_port": 502                 // optional, shared
}
```

Response always 202 if syntactically valid:

```json
{
  "schema": "api:v1",
  "data": {
    "submitted": [<Job>, <Job>, ...],
    "errors":    [{"index": 1, "input": "", "error": "..."}]
  }
}
```

Per-input failures don't fail the batch. The
client correlates errors by `index` (position in
the original `inputs` array).

## Limit + rationale

Capped at 200 inputs per request. Sized to:

  - Fit in one operator's mental model ("paste a
    list of /24s" — typical lists are 5-50).
  - Not hold the goroutine pool open arbitrarily
    long (200 sequential Submits to the
    MemoryStore is fast; against DBStore each
    INSERT is ~1ms).
  - Force operators with thousands of inputs to
    batch — 200 is enough granularity that even
    10k inputs is 50 calls, and a single
    misbehaving operator can't lock the worker.

## Dashboard panel

Toggle button "Bulk…" next to the Submit button
reveals a panel:

  - Textarea (one input per line, monospace).
  - The plugin + default-port fields above are
    SHARED — operator sets once, applies to every
    line.
  - "Bulk submit" button POSTs.
  - Status line shows "queued N, failed M".

UX choices:

  - On full success, textarea is cleared (operator
    can paste a fresh batch).
  - On partial failure (errors > 0), textarea is
    preserved so the operator edits + resubmits
    without re-pasting the whole list.
  - The toggle is sticky for the page session —
    operators who like bulk leave it open.

## Honest scope

  - **Atomic batch is NOT supported.** A partial
    failure leaves some Submitted; the client
    decides whether to resubmit only the failed
    ones or accept the partial state. Bulk
    transactions across MemoryStore / DBStore
    aren't worth the implementation complexity for
    a feature whose typical input is "paste a
    list".
  - **No bulk cancel.** Cancelling many jobs is
    one DELETE-equivalent per job. A bulk-cancel
    endpoint could mirror this; deferred.
  - **No CSV / file upload.** The textarea is
    paste-or-type only. A file-upload endpoint
    that consumed a list-file would compose nicely
    with the existing list:`<path>` input form;
    deferred.
