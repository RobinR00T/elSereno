# Snapshot — v2.8.0: schedule CLI mutating verbs

Single-chunk cycle. Closes v2.3 carryover. v2.3 shipped the
read-only subset of `elsereno schedule`; v2.8 completes the
verb tree with mutating commands.

## New verbs

```
elsereno schedule enable <id>
elsereno schedule disable <id>
elsereno schedule clone <id> [--name foo]
elsereno schedule import <file> [--on-conflict skip|overwrite|rename]
elsereno schedule pause-all
elsereno schedule resume-all
```

All honour the global `--dry-run` flag — print the intended
action without firing the HTTP call.

## httpDo extension

The v2.3 `httpDo(opts, method, path)` was body-less. v2.8
adds `httpDoWithBody(opts, method, path, body, contentType)`:

  - When `body` non-nil, sets Content-Type
    (`application/json` default).
  - Returns the same response body / error shape as httpDo.
  - httpDo is now a thin wrapper that passes nil body.

`elsereno schedule import` uses file-extension to pick
Content-Type:
  - `.json` → `application/json`
  - `.ndjson` → `application/x-ndjson`
  - other → `application/x-ndjson` (server auto-detects via
    the first non-whitespace byte; v1.99 added the fallback).

## Tests

Smoke-tested via `elsereno schedule --help` showing all
11 verbs registered. No unit tests added — the CLI is a
thin transport over already-tested REST endpoints; an
httptest-server-based suite would land in a future cycle.

## Honest scope

  - No `create` or `update` verbs from the CLI yet — those
    require a JSON-body composer with cadence + tags +
    audit_retention helper. Operators with that workflow
    use `import` with a single-line NDJSON file, or curl.
  - No multi-target verbs (`elsereno schedule enable a b c`).
    The user iterates in shell; bulk-enable/disable cover
    the common pattern.
  - INSTALL.md unchanged.

## Build sizes

Unchanged across all 3 variants.

Adds ~180 LoC in cmd_schedule.go.
