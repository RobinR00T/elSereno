# Snapshot — v1.15.0: CWMP polish + discover + STIX + flock + SIGHUP

Closed **2026-04-26**. Five-chunk cycle covering loose-end
closure on multiple surfaces: CWMP firmware-pin observability,
operator-UX scanning, threat-intel interop, audit concurrency
hardening, and supervisor-driven reload.

## Why this cycle matters

v1.13 closed every BACnet mutating service. v1.14 lifted IPv6
across the codebase. v1.15 collects the remaining loose ends:

- CWMP chunk 10's firmware allowlist had no observation hook
  for the CPE → ACS TransferComplete report — operators
  authorised a Download but had no audit trail of the
  outcome. Chunk 1 closes that.
- Operators wanting "point at a CIDR, get a port-listening
  inventory" had to build the port list manually. Chunk 2
  ships `discover --auto`.
- Findings couldn't flow into MISP / OpenCTI / ThreatBus
  without a STIX 2.1 export. Chunk 3 ships the sink.
- Two ElSereno processes appending to the same audit log
  could break the chain. Chunk 4 fixes the race with flock.
- Reloading the proxy listen allow-file required restarting
  the process manually. Chunk 5 wires SIGHUP to a
  supervisor-restart-style exit.

## Per-chunk delta

### 1 `476b404` — CWMP TransferComplete observer

New `OnTransferComplete` callback on `WriteGatedHandler` fires
when CPE → ACS TransferComplete envelopes traverse the gate.
Default CLI observer emits structured stderr log lines
(`level=info msg=cwmp_transfer_complete target=… status=ok|fault
command_key=… fault_code=… fault_string=… start=… complete=…`).

The library wire parser uses the streaming xml.Decoder pattern
shared with extractDownloadURL + extractParameterNames — no
full SOAP tree materialisation. Tolerates older CPEs missing
the CommandKey. Strict FaultCode "0" → success.

What it ISN'T: byte-level SHA-256 verification. TR-069 doesn't
carry a hash in TransferComplete. The observer surfaces the
CPE's own success/failure report; correlation with the
operator's pinned `--firmware url=…;sha256=…` happens via the
audit chain (CommandKey field).

6 tests: success path, fault path, non-TC RPCs not invoked,
nil observer no-op, missing CommandKey tolerance, IsSuccess()
semantics.

### 2 `389ff5d` — `elsereno discover --auto <CIDR>`

Operator-UX win: skip the manual port-list maintenance step.
Iterates the CIDR (capped by `--max-hosts`, default 256),
probes the well-known port of every registered plugin with
parallel TCP-connect (default 64 concurrent), emits
responsive `(host, port)` pairs as NDJSON or `host:port` list
to stdout — pipe-friendly with `scan --input list:-`.

Port-registry collisions (future IEC 61850 MMS sharing 102
with S7) emit ALL claiming plugins in `protocol_hints`. CIDR
expansion is netip-native + IPv6-aware (chunk-4-of-v1.14
contract).

9 tests: CIDR expansion (v4 /30, v6 /126, max-hosts cap,
malformed input), port-registry helpers, sweep e2e
(responding port detected, dead port ignored), output format
dispatch.

### 3 `e205cd8` — STIX 2.1 export sink

Findings can flow into threat-intel platforms (MISP, OpenCTI,
ThreatBus) via `scan --output-format stix`. Each finding maps
to three STIX 2.1 objects:

- ipv4-addr or ipv6-addr SCO (target's address; selected per
  `netip.Addr` family).
- network-traffic SCO (target's port + protocols list, with
  BACnet/IAX2 → "udp" transport).
- observed-data SDO (timestamps + severity/protocol labels,
  references the network-traffic SCO via object_refs).

Deterministic UUIDv5 IDs keyed on `finding.ID` + ElSereno-
private namespace UUID — same input → byte-identical inner
objects (the bundle id itself is timestamp-bound).

9 tests: bundle layout, IPv4/IPv6 SCO selection, protocol
transport mapping, label population, deterministic IDs,
empty-addr fallback, missing-ID error path, spec_version
conformance.

### 4 `dd92a39` — Audit chain cross-process merge via flock

The race: two ElSereno processes (e.g. `serve` + `proxy listen`
on the same operator workstation) both opening
`~/.elsereno/audit.jsonl` and appending concurrently could
produce two entries claiming the same prev_hash — the chain
invariant breaks the moment the operator runs `audit verify-
file`.

Fix: an exclusive `unix.Flock(LOCK_EX)` guards the
`Append`/`appendVerbatim` read-then-write critical section.
Inside the lock, `resume()` re-reads the file from offset 0
to pick up entries another process may have appended,
advancing nextID + prevHash so our entry continues the merged
chain.

Linux + macOS only via `golang.org/x/sys/unix`; Windows stub
returns nil from both lock/unlock methods (Windows is a
v1.15+ cross-cutting cycle including AppContainer / Job
Objects, not just flock).

2 tests: 50-entry concurrent interleave (asserts strictly
increasing IDs + chain invariant + both actors represented),
sequential appends (proves unlock fires correctly).

### 5 `1264998` — SIGHUP reload-style graceful exit

The constraint: each per-session confirm-token is computed
from the allowlist hash, so any in-process allow-file reload
invalidates the existing token. Rather than building a
token-generation cookie scheme + re-authorise-in-place
plumbing, chunk 5 takes the supervisor-restart path:

- SIGHUP → exit 75 / EX_TEMPFAIL (distinguishable from a real
  crash via systemd's `RestartPreventExitStatus=`).
- SIGINT/SIGTERM → exit 0 (clean-stop behaviour preserved).

Operator workflow:

1. Edit `/etc/elsereno/<plugin>-gate.yaml`.
2. Run `elsereno-offensive write <plugin> dry-run --allow-file
   ... --vault-passphrase-file ...` to mint a fresh
   confirm-token.
3. Write the new token to the supervisor's environment file.
4. `kill -HUP $(pidof elsereno-offensive)` — supervisor
   restarts with updated config.

1 test pinning the errReloadRequested sentinel contract.
