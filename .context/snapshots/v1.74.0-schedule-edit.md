# Snapshot — v1.74.0: schedule edit form

Single-chunk cycle. Schedules were immutable in v1.70-
v1.73 (delete + recreate to change). v1.74 adds a real
edit path: a new ScheduleStore.Update method, PUT REST
endpoint, and a dashboard edit-mode toggle.

## Why

Operator feedback gap: changing a schedule's interval
or template required Delete + Create + reapply. The
Delete loses the LastFiredAt history; the Create
churns the ID. Neither is severe, but it's friction
the operator hits regularly.

## Contract change

```go
type UpdateScheduleRequest struct {
    Name            string        `json:"name"`
    Template        SubmitRequest `json:"template"`
    IntervalSeconds int           `json:"interval_seconds,omitempty"`
    CronExpr        string        `json:"cron_expr,omitempty"`
}

ScheduleStore.Update(ctx, id, req) (ScanSchedule, error)
PUT /api/v1/schedules/{id}
```

Same shape as CreateScheduleRequest minus `operator` —
operator is immutable. Validation rules mirror Create:
name + template.input non-empty; exactly one cadence;
cron parses if specified.

**Identity-preserving fields**: ID, CreatedAt,
LastFiredAt, Operator, Enabled. None of these change
across an Update. Operators who want to "reset"
LastFiredAt or change the operator must Delete +
Create.

## Validation refactor

Extracted from `buildScheduleFromRequest`:

  - `validateScheduleFields(name, template, interval,
    cron)` — shared rules.
  - `applyCadence(s *ScanSchedule, interval, cron)` —
    resets both cadence fields and sets the chosen
    one.

The "reset both" logic is critical for Update: when
the operator switches modes (interval → cron or
vice versa), the previous-mode field must clear,
otherwise the persisted row violates the cadence
XOR. Tests cover this explicitly
(Update_Happy + Update_ValidationMirrors_Create).

## DBScheduleStore.Update

Single UPDATE ... RETURNING:

```sql
UPDATE scan_schedules
SET name = $2, template_input = $3, template_plugins = $4,
    template_default_port = $5, interval_seconds = $6,
    cron_expr = $7
WHERE id = $1
RETURNING <scheduleColumns>
```

The migration-00008 CHECK XOR catches cadence
violations atomically. 0-row response →
ErrScheduleNotFound (consistent with Delete +
MarkFired).

## Dashboard edit mode

Per-row Edit button populates the create form with
the schedule's current values (fetched via GET) +
flips submit button label from "Create" to "Update"
+ reveals a Cancel button. The form scrolls into
view + focuses the name field so the operator can
immediately edit.

cancelEditSchedule resets form fields + button labels
back to create mode.

submitSchedule branches:
  - edit mode → PUT /schedules/{id}
  - create mode → POST /schedules (as before)

editingScheduleID module-scoped JS variable tracks
state across button clicks.

## Honest scope

  - **No "preview next fire"** in edit mode. The
    form accepts a cron expression but doesn't
    show "next fire: 2026-05-10 02:00 UTC" before
    submit. Same gap as Create; deferred to v1.77+.
  - **No optimistic locking**. Two concurrent
    edits to the same schedule race; the last
    write wins. Acceptable for typical
    single-operator deployments. A future cycle
    could add an `updated_at` column + If-Match
    header for multi-operator setups.
  - **No bulk edit**. One at a time. Same
    rationale as bulk delete (deferred).
