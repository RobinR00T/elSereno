# Snapshot — v2.10.0: source_schedule_id provenance column

Single-chunk cycle. Closes v2.1 carryover. v2.1's
`cloned_from` audit event captured clone provenance but
answering "show me everything cloned FROM X" required
scanning the audit log + grepping payload_before.id. v2.10
adds a direct FK column to scan_schedules.

## Schema (migration 00017)

```sql
ALTER TABLE scan_schedules
    ADD COLUMN source_schedule_id TEXT NULL
        REFERENCES scan_schedules(id) ON DELETE SET NULL;

CREATE INDEX idx_scan_schedules_source
    ON scan_schedules (source_schedule_id)
    WHERE source_schedule_id IS NOT NULL;
```

Partial index — clones are a minority of rows in typical
fleets; keep the v2.5 list query cost zero for non-clones.
ON DELETE SET NULL preserves the clone row when the source
is deleted (matches v1.88/v1.92 patterns).

## Go

`ScanSchedule.SourceScheduleID string` (JSON
`source_schedule_id,omitempty`). Two new Store methods:

  - `CreateClone(ctx, req, op, sourceID)` — stamps the
    source FK at insert time.
  - `ListClonesOf(ctx, sourceID)` — fast lookup via the
    partial index.

`Create` is now a thin wrapper over `CreateClone` with
empty sourceID. Memory + PG impls.

## REST

  - `cloneSchedule` handler now calls `CreateClone(...,
    source.ID)` instead of `Create(...)`.
  - New endpoint `GET /api/v1/schedules/{id}/clones`
    returns the schedules whose source_schedule_id = id.
    404 when the source id doesn't exist (guards against
    typo-queries returning empty + confusing operators).

## Tests (+2)

  - TestListScheduleClones_AfterClone: 2 clones via POST →
    /clones returns both, each carrying
    SourceScheduleID = source.ID.
  - TestListScheduleClones_None: never-cloned schedule →
    empty array, 200.

## fakeRows extension

The scan_schedules projection grew 15 → 16 cols. The
fakeRows switch now disambiguates 16-cols as
scan_schedules v2.10 (jobs table is still 15 max from
v1.92). makeScheduleRow stub includes
`source_schedule_id: (*string)(nil)` for default-NULL
rows.

## Honest scope

  - No back-fill: clones predating v2.10 have NULL here
    (the v2.1 audit log still records their provenance
    via `cloned_from` events).
  - No multi-source provenance (a schedule has at most
    one source; chain queries walk the FK manually). The
    audit log is the canonical history.
  - INSTALL.md unchanged. Build sizes unchanged.
