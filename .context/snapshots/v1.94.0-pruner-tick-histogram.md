# Snapshot — v1.94.0: pruner tick-duration histogram

Single-chunk cycle. Closes v1.91 carryover: detect "the
audit table grew enough that pruner ticks are slowing"
before it becomes a deployment incident.

## Metric

`elsereno_audit_pruner_tick_duration_seconds` — histogram.
Buckets sized for realistic prune workloads:

  - 1ms → empty/no-op prune (lock-skip, nothing to delete).
  - 10ms → typical daily prune on a small fleet.
  - 100ms → a few thousand events.
  - 1s → ~100k events.
  - 10s → catch-up after long downtime.
  - 60s+ → schema/index degradation; alert.

## Wire-up

`AuditPruner.OnTick func(duration time.Duration)` — new
optional callback. Fires on every tick after OnPrune /
OnError / OnLockSkipped. Measured via `time.Since(start)`
using the real wall-clock (not the test-seam Now fixture)
so observations stay honest even when tests time-travel.

`cmd_serve` wires OnTick to
`telemetry.Global().AuditPrunerTickDurationSeconds.Observe`.

## Tests (+1, parametric)

`TestAuditPruner_OnTick_FiresOnEveryOutcome` covers all 3
outcome branches (happy / error / lock-skip) with a
shared parametric driver. Asserts OnTick fires exactly
once per Tick with a non-negative duration.

## Honest scope

  - Histogram is unlabelled — the result label is on
    runs_total. A labelled histogram would give p99 per-
    outcome but multiplies series count without much
    operational lift; the unlabelled blend is the right
    default.
  - Buckets chosen pragmatically. Operators with very
    fast or very slow workloads can tune via a future
    config knob (deferred).
  - INSTALL.md unchanged. Build sizes unchanged.

## Build sizes

Unchanged across all 3 variants.

Adds ~40 LoC + 1 test.
