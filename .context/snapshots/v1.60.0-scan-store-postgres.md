# Snapshot — v1.60.0: postgres-backed scan-job Store

Single-chunk cycle. Continues the dashboard scan-
orchestration feature opened in v1.58 (shell) and
extended in v1.59 (Worker). v1.60 lands the
**persistent storage** so scan jobs survive a
`serve` restart.

## Why a separate cycle

The v1.58/59 in-memory MemoryStore was deliberate:
ship the contract + execution machinery first, let
operators exercise the API surface, then add the
schema knowing the contract is stable. Splitting it
into v1.60 means:

  - **MemoryStore stays available for tests + dev**
    — no DB needed for the unit tests in v1.58/59
    cycles; same goes for any future cycle that
    wants to test orchestration without a Postgres.
  - **Schema review is decoupled**: the migration
    file can be reviewed against the v1.58 model
    without code-review noise.

## What landed

### Migration: 00005_scan_jobs.sql

```sql
CREATE TABLE scan_jobs (
    id              TEXT        PRIMARY KEY,
    state           TEXT        NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL,
    started_at      TIMESTAMPTZ,
    finished_at     TIMESTAMPTZ,
    input           TEXT        NOT NULL,
    plugins         TEXT[]      NOT NULL DEFAULT ARRAY[]::TEXT[],
    default_port    INTEGER     NOT NULL DEFAULT 0,
    targets_seen    BIGINT      NOT NULL DEFAULT 0,
    targets_scanned BIGINT      NOT NULL DEFAULT 0,
    findings_count  BIGINT      NOT NULL DEFAULT 0,
    error_msg       TEXT        NOT NULL DEFAULT '',
    operator        TEXT        NOT NULL DEFAULT '',
    CHECK (state IN ('queued','running','completed',
                     'failed','cancelled'))
);

CREATE INDEX idx_scan_jobs_created_at
    ON scan_jobs(created_at DESC);
CREATE INDEX idx_scan_jobs_state
    ON scan_jobs(state);
```

**Schema-side state CHECK** is the key safety
property: the Go State enum and the SQL enum are
synced. Any out-of-band INSERT (psql / data import)
can't introduce a bogus state and corrupt the
worker's state-machine assumptions. Source of
truth: SQL DDL (matches PITF-030).

**Indices**: created_at DESC supports the dashboard
List(); state index supports the worker drain
`WHERE state = 'queued'` query.

### DBStore (internal/scanorch/store_pg.go)

  - **Querier interface**: narrow pgx surface (Query
    + QueryRow + Exec). Mirrors repo.Querier shape
    but kept local to scanorch — no
    internal/scanorch → internal/repo dependency.
  - **DBStore.Submit**: parameterised INSERT with a
    just-generated 16-hex ID + UTC microsecond
    truncated created_at.
  - **DBStore.Get / List**: SELECT against the
    shared jobColumns projection. List clamps
    limit to [1, 1000] (chunk-2 dashboards may
    surface large ranges).
  - **DBStore.Transition** is **atomic**:
    `UPDATE scan_jobs SET ... WHERE id=$1 AND state
    IN (valid_from_states) RETURNING *`. Two workers
    racing for the same queued job: at most one
    wins the UPDATE; the loser sees 0 rows and the
    classifier returns ErrInvalidTransition. No
    advisory locks needed — the table CHECK + row-
    level locking that UPDATE acquires are
    sufficient.
  - **0-rows-affected disambiguator**: when the
    UPDATE returns 0 rows, a follow-up SELECT
    distinguishes "job missing" (ErrJobNotFound)
    from "wrong current state"
    (ErrInvalidTransition) so the REST handler
    surfaces the right HTTP status (404 vs 409).
  - **Compile-time guard**:
    `var _ Store = (*DBStore)(nil)` keeps the
    interface contract honest.

### transitionFromStates table (truth source)

```
StateRunning   ← queued
StateCompleted ← running
StateFailed    ← running
StateCancelled ← queued, running
```

Mirrors validTransitions in job.go but inverted
(target → valid sources) for the SQL UPDATE
WHERE-IN clause.

## Atomicity property

The transition guarantee: **for any job, at any
moment, at most one Transition() call will return a
non-error result for a given (id, to-state) pair**.
Two concurrent calls either:

  - Both target the same to-state from the same
    valid from-state → exactly one's UPDATE matches
    the WHERE IN, the other's matches 0 rows.
  - Target different to-states with overlapping
    from-states → exactly one's UPDATE moves the
    row out of the from-state set, the other's
    matches 0 rows.

This is the property the worker depends on for
"no double-claim" — the v1.59 Worker.Process
contract is preserved verbatim.

## Honest scope

  - **Integration test against dev-db** NOT in this
    chunk. The fakeQuerier covers the SQL shape +
    parameter binding contract; a real-Postgres
    integration test that exercises the CHECK
    constraint + the UPDATE-RETURNING atomicity
    against concurrent workers lands when the cmd-
    side runner does (v1.61).
  - **Listing filters** (state filter, operator
    filter, time-range cursor): NOT in this chunk.
    The contract is List(limit) — adding filters
    extends the contract without breaking it.
  - **TTL / archival policy** for completed jobs:
    NOT in this chunk. The table grows
    unbounded until a future cycle adds a retention
    sweeper (mirroring the audit retention pattern
    from v1.6).
