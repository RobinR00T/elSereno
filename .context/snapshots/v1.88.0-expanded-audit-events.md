# Snapshot — v1.88.0: expanded audit event types

Single-chunk cycle. v1.84 logged `force_overwrite` only;
v1.88 adds `delete` + `set_enabled_true` +
`set_enabled_false` so the schedule audit covers the
full lifecycle of operator-driven mutations.

## Schema (migration 00012)

```sql
ALTER TABLE scan_schedule_audit
    ALTER COLUMN schedule_id DROP NOT NULL;

-- v1.84's CASCADE wiped audit history on schedule delete.
-- SET NULL preserves the audit row with schedule_id NULL.
ALTER TABLE scan_schedule_audit
    DROP CONSTRAINT scan_schedule_audit_schedule_id_fkey;
ALTER TABLE scan_schedule_audit
    ADD CONSTRAINT scan_schedule_audit_schedule_id_fkey
    FOREIGN KEY (schedule_id) REFERENCES scan_schedules(id)
    ON DELETE SET NULL;

ALTER TABLE scan_schedule_audit
    DROP CONSTRAINT scan_schedule_audit_event_type_check;
ALTER TABLE scan_schedule_audit
    ADD CONSTRAINT scan_schedule_audit_event_type_check
    CHECK (event_type IN (
      'force_overwrite', 'delete',
      'set_enabled_true', 'set_enabled_false'
    ));
```

## Go enum

ValidScheduleAuditEventTypes extended to 4 constants:

  - ScheduleAuditEventForceOverwrite (v1.84)
  - ScheduleAuditEventDelete (v1.88)
  - ScheduleAuditEventSetEnabledTrue (v1.88)
  - ScheduleAuditEventSetEnabledFalse (v1.88)

SQL DDL is the source of truth (PITF-030).

## Handlers

`deleteSchedule(store, audit)`: snapshot before, DELETE,
write `delete` audit (payload_before = schedule,
payload_after = JSON null). FK SET NULL nulls
schedule_id during the cascade — the audit row persists.

`setScheduleEnabled(store, audit, enabled)`: snapshot
before, SetEnabled, write `set_enabled_{true,false}`
with full before/after snapshots. The v1.85
computeAuditEventDiff renders the enabled flip in the
dashboard table.

Both audit appends are best-effort: failure surfaces as
`X-Schedule-Audit-Warning` response header (v1.84
pattern). The mutation succeeds even if audit fails.

`Schedules()` factory now passes `audit` to both handlers
in addition to v1.84's updateSchedule.

## Honest scope

- FK SET NULL → orphaned audit rows. List(schedule_id)
  doesn't surface the delete event itself (its
  schedule_id is now NULL). Post-delete forensics
  require querying by event_type directly (deferred).
- No backfill of pre-v1.88 deletes/toggles.
- v1.85 audit-history view renders delete events as
  "every field changed to —" — works, but a dedicated
  "Deleted" badge is deferred.
- audit/events_test.go regex (v1.84-scoped to
  audit_log) means new event types don't bleed into
  AllEventTypes drift checks.
