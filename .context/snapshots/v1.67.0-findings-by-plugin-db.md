# Snapshot — v1.67.0: DBStore persistence for findings_by_plugin

Single-chunk cycle. Closes the v1.66 honest-scope gap:
the per-plugin findings breakdown now persists across
serve restarts in db-store mode.

## Why a separate cycle

v1.66 deliberately landed the breakdown without
schema change so the contract could get reviewed
against the dashboard before the migration was
written. The persistence gap was documented as
honest-scope. v1.67 closes it cleanly: one new
column, JSONB, NOT NULL DEFAULT, no rewrites of the
v1.66 wiring above the SQL layer.

## Migration: 00006_scan_jobs_findings_by_plugin.sql

```sql
ALTER TABLE scan_jobs
  ADD COLUMN findings_by_plugin JSONB NOT NULL DEFAULT '{}'::jsonb;
```

Decisions:

  - **JSONB, not a child table.** The map is bounded
    (≤30 plugins per scan even with empty Plugins),
    only read in aggregate, never queried by plugin
    name. JSONB lets pgx round-trip Go map[string]int
    without an extra join.
  - **NOT NULL DEFAULT '{}'**. Pre-v1.67 rows from
    migration 00005 get a sensible default. The Go
    side decodes empty JSONB to a nil map so the
    JSON output stays clean (omitempty drops nil but
    keeps empty-non-nil maps).

## DBStore changes

  - **jobColumns** projection: 13 → 14 columns.
  - **scanJob** scans the new column as `[]byte` then
    json.Unmarshal into `map[string]int`. Empty
    `{}` decodes to a nil map (post-Unmarshal
    normalisation).
  - **runTransition** for StateCompleted /
    StateFailed: binds findings_by_plugin as the new
    last UPDATE arg via `encodeFindingsByPlugin`.
  - **encodeFindingsByPlugin**: nil/empty maps yield
    `[]byte("{}")` so the NOT NULL DEFAULT-driven
    constraint is never violated.

DBStore.Submit doesn't bind the column — it relies
on the DEFAULT for new rows. Workers fill it in via
the running → completed/failed transition. This
keeps the INSERT clause unchanged (back-compat with
any external INSERT that hasn't been updated to bind
the new column).

## Test fixture migration

`fakeRows.Scan` 13-column → 14-column. `makeRow`
seeds the new key with `[]byte("{}")`. Three new
test cases:

  - `Transition_PersistsFindingsByPlugin`: writes a
    map, reads it back, asserts SQL contains the new
    column name.
  - `Transition_NilFindingsByPluginYieldsEmptyJSON`:
    nil map binds as `{}`.
  - `Get_DecodesFindingsByPlugin`: JSONB → map
    round-trip on the SELECT path.
  - `Get_EmptyJSONBYieldsNilMap`: column default
    decodes to nil (no spurious empty tooltip).

All pass; lint clean.

## Operator upgrade

```sh
elsereno db migrate
```

Applies 00006 atomically. Existing rows get the
column default. After this, completed jobs preserve
their breakdown across serve restart.

Pre-migration jobs that completed in db-store mode
between v1.66 and the upgrade have lost their
breakdown (the v1.66 honest-scope behaviour). The
column default for these rows is `{}` which decodes
as nil — the dashboard shows them with no tooltip,
the same as a single-plugin job.

## Honest scope

  - **No backfill** for pre-v1.67 completed jobs.
    They show no tooltip; their breakdown is
    irretrievably lost. Acceptable: the operator's
    next scan rebuilds the per-plugin signal.
  - **Down-migration** drops the column. No data
    preservation across rollback — pre-existing
    columns survive but the breakdown is gone.
