# Snapshot — v1.30.0: gate record-wireup + TUI scan launcher + audit filter

Four-chunk cycle that closes two outstanding deferrals from
prior cycles and adds two operator-facing QoL improvements:

1. **Closes v1.28-chunk-3 deferral**: extends the optional
   `Recorder *replay.Recorder` field from the two session-level
   POC gates (pcworx + mms) to all 9 wire-aware gates that
   parse the `io.ReadWriter` internally.
2. **Adds operator-facing capture/replay verbs**: `--record
   FILE` flag on `proxy listen` + new `proxy replay FILE`
   sub-verb that renders captures.
3. **Closes v1.29-chunk-2 deferral**: replaces `feeds.Empty`
   (the placeholder for "interactive mode") with
   `feeds.Interactive` that runs a `scanner.Scanner` from
   inside the TUI.
4. **Adds audit-pane filter**: `/` enters substring-edit mode
   on the audit pane, Enter commits, Esc clears.

The cycle is **purely additive** — every existing batch verb
and every prior TUI mode keeps its current behaviour. New
operator surface is opt-in (operators who don't pass `--record`
or `--input` see no change).

## Chunks

### Chunk 1 — Wire `replay.Recorder` into the 9 wire-aware gates (`1a9bc65`)

The v1.28 chunk-3 commit added the Recorder hook to the two
session-level gates (pcworx, mms) where the gate doesn't parse
the bytes — it just blindly proxies. The commit explicitly
deferred the 7 wire-aware gates because each constructs its
own reader (`bufio.NewReader`, `wire.ReadFrame`,
`http.ReadRequest`, etc.) on the supplied `io.ReadWriter`,
and the wrap point matters: the recorder must wrap BEFORE the
parser reads, otherwise the recording captures nothing.

This chunk extends the same 4-line pattern to 9 gates (the 7
explicit + enip + s7 which v1.28 missed but share the shape):

```go
type WriteGatedHandler struct {
    // ... existing fields ...
    Recorder *replay.Recorder  // NEW
}

func (h *WriteGatedHandler) Handle(ctx, client, upstream io.ReadWriter) error {
    if !h.authorised {
        return ErrSessionNotAuthorised
    }
    if h.Recorder != nil {                         // NEW
        client = h.Recorder.WrapClient(client)
        upstream = h.Recorder.WrapUpstream(upstream)
    }
    // existing goroutine spawn ...
}
```

| Gate     | Lines | Parser shape          |
|----------|-------|-----------------------|
| modbus   | 317   | binary MBAP frames    |
| enip     | 137   | encapsulation packets |
| s7       | 178   | TPKT envelopes (COTP) |
| iax2     | 343   | UDP datagrams         |
| pbxhttp  | 399   | http.ReadRequest      |
| opcua    | 922   | wire.ReadChunk        |
| sip      | 1339  | bufio + textproto     |
| cwmp     | 1671  | http + SOAP envelope  |
| bacnet   | 2306  | UDP BVLC + NPDU + APDU|

Side benefit for forensics: because the wrap happens BEFORE
the parser reads, the recording captures EVERY byte the
client sent, including refused frames + pre-parse junk —
useful for "what did the attacker actually send?" post-mortems
even when the gate's allowlist routing dropped the request.

Test coverage: `TestHandle_RecordsBytesWhenRecorderSet` in
`offensive/write/modbus/gatedproxy_test.go` drives a real
`net.Pipe` pair through the canonical wire-aware gate with
recorder attached, sends an allowed FC=06 WriteSingleRegister,
closes + replays + verifies file perms (0600), header
protocol, and at least one client→upstream chunk. The other
8 gates share the wrap point so any regression in the
recorder package surfaces here first (same precedent as v1.28
chunk 3 testing only pcworx).

### Chunk 2 — `--record FILE` flag + `proxy replay` verb (`cab5b8c`)

CLI half of chunk 1: operators can now actually capture.

`--record FILE` on `proxy listen`:

  - `replay.Open(path, plugin, target)` runs BEFORE the
    `Authorise()` call. A permission failure on the capture
    file fails fast (no audit row for an unstartable session).
  - `attachRecorder()` type-switches on the concrete handler
    and sets the `Recorder` field. Threading via the interface
    would have required widening the narrow `Authorise+Handle`
    surface; keeping the type-switch isolated to one helper
    preserves the interface intent.
  - Operator visibility: prints `proxy: recording to <path>
    (NDJSON, schema=elsereno-replay/v1)` on listener bind.
  - Closes the recorder on every exit path (defer).
  - Unsupported plugins (everything except the 7 listed in
    chunk 1's commit) → EX_USAGE with a friendly "supported:
    sip, iax2, pbxhttp, modbus, opcua, bacnet, cwmp" list.

`elsereno proxy replay FILE` (new sub-verb):

  - `SeekHeader` prints `# protocol`, `# target`, `# started`.
  - `replay.Replay` walks each chunk, formats as:
    `[HH:MM:SS.uuuuuu] c→u  NNB  hex-preview…`
  - `--dir client|upstream|both` filter (default both;
    unknown values → both, fail-open is friendlier than
    fail-silent for operator typos).
  - `--hex-limit N` truncates hex preview at N bytes
    (default 32; 0 = full). 32 keeps each line readable in
    a 120-col terminal.
  - Observe-only: re-playing recorded bytes against a lab
    PLC is library-level (`replay.Replay`'s callback is
    generic enough for tools to do that); the CLI prioritises
    forensic post-mortem, the 90% use case.

Tests in `cmd/elsereno/cmd_proxy_replay_offensive_test.go`:
4 cases — header preamble + chunk roundtrip, `--dir` parser
across 7 inputs (incl. aliases + garbage), arrow rendering
(c→u / u→c), hex-limit truncation behaviour.

### Chunk 3 — TUI scan launcher (interactive mode) (`4bd7a8e`)

Closes the v1.29-chunk-2 deferral. The previous "interactive
mode" used `feeds.Empty` (no events flow); operators who
wanted live findings had to pipe `scan --output-format
ndjson | tui --feed -`. v1.30 replaces that with
`feeds.Interactive` running a real `scanner.Scanner` inside
the TUI process.

`internal/tui/feeds/interactive.go`:

  - Spawns `scanner.New(opts).Run(ctx, targets, probe)` and
    drains both findings + errs channels using the 2-phase
    nil-channel idiom (set the channel to nil after closure
    so the next select drops it without spinning).
  - Per finding: emits `tui.FindingMsg` + bumps an atomic
    counter + emits `tui.ScanProgressMsg{Completed, Total}`.
  - Per scanner error: emits `tui.AuditMsg{Line: "scan: …"}`
    (warn-and-continue, mirroring batch `scan`).
  - Initial `ScanProgressMsg{Completed: 0, Total: N}` so the
    bar renders 0/N immediately rather than "idle" until the
    first finding lands.
  - Final `ScanProgressMsg{Total: 0}` on close so the bar
    transitions back to "idle" once every target has been
    probed.
  - Empty target list → `AuditMsg{"interactive: no targets
    to scan"}` rather than silent run.

CLI wiring: `--input list:FILE` + `--default-port` flags on
`elsereno tui`. Pre-flight stat in `cmd_tui.go` so a missing
file fails fast before the alt screen takes over (same
pattern as `--replay` from v1.29 chunk 3). Other `--input`
kinds (nmap / shodan / censys / fofa / zoomeye / onyphe /
internetdb) are deferred — operators with cloud sources can
pipe `scan --output-format ndjson | tui --feed -`.

`pickFeed` signature was refactored to take a `pickFeedArgs`
struct because adding `--input` + `--default-port` pushed the
positional arg count past the linter's ceiling.

Tests in `internal/tui/feeds/interactive_test.go` (5 cases):
happy path with 3 targets, empty target list, probe error →
audit fold, context cancellation, embedded target count in
`Name()`.

### Chunk 4 — TUI audit-pane substring filter (`15f954e`)

Operator triaging a long-running scan can now narrow the
audit pane to "vault" / "scan" / "tls" / their target IP
without scrolling.

Keys (when audit pane focused):

| Key       | Action                                       |
|-----------|----------------------------------------------|
| `/`       | enter filter-edit mode                       |
| Enter     | commit draft to `AuditFilter`                |
| Esc       | cancel draft (preserve previous filter)      |
| Backspace | pop one rune from draft                      |

Outside the audit pane, `/` is a no-op so a misclick on a
different pane doesn't surprise. Esc outside edit mode
clears any active filter.

Implementation:

  - `internal/tui/model.go`: 3 new fields (`AuditFilter`
    committed string, `FilterEditing` bool, `FilterDraft`
    in-progress string) + `FilteredAuditEvents()` returns the
    AuditEvents slice narrowed by `AuditFilter` (full
    pass-through when filter is empty; case-insensitive
    `strings.Contains` on lowercased pair).
  - `internal/tui/update.go`: `handleFilterKey` consumes keys
    while editing — Enter commits, Esc cancels, Backspace
    pops a rune, KeyRunes / KeySpace append. `handleNavKey`
    extracted from `handleKey` to bring its cyclomatic
    complexity back under the linter ceiling.
  - `internal/tui/view.go`: pane header shows live draft
    while editing, committed filter when set, and an empty-
    state message that distinguishes "no events yet" from
    "no events match /<filter>".

Tests in `internal/tui/filter_test.go` (9 cases): filter no-op
when empty, case-insensitive match, no-match → empty (non-nil)
slice, edit cycle (`/` → type → Enter), Esc cancels +
preserves previous filter, Backspace pops, `q` while editing
does not quit, Esc outside edit clears, `/` non-audit pane is
no-op.
