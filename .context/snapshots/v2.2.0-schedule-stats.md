# Snapshot — v2.2.0: per-schedule run-stats aggregate

Single-chunk cycle. Operators want a single answer to
"is my critical schedule healthy?" — success rate, average
duration, total findings — without scrolling /runs. v2.2
adds an aggregate endpoint computed in a single SQL query
(or in-memory walk).

## REST

```
GET /api/v1/schedules/{id}/stats
GET /api/v1/schedules/{id}/stats?days=N
```

`days` clamped to [1, 365], default 7.

Response:

```json
{
  "schedule_id": "abc...",
  "window_days": 7,
  "since": "2026-05-06T13:35:00Z",
  "now": "2026-05-13T13:35:00Z",
  "total_runs": 42,
  "completed": 38, "failed": 2, "cancelled": 1, "running": 1, "queued": 0,
  "success_rate": 0.9047619047619048,
  "avg_duration_seconds": 12.34,
  "avg_findings_per_run": 5.71,
  "total_findings": 240
}
```

`success_rate = completed / total_runs`. Zero runs → 0
(defensive — JSON NaN is nasty). `avg_duration_seconds`
only averages jobs with both StartedAt + FinishedAt set
(in-flight + queued excluded from the divisor; in-mem
implementation explicit, PG via FILTER).

## Store

New method on `scanorch.Store`:

```go
StatsBySchedule(ctx, scheduleID string, since time.Time) (ScheduleRunStats, error)
```

Zero `since` defaults to "now - 7 days" inside the impl.

  - **Memory**: single pass over `s.order`, filter by
    schedule ID + CreatedAt >= since, accumulate
    per-state counts + findings + duration.
  - **PG**: single SQL with COUNT(*) FILTER (WHERE state
    = ...) per state + AVG(EPOCH(finished-started))
    FILTER (WHERE both set) + SUM(findings_count).
    Planner uses idx_scan_jobs_triggered_by_schedule
    (v1.92 partial index).
  - **BroadcastingStore**: pure passthrough.

`ScheduleRunStats` struct documented field-by-field with
ranges + null-semantics.

## Tests (+3)

  - TestScheduleStats_Empty: 0 runs → all zeros, no NaN
    in success_rate; default window=7 days.
  - TestScheduleStats_HappyMix: 2 completed + 1 failed
    in window → success_rate=2/3, total_findings=15
    (3 jobs × 5 findings each), window_days=30 from
    explicit param.
  - TestScheduleStats_BadDays: ?days=abc → 400.

## Honest scope

  - No time-bucketed series (no daily/weekly breakdown
    in this cycle). Operators wanting a sparkline can
    call /runs with cursor pagination + bucket
    client-side. A future cycle could add
    /stats/timeseries.
  - PG variant assumes the v1.92 partial index covers
    the query. Without the index, the COUNT FILTER would
    still work but scan the full table.
  - INSTALL.md unchanged. Build sizes unchanged.

## Build sizes
