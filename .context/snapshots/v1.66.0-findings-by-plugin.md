## Why a separate cycle

`stats.findings_count` is a useful aggregate but
opaque under multi-plugin: a job that ran modbus +
s7 + enip and produced 12 findings could be
12 modbus, or 0 modbus + 12 s7, or any mix. The
dashboard tooltip on the Findings cell now answers
this directly: hover and see `modbus: 7, s7: 3,
enip: 2`.

## Runner implementation

`defaultScanRunner` tracks per-plugin counters with a
`map[string]*atomic.Int64` keyed by plugin name:

```go
byPlugin := make(map[string]*atomic.Int64)
for _, plugin := range plugins {
    var perPlugin atomic.Int64
    byPlugin[plugin.Name] = &perPlugin
    ...
    drainPluginRun(findings, errs, &findingsCount,
                   &targetsScanned, &perPlugin, emit)
}
```

Each `drainPluginRun` increments the right counter on
every finding. `snapshotByPlugin()` builds a fresh
`{name â†’ int}` map for each `report()` call (so the
SSE listener gets a coherent snapshot).

Run loop refactored: extracted `runState` type with
`dispatchAll` helper to keep Run under the funlen
60-line ceiling.

## SSE wire shape

Both the existing event kinds gain the breakdown:

```json
// scan_state_change
{
  "id": "abc12345",
  "state": "completed",
  ...
  "stats": { "targets_seen": 50, "findings_count": 12 },
  "findings_by_plugin": {
    "modbus": 7, "s7": 3, "enip": 2
  }
}

// scan_stats_progress
{
  "id": "abc12345",
  "stats": { "targets_seen": 50, "targets_scanned": 30,
             "findings_count": 8 },
  "findings_by_plugin": {
    "modbus": 5, "s7": 2, "enip": 1
  }
}
```

Throttle policy unchanged: identical-snapshot
suppression keys on Stats only. The byPlugin map may
drift between throttle-window boundaries without
changing aggregate findings_count (e.g., a finding
moves a per-plugin counter from 0 to 1 while
findings_count was already 5 from earlier plugins).
For dashboard purposes the breakdown is
informational; the counters are primary.
