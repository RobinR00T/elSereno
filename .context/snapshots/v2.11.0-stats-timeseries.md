## REST

```
GET /api/v1/schedules/{id}/stats/timeseries?bucket=hour|day|week&days=N
```

Defaults: bucket=day, days=7. Returns:

```json
{
  "schedule_id": "abc...",
  "bucket": "hour",
  "window_days": 7,
  "series": [
    {"bucket_start": "2026-05-13T03:00:00Z", "total_runs": 0, "completed": 0, "failed": 0, "cancelled": 0, "total_findings": 0},
    {"bucket_start": "2026-05-13T04:00:00Z", "total_runs": 1, "completed": 1, "failed": 0, "cancelled": 0, "total_findings": 7},
    ...
  ]
}
```

Memory variant pre-fills empty buckets so the dashboard
sparkline has no gaps. PG variant omits empty buckets
(operator generates time series client-side or via a
future generate_series JOIN).

## Store

  - `Store.StatsTimeseries(ctx, scheduleID, since, bucket)`.
  - Constants `StatsBucketHour`, `StatsBucketDay`,
    `StatsBucketWeek`.
  - Sentinel `ErrInvalidStatsBucket` â†’ 400 in REST.
  - **Memory**: `bucketDuration` + `truncateToBucket`
    helpers; pre-fills `since..now` buckets.
  - **PG**: `date_trunc(<bucket>, created_at)` + GROUP BY.
    Bucket name whitelisted before being interpolated into
    the SQL (not parametric in PG; safe via the switch
    guard).
  - **BroadcastingStore**: passthrough.

## Honest scope

  - Memory pre-fill is bounded by `days * (24|24*7|1)`
    buckets â€” max 168 hourly over 7 days, or 8760 over a
    year. Acceptable.
  - PG "week" uses `date_trunc('week', â€¦)` which is
    Monday-start in ISO-8601 (matches the Memory variant's
    Monday-aligned truncateToBucket helper).
  - Counters for queued+running jobs are NOT broken out
    (terminal states only). TotalRuns includes them so
    the dashboard can show "1 in-flight" by subtraction.
