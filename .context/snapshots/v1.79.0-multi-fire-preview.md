# Snapshot — v1.79.0: multi-fire preview

Single-chunk cycle. v1.77 shipped a single-fire preview but
non-trivial cron patterns ("every weekday at 09:00 NY" with
a restricted DoM) need 2-5 fires to visually verify, not
just the first one. v1.79 extends `/preview` to return up
to 10 fires.

## API

```go
const PreviewNextFiresMaxCount = 10

// New canonical multi-fire predictor.
func (s ScanSchedule) NextFires(now time.Time, count int) []time.Time

// Free function for /preview.
func PreviewNextFires(req CreateScheduleRequest, now time.Time, count int)
    ([]time.Time, error)

// v1.77 single-fire helpers preserved as thin wrappers
// for back-compat.
func (s ScanSchedule) NextFire(now time.Time) time.Time
func PreviewNextFire(req CreateScheduleRequest, now time.Time)
    (time.Time, error)
```

REST:

```
POST /api/v1/schedules/preview?count=5
Body: CreateScheduleRequest

200 → { "data": {
    "next_fire_at": "...",       // == next_fires[0] (back-compat)
    "next_fires":   ["...", ...], // up to count elements
    "timezone":     "..."
} }
400 on validation OR malformed count.
```

count clamping happens server-side: 0 / negative → 1;
100 → PreviewNextFiresMaxCount = 10. Cap chosen because
each cron `c.Next()` walk is up to 1 year bounded; 10×1y
is a worst-case upper bound on compute.

## Implementation

`intervalNextFires`:
  - Never-fired schedules eagerly include `now` as the
    first fire (matches v1.77 NextFire semantics), then
    walk last+I, last+2I, …
  - Fired schedules walk last+I, last+2I, …

`cronNextFires`:
  - Calls `c.Next()` count times, anchoring each step on
    the previous result.
  - Truncates the slice at the first no-match-within-1y
    boundary (defence in depth — Create validates the
    cron, so this only triggers on the
    "0 0 30 2 *" / "Feb 30" never-matches case).

## Dashboard

cron-mode "Preview next fire" now requests `count=5` and
renders the result as an ordered list (`<ol
class="next-fires-list">`). Interval mode stays at count=1
because the cadence is trivially extrapolated. Overdue
rows in the list get the "— overdue" suffix.

`next_fire_at` is preserved on the response as
`next_fires[0]` for v1.77/v1.78 callers.

## Persistence

None. `next_fires` is computed on every preview request.
No new DB columns.

## Honest scope

  - Cap at 10. Operators wanting more should script against
    the API + step locally.
  - No "next N" column on the schedules table — bloat.
  - Interval mode in the dashboard hardcoded to count=1;
    the API itself accepts count for interval.
  - No DST-skip annotation in the list. Operators who
    care about wall-clock-vs-UTC drift can compare against
    the timezone label.
