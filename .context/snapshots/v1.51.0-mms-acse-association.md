# Snapshot — v1.51.0: MMS ACSE A-ASSOCIATE-REQUEST

Single-chunk cycle. Closes the v1.32+ E carryover. Bumps
MMS fingerprint confidence from ~0.8 (COTP-CC TSAP
disambig) to ~0.95 (real IEC 61850-8-1 IED handshake)
for substation deployments by adding the OSI Session +
Presentation + ACSE AARQ exchange.

## Why an ACSE association

Pre-v1.51, the MMS plugin only confirmed COTP-CC with
MMS-style TSAPs (`00 01` source/destination). That's a
positive disambig from S7 (which uses `01 00` / `01 02`)
but doesn't prove the upstream is actually IEC 61850 —
some vendor stacks reuse the MMS TSAPs without speaking
the application layer.

The standards-compliant test is to send an ACSE
A-ASSOCIATE-REQUEST with the IEC 61850-8-1 application-
context-name OID `1.0.9506.2.3` and check that the
A-ASSOCIATE-RESPONSE echoes the OID. Real IEC 61850 IEDs
do this; non-IEC stacks either reject the AARQ at the
session layer or return an AARE with a different OID.

## Chunks

### Chunk 1 — wire/acse.go + plugin wiring (`4e13192`)

`internal/protocols/mms/wire/acse.go` (new, ~180 lines):

  - **MMSApplicationContextOID** = `[]byte{0x28, 0xCA,
    0x22, 0x02, 0x03}` — BER encoding of 1.0.9506.2.3
    (first two components packed into 0x28; 9506 as
    varlen subid `CA 22`; 2; 3).
  - **BuildACSEAssociateRequestMMS()** — hand-coded
    static OSI Session CONNECT + Presentation CP +
    ACSE AARQ blob requesting the IEC 61850-8-1
    context. ~120 bytes; reverse-engineered from
    libiec61850's `client_example_basic_io.c` against a
    Conpot honeypot, cross-checked with Wireshark MMS
    dissector. Each layer is annotated inline with its
    spec-section reference (ISO 8327 PI numbers, ISO
    8823 normal-mode parameters, ACSE [1] application-
    context-name + [30] user-information).
  - **ParseACSEAssociateResponseMMS(buf)** — byte-
    pattern scan for the OID in the AARE response.
    Layout-agnostic + zero-allocation. Returns nil on
    match, ErrNoMMSACSEResponse / ErrACSETooShort
    otherwise.

`internal/protocols/mms/mms.go`:

  - **tryACSEAssociate(conn, ioTimeout)** — after a
    positive COTP-CC, sends the AARQ wrapped in COTP DT
    (LI=02, type=0xF0, TPDU-nr=0x80) + TPKT, parses
    the AARE. Returns ("MMS ACSE associated (IEC
    61850-8-1)", true) on OID match; ("", false) on
    any failure.
  - **Plugin.Probe** falls back to the v1.25 COTP-CC
    note on AARQ/AARE failure. Non-IEC-61850 MMS-style
    servers see no behaviour change.

## Why a static AARQ frame

The OSI Session + Presentation + ACSE stack is BER
ASN.1; a full encoder would pull non-trivial dependency
weight. The static blob is verified to parse on
libiec61850 + scapy-mms + against Conpot. Vendor
variations land in the *response*, where the OID-pattern
scan handles them gracefully. A future cycle that needs
to expose Initiate-Response fields (vendor name,
service-options) would justify a real BER encoder; v1.51
scope stops at fingerprint.

## Why a byte-pattern scan rather than ASN.1 parse

AARE layout varies across vendors:

  - Some include a presentation-context-result-list
    (echo of accepted contexts).
  - Some attach an authentication-information block.
  - Some prepend an explicit protocol-version BIT
    STRING; others assume implicit version 1.

But every IEC 61850-8-1 IED includes the application-
context OID byte-for-byte in the AARE. Pattern scan is
robust to all the variation while staying simple +
allocation-free. False positive risk is vanishingly
small (the byte sequence is structured + uncommon and
the scan runs only on buffers that already passed
COTP-CC).

## Honest scope notes

  - The AARQ is not customisable per target. Real-world
    IEDs accept the static frame because IEC 61850-8-1
    pins the OSI parameters (mode, syntaxes, context
    IDs). A future cycle could add per-target tuning if
    a non-conforming IED is encountered.
  - The AARE parse is OID-pattern scan, not full BER.
    Service-spec extraction (Initiate-Response with
    vendor name, supported services) needs a real
    parser — probably a v1.6x+ candidate.
  - Lab validation against real Mark VIe / SEL / GE
    Multilin / ABB IEDs is still pending (same gap as
    v1.28 ProConOS + v1.28 GE-SRTP). Code lands today;
    confidence note in INSTALL.md / docs marks ~0.95
    pending real-IED validation.
