# Snapshot — v1.77.0: dashboard next-fire preview

Single-chunk cycle. Operators creating or editing a cron
schedule had no way to see what their cadence would produce
before submitting (gap explicitly noted in the v1.74
honest-scope section). v1.77 adds a tz-aware predicted-next-
fire computation + a REST preview endpoint + dashboard
plumbing.

## Why

UX. "Every weekday at 09:00" + a typo in the cron field
produces a silently-wrong fire schedule that only manifests
when the operator wonders why nothing fired. A "next fire"
preview catches the typo before submit. For overdue
schedules (next fire in the past), the same column makes it
obvious which ones will trigger on the next tick.

## API

```go
// On ScanSchedule (computed at read time, not persisted):
NextFireAt time.Time `json:"next_fire_at,omitempty"`

// Method:
func (s ScanSchedule) NextFire(now time.Time) time.Time

// Free function:
func PreviewNextFire(req CreateScheduleRequest, now time.Time)
    (time.Time, error)
```

REST:

```
POST /api/v1/schedules/preview
Body: CreateScheduleRequest (cadence + tz)
200 → { "data": { "next_fire_at": "...", "timezone": "..." } }
400 on validation error (same sentinels as Create)
```

## Implementation

NextFire branches on `Enabled` + `CronExpr != ""`:

  - Disabled → zero time.
  - Cron path: shares cronNextFire helper with cronIsDue.
    Both anchor and now are converted to s.Timezone before
    Next/Match. Returns UTC.
  - Interval path: never-fired returns now (eager); fired
    returns LastFiredAt + IntervalSeconds.

The cron-path return is the literal next match — for
overdue schedules it can be in the past, which the
dashboard renders as "(overdue)" so operators see the
schedule will fire on the next tick.

cronIsDue now delegates to cronNextFire (DRY). Same
observable behaviour; less code drift.

## Validation refactor

Extracted `writeScheduleValidationError` in
`handlers/schedules.go`. The same five sentinels
(Name/TemplateInput/CadenceRequired/CadenceConflict/
CronInvalidField/InvalidTimezone) all map to 400; default
500. createSchedule + updateSchedule (after the not-found
branch) + previewSchedule all use the helper. Closes the
dupl linter complaint that fell out from the Preview path.

## Persistence

No migration. NextFireAt is computed at read time only.
The DB and Memory stores both leave it at zero. The REST
layer's `withNextFire(s, now)` and `withNextFireSlice(s,
now)` populate the field before serialisation.

## Dashboard

  - Schedules table gets a "Next fire" column. Empty
    next_fire_at renders "—". Local-clock string with
    "(overdue)" suffix when in the past. colspan bumped
    7 → 8 across all empty/error templates.
  - "Preview next fire" button next to submit. Clicking
    sends the current form values to /preview and renders
    the response in a sub panel below the submit row.
    Errors surface inline ("preview error: invalid cron
    expression: …") so the operator can fix the form
    before submitting.

## Honest scope

  - **No live preview** as fields change. Button-click
    only — typing "0 9 * * 1-" without the trailing
    "5" would otherwise show a parse error mid-edit.
  - **Single fire, not next-N**. A future cycle could
    render the next 3-5 fires for "does this look right"
    sanity-checking; v1.77 keeps it minimal.
  - **Server "now"**. The preview is anchored to server
    wall-clock; client clock skew can produce
    "(overdue)" that the operator wouldn't expect.
    Acceptable for loopback dashboards.
  - **NextFireAt is not a request input**. It only
    appears on responses; setting it on a body is
    silently ignored.
