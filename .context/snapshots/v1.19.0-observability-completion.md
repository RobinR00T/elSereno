# Snapshot — v1.19.0: observability completion

Three chunks closing observability loose ends from v1.16 +
v1.17. Operators can now see audit-chain entries on the
dashboard, watch reload cadence at a glance, and detect post-
flash firmware swaps automatically.

## Chunk 1 — Audit log API endpoint + dashboard panel

`(commit hash)` Operators can finally see audit-chain entries
on the dashboard without `tail -f ~/.elsereno/audit.jsonl`.
New `GET /api/v1/audit?event_type=&actor=&occurred_after=&
limit=` returns the newest 50 (clamped [1, 500]) audit
entries descending by occurred_at. New
`GET /api/v1/audit/cadence?event_type=&days=N` returns per-
day counts (clamped [1, 90]) — used by chunk 2's reload-
cadence panel + future events-over-time charts.

Library: new `internal/repo/audit.go` with `AuditEntry` /
`AuditQuery` / `AuditCadence` / `ListAuditLog` /
`ListAuditCadence`. Tombstoned rows (audit-purge per ADR-013)
come back with `payload=null` + `tombstoned=true` so
consumers render `[redacted]` while preserving the chain
entry. Chain-integrity columns (prev_hash / entry_hash) are
deliberately NOT exposed by the API — they live in the chain
proper for offline `audit verify-file`.

Dashboard: new "Audit feed" panel (event_type dropdown +
actor filter + 4-column table with payload excerpt ≤120
chars). 6 new tests.

## Chunk 2 — Reload cadence dashboard panel

`d2ebb0f` Surfaces v1.17-chunk-5 `proxy_allowlist_reload`
audit rows as a per-day count chart for the last 7 days.
Reuses chunk-1's `/api/v1/audit/cadence` endpoint; renders a
text-based bar chart (`█` width-30 scaled to the row with
the max count) so the dashboard doesn't pull in a chart
library. Pure dashboard addition; no new API surface, no
new tests (chunk 1 covers the underlying endpoint).

## Chunk 3 — CWMP TransferComplete async firmware re-fetch

Closes the v1.16-chunk-1 loose end. TR-069 doesn't carry
the SHA-256 in TransferComplete — the CPE just reports
success/failure — so a firmware swap on the source server
(compromised ACS staging host) passes undetected if the
operator only relied on the v1.13-chunk-2 pre-flight
`verify-firmware`. v1.19 chunk 3 closes this by re-fetching
+ hashing the URL post-flash, audit-emitting any mismatch.

CLI: `proxy listen --plugin cwmp --verify-firmware-on-
complete --verify-firmware-timeout 5m` (off by default;
opt-in for high-stakes ISP-grade fleets).

Architecture:
- `verifyingTransferCompleteObserver` wraps the v1.15-
  chunk-1 default observer. On every successful TC carrying
  a resolved Authorisation with non-empty AllowlistSHA256,
  spawns a goroutine that fetches AllowlistURL, hashes the
  body, compares against AllowlistSHA256, emits a
  `cwmp_firmware_verify` audit row.
- `chooseTransferCompleteObserver` returns the wrapper when
  the flag is set; default observer otherwise (byte-
  identical for non-opt-in).
- Goroutine-detached. Streams + hashes (no full-image
  buffering — firmware can be tens of MiB). Network failures
  produce `unreachable` audit row, not a missed audit.

Audit: new `audit.EventCWMPFirmwareVerify` const + migration
`00004_*` extending the audit_log CHECK enumeration.

9 new tests covering match/mismatch/unreachable
classification + observer-skip cases (no auth, empty SHA-
256, fault path) + defensive nil-runtime no-ops + opt-
in/opt-out switch.
