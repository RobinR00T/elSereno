# Snapshot — v1.70.0: scheduled scans

Single-chunk cycle. Saved Job templates that fire
automatically on a fixed interval — operators get
continuous-monitoring workflows ("scan my fleet
every 6 hours") without external cron scripts.

## Why interval-based, not cron-style

Continuous security scanning wants "every N seconds"
cadence. Wall-clock anchors ("every Monday at 02:00")
matter for human-facing reports but not for asset
monitoring. Interval-based:

  - Simpler parser (just an int).
  - More-useful default for the security use-case.
  - Handles "scan as soon as the last scan finishes"
    cleanly: set IntervalSeconds = 60 and the
    Scheduler fires it next-tick whenever the
    interval has elapsed.

A future cycle can add a CronExpr alternative — the
Scheduler just needs an `IsDue(now)` predicate.

## Architecture

```
                  ┌──────────────────────┐
                  │ ScheduleStore        │
                  │  ScanSchedule[]      │
                  │   ID, Name,          │
                  │   Template,          │
                  │   IntervalSeconds,   │
                  │   Enabled, ...       │
                  └──────┬───────────────┘
                         │
           ┌─────────────┴───────────────┐
           │                             │
   POST /schedules                  Scheduler
   GET  /schedules                  goroutine
   DELETE /schedules/{id}           ticks every 30s
   POST /schedules/{id}/enable      ↓
   POST /schedules/{id}/disable     for s in store.List():
                                      if s.IsDue(now):
                                        store.MarkFired(s.ID)
                                        ScanStore.Submit(s.Template)
                                        OnFire(s.ID, job)
```

The ScheduleStore goes into APIV1Deps; the Scheduler
goroutine spawns from cmd_serve at startup.

## Fire ordering

**MarkFired BEFORE Submit.** A Submit that failed
without an updated LastFiredAt would re-fire on every
tick — the worst failure mode (tight retry loop on a
broken scan). Stamping first means a Submit failure
pauses the schedule for one interval; the operator's
next tick re-evaluates with the correct timestamp.

The cost: a transient Submit failure loses one
firing window. Acceptable trade vs. a potential
flood; operators see the OnFireError stderr log.

## Clamping bounds

`IntervalSeconds` is clamped to [60s, 7d]:

  - **60s lower** — prevents accidental 1-Hz scanners
    hammering the pool.
  - **7d upper** — monthly+ belongs to manual scans;
    keeps the LastFiredAt math sane.

`Scheduler.TickInterval` is clamped to [10s, 5min]:

  - **10s lower** — finer than that wastes CPU on
    a quiet store.
  - **5min upper** — coarser than that and a 60s
    schedule won't fire close to its expected
    cadence (worst-case 5min slip).

Default 30s. Sized so a 60s schedule fires within
30s of its expected cadence in the worst case.

## REST endpoints

```
POST   /api/v1/schedules                create (201)
GET    /api/v1/schedules                list (200, sorted by Name)
GET    /api/v1/schedules/{id}           one (200, 404)
DELETE /api/v1/schedules/{id}           remove (204, 404)
POST   /api/v1/schedules/{id}/enable    toggle on (200)
POST   /api/v1/schedules/{id}/disable   toggle off (200, history preserved)
```

Sub-router behind APIV1Deps.ScheduleStore. Nil store
→ 503 on every endpoint (same degraded-deps pattern
as Scans).

## cmd_serve wiring

Scheduler only spawns when `--scan-store != off` —
no point having schedules without a place to fire
them. `OnFire` / `OnFireError` log to stderr; a
future cycle could wire metrics counters here.

`buildScanAndSchedule` extracted from runServe to
keep gocyclo under 15. Returns scan + schedule
stores plus a stop closure that the caller defers.

## Honest scope

  - **DBStore persistence** NOT in this chunk.
    Schedules don't survive `serve` restart in
    v1.70; v1.71 adds the migration + DBStore
    counterpart of MemoryScheduleStore.
  - **Dashboard UI** NOT in this chunk. Operators
    use curl. v1.72 adds a panel.
  - **Cron expressions** NOT supported. Interval
    is the only cadence model. Future cycle adds
    CronExpr as an alternative IsDue trigger.
  - **Multi-process advisory locking** for
    DB-backed Stores when multiple `serve`
    instances share a DB: deferred. The in-memory
    store is single-process by design; the future
    DB chunk will need SELECT FOR UPDATE on the
    schedule row before MarkFired.
