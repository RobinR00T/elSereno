# Snapshot — v1.71.0: DB-backed scheduled scans

Single-chunk cycle. Closes the v1.70 honest-scope
gap: scan schedules persist across serve restarts in
db-store mode. Memory-store mode unchanged (still
in-process).

## Migration: 00007_scan_schedules.sql

```sql
CREATE TABLE scan_schedules (
    id                    TEXT PRIMARY KEY,
    name                  TEXT NOT NULL,
    template_input        TEXT NOT NULL,
    template_plugins      TEXT[] NOT NULL DEFAULT '{}',
    template_default_port INTEGER NOT NULL DEFAULT 0,
    interval_seconds      INTEGER NOT NULL,
    enabled               BOOLEAN NOT NULL DEFAULT TRUE,
    operator              TEXT NOT NULL DEFAULT '',
    created_at            TIMESTAMPTZ NOT NULL,
    last_fired_at         TIMESTAMPTZ,
    CHECK (interval_seconds BETWEEN 60 AND 604800),
    CHECK (template_input <> '')
);

CREATE INDEX idx_scan_schedules_name    ON scan_schedules(name);
CREATE INDEX idx_scan_schedules_enabled ON scan_schedules(enabled)
    WHERE enabled = TRUE;
```

**Schema decisions**:

  - `template_*` columns flatten the SubmitRequest.
    Three columns over a JSONB blob because the
    shape is small + stable, and we'd want
    indexable plugin queries in a future cycle
    (e.g., "find all schedules using the modbus
    plugin").
  - Two CHECK constraints mirror the Go-side
    validation: interval clamp + template-input
    non-empty. Defense in depth against an
    out-of-band INSERT (psql / data import).
  - Partial index on `enabled = TRUE`: most
    schedules are enabled, but the Scheduler
    skips disabled ones. The index is small + the
    query plan benefits from skipping disabled
    rows during a tick.

## DBScheduleStore

Implements ScheduleStore via the same Querier
interface DBStore uses (no separate type per
package). Methods:

  - **Create**: parameterised INSERT.
  - **Get**: SELECT + scanSchedule decode.
  - **List**: SELECT ORDER BY name ASC.
  - **Delete / MarkFired / SetEnabled**:
    single-statement Exec. RowsAffected() == 0 →
    ErrScheduleNotFound.

Single-statement mutations are safe at the row
level. SELECT FOR UPDATE on the row before
MarkFired would be needed if multiple `serve`
processes shared a DB; deferred — typical
deployment is single-process.

## cmd_serve wiring

`buildScanAndSchedule` branches on `opts.scanStore`:

  - `memory` → MemoryScheduleStore (single-process,
    lost on restart).
  - `db` → DBScheduleStore (survives restart;
    requires migration 00007 applied).
  - `off` → no schedule store, /schedules returns
    503.

The match between scan-store mode and schedule-
store mode is automatic — operators don't pick
schedule persistence separately.

## Test fixture migration

`fakeRows.Scan` was 14-column-only (scan_jobs).
v1.71 routes by `len(dst)`:

  - 14 columns → scan_jobs (v1.67+).
  - 10 columns → scan_schedules (v1.71+).

Same fakeQuerier handles both stores. An
`execRowsAffected` field on the fake configures
the CommandTag so Delete/MarkFired/SetEnabled
tests can distinguish 1-row hits from 0-row
misses without a separate fake.

## Honest scope

  - **No backfill** for pre-v1.71 schedules. The
    in-memory store didn't persist anything; the
    operator's first restart after upgrading to
    v1.71+ starts with an empty `scan_schedules`
    table. Acceptable: schedules are typically
    re-created from documented configs.
  - **No multi-process advisory locking**. Two
    `serve` processes against the same DB would
    race on MarkFired between tick and Submit; the
    fix is SELECT FOR UPDATE on the schedule row
    before stamping. Deferred — typical deployment
    is single-process. The DB CHECK constraints
    prevent corruption either way.
  - **No SCHEMA migration for existing schedules
    that exceed the new CHECK bounds**. The
    interval CHECK was already enforced in the Go
    layer (v1.70 clampInterval), so any
    legitimately-created schedule fits. Manual
    DB inserts that violate the bounds need a
    schema-aware migration on the operator side.
