# Snapshot — v1.75.0: per-schedule timezone

Single-chunk cycle. Cron schedules in v1.73/v1.74 evaluated
against UTC only. v1.75 adds a `timezone` field on
ScanSchedule + Create/Update so cron expressions evaluate
against the operator's IANA zone (e.g. "America/New_York",
"Europe/Madrid"). Empty timezone falls back to UTC,
preserving v1.73/v1.74 behaviour.

## Why

Operator feedback: "every weekday at 09:00 NY time" required
manually translating the wall-clock to UTC + handling DST
drift twice a year. Operators want a tz-aware cron just like
unix cron + `TZ=...` would give them.

## Contract change

```go
type ScanSchedule struct {
    // ... existing fields ...
    Timezone string `json:"timezone,omitempty"` // v1.75+
}

type CreateScheduleRequest struct { /* + Timezone */ }
type UpdateScheduleRequest struct { /* + Timezone */ }

var ErrScheduleInvalidTimezone = errors.New(...) // 400
```

JSON shape:

```json
{
  "name": "weekday-9am-ny",
  "template": {"input": "list:fleet.txt", "plugins": ["modbus"]},
  "cron_expr": "0 9 * * 1-5",
  "timezone": "America/New_York"
}
```

Empty `"timezone"` (or omitted) ⇒ UTC. Validated via
`time.LoadLocation` at Create + Update time. Bad name → 400
`ErrScheduleInvalidTimezone`.

## cronIsDue (v1.75 — tz-aware)

```go
func (s ScanSchedule) cronIsDue(now time.Time) bool {
    c, err := ParseCron(s.CronExpr)
    if err != nil { return false }
    loc := time.UTC
    if s.Timezone != "" {
        l, err := time.LoadLocation(s.Timezone)
        if err != nil { return false }
        loc = l
    }
    anchor := s.LastFiredAt
    if anchor.IsZero() { anchor = s.CreatedAt }
    next, err := c.Next(anchor.In(loc))
    if err != nil { return false }
    return !now.In(loc).Before(next)
}
```

Both anchor (LastFiredAt or CreatedAt) and now are converted
to `s.Timezone` before computing Next/Match. The cron
expression is evaluated against wall-clock-local fields
(hour/minute/dow), matching the operator's mental model —
DST transitions move the fire time absolute-clock-wise but
cron keeps firing at the same local hour:minute.

## Validation refactor

`validateScheduleFields` gains a 5th `timezone` parameter:

```go
func validateScheduleFields(name string, template SubmitRequest,
    intervalSeconds int, cronExpr, timezone string) error {
    // ... existing checks ...
    if timezone != "" {
        if _, err := time.LoadLocation(timezone); err != nil {
            return fmt.Errorf("%w: %s: %w",
                ErrScheduleInvalidTimezone, timezone, err)
        }
    }
    return nil
}
```

`applyCadence` is unchanged — timezone is set/preserved by
the caller (`buildScheduleFromRequest` for Create,
`MemoryScheduleStore.Update` + `DBScheduleStore.Update` for
Update). An Update that switches modes (interval → cron or
vice versa) keeps the supplied timezone; if the operator
wants to clear it, they pass `"timezone": ""`.

## Migration 00009

```sql
ALTER TABLE scan_schedules
    ADD COLUMN timezone TEXT NOT NULL DEFAULT '';
```

No SQL CHECK. PostgreSQL's view of valid IANA names depends
on the server tzdata bundle which isn't a stable contract;
validation is Go-side only.

## DBScheduleStore

`scheduleColumns` 11 → 12 (timezone slotted between
`cron_expr` and `enabled`). Both INSERT (Create) and
UPDATE ... RETURNING (Update) bind the new column.
`scanSchedule` reads `&s.Timezone` in column order.

## Honest scope

  - **Validation only at Create/Update**. A schedule whose
    timezone disappears from the server tzdata between
    Create and tick-time logs a parse error and skips. tzdata
    rotation is the operator's responsibility.
  - **No fixed-offset zones**. Operators wanting "+05:30"
    must use IANA names ("Asia/Kolkata"). The validation
    surface is whatever `time.LoadLocation` accepts.
  - **No DST-skip handling**. A cron at 02:30 daily during a
    spring-forward day fires next at 02:30 local on the
    following day. Documented in INSTALL.md.
  - **Empty timezone clears across Update**. An Update without
    a timezone field clears the previously-set zone (UTC
    fallback). Matches the explicit-shape pattern from v1.74.
  - **No "next fire in zone" preview** in the dashboard.
    Same gap as cron itself; deferred to v1.77.
