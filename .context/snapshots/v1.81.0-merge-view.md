# Snapshot — v1.81.0: 412 merge-view UI

Single-chunk cycle. Dashboard-only JS + HTML. v1.78
surfaced concurrent-edit conflicts as a bare 412 + error
toast. v1.81 replaces that with a structured merge-view
panel that shows the field-level diff + two explicit
resolution buttons.

## Why

UX. The operator getting a 412 had to manually re-load
the form, re-type their edits, and re-submit (hoping no
one else raced again). v1.81 surfaces the diff so the
operator decides explicitly: take server, force overwrite,
or close and refresh.

## UI

```html
<div id="schedule-merge-view" hidden>
  <strong>Schedule was modified by another operator.</strong>
  <ul id="schedule-merge-diff">
    <li>name: mine=foo · server=bar</li>
    <li>cron_expr: mine=0 9 * * 1-5 · server=0 9 * * *</li>
    ...
  </ul>
  <button onclick="acceptServerSchedule()">Take server (discard my edits)</button>
  <button onclick="forceOverwriteSchedule()">Force overwrite</button>
</div>
```

## Flow

1. submitSchedule captures the payload as
   pendingMergePayload before fetch.
2. On 412, calls enterMergeView(id).
3. enterMergeView GETs /schedules/{id}, caches the
   response on window.scheduleMergeServer + the new
   updated_at on editingScheduleUpdatedAt (so a follow-up
   "submit" lands).
4. computeScheduleDiff compares mine vs. server for the
   six editable fields (name, template.input,
   template.plugins, interval_seconds, cron_expr,
   timezone).
5. Operator clicks one of:
   - Take server: re-loads form via beginEditSchedule
     (clean baseline; pending payload discarded).
   - Force overwrite: confirm() prompt, then re-issues
     PUT without If-Match. Last-write-wins.

## Helpers

```js
var pendingMergePayload = null;
function enterMergeView(id) { ... }
function computeScheduleDiff(mine, server) { return [{field, mine, server}, ...] }
function acceptServerSchedule() { beginEditSchedule(server.id); hide(panel); }
function forceOverwriteSchedule() {
  PUT without If-Match → cancelEditSchedule + renderSchedules
}
```

acceptServerSchedule + forceOverwriteSchedule exposed as
window globals (consumed by inline onclick handlers).

## Tests

The 412 path itself is exercised by the v1.78 REST tests
(TestUpdateSchedule_IfMatchMismatch); the JS plumbing
is dashboard-static-rendering only.

## Honest scope

- Server-state cached on `window.scheduleMergeServer`
  (global namespace pollution; acceptable for the
  dashboard scope).
- No per-field cherry-pick — operator picks one side
  for the whole record.
- Force-overwrite re-issues PUT without If-Match (the
  v1.78 back-compat path), so a third operator
  editing during the review window will still get
  overwritten. The operator is consciously discarding
  the precondition.
