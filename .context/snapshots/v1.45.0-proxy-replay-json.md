# Snapshot — v1.45.0: proxy replay --json

Single-chunk DX cycle. Adds machine-readable JSON output
to `elsereno proxy replay` so operators can pipe the
forensic stream into jq / their own tooling without
parsing the human-readable formatted lines.

## Workflow

```
$ elsereno proxy replay session.ndjson --json | head -2
{"schema":"elsereno-replay/v1","ts":"2026-05-04T12:00:01.123456Z","dir":"client_to_upstream","len":12,"hex":"010100…"}
{"schema":"elsereno-replay/v1","ts":"2026-05-04T12:00:01.456789Z","dir":"upstream_to_client","len":18,"hex":"030600…"}

$ elsereno proxy replay session.ndjson \
    --since 2026-05-04T12:00:00Z \
    --json \
  | jq -c 'select(.dir == "client_to_upstream") | {ts, len}'
```

## Chunks

### Chunk 1 — --json + DirHeader skip (`05c96a5`)

`cmd/elsereno/cmd_proxy_replay_offensive.go`:

  - `--json bool` flag; defaults false (legacy formatted
    lines).
  - Header preamble (`# capture …`) suppressed under
    --json so stdout stays a clean NDJSON stream.
  - Each chunk emits via `json.NewEncoder(stdout).Encode(ev)`
    — one ChunkEvent per line.
  - Composes with --dir / --since / --until filters from
    v1.44; --json controls presentation only.

### Side fix

The dispatcher now explicitly skips `DirHeader` events.
SeekHeader already surfaces protocol + target + start in
the printed preamble; letting the header event fall
through to formatChunk produced a phantom "c→u  0B  "
line because the formatter's `if Dir == DirUpstreamToClient`
check only renames the arrow for one specific direction
and otherwise defaults to "c→u". This was a long-standing
buglet that `TestProxyReplay_RendersHeaderAndChunks` was
accidentally relying on (it asserted "c→u" was in
output, matching the phantom line).

The test is updated to assert on the actual chunk arrow
("u→c", since WrapClient + Write produces a
DirUpstreamToClient direction in the recorder's mapping).
