# Snapshot — v1.62.0: dashboard scan-jobs panel

Single-chunk cycle. Adds the trigger-from-button UI
on top of the v1.58/59/60/61 stack. The dashboard's
Scan jobs panel lets operators submit, watch, and
cancel scan jobs without touching curl. Closes the
last "no UI hookup yet" honest-scope note from v1.58.

## What landed

A new panel between the existing "Recent runs" and
"Reload cadence" sections. Three concerns:

  1. **Submit form** — input + plugin + default_port
     as plain HTML inputs. Submit POSTs to
     /api/v1/scans with X-Operator inherited from the
     authenticated session (same path as audit).
  2. **Jobs table** — newest-first, top 20. Columns:
     state, operator, plugin, input, targets/findings
     stats, created-at, short job ID, action.
  3. **Cancel** — every queued/running row gets a
     button; POSTs to /scans/{id}/cancel.

## Polling cadence

Two-tier:

  - **2s** while any visible job is queued or
    running. Operators see scans advance through
    the state machine in near-real-time.
  - **10s** when every visible row is terminal. An
    idle dashboard doesn't hammer the API.

The cadence flips automatically based on row state
in renderScans(). No SSE wiring yet — that's a
future cycle (the orch endpoints don't emit SSE
events; they'd need a new event type alongside
finding/run_start/run_end).

## 503 surface

If serve was started without `--scan-store`, the
endpoints return 503 and the renderer surfaces a
hint: "scan orchestration disabled (start serve
with --scan-store=memory or =db)". Better than a
silent empty table.

## Implementation

`internal/web/handlers/dashboard.go` (extended):

  - **`<section class="panel" id="scans-panel">`**
    sits between Recent runs and Reload cadence in
    the existing layout.
  - **renderScans()** fetches /api/v1/scans?limit=20,
    decides table content based on res.unavailable +
    rows.length, and reschedules via
    `scheduleScansPoll(2000 | 10000)` based on
    whether any rendered row is non-terminal.
  - **submitScan(ev)** reads the form, JSON-encodes
    the SubmitRequest, POSTs to /api/v1/scans, then
    immediately calls renderScans() to surface the
    new row.
  - **cancelScan(jobID)** POSTs to
    /scans/{id}/cancel; the renderScans() call after
    flips the row's action cell.
  - **escAttr()** helper for unquoted attribute
    contexts (data-job-id, etc.). Stricter than
    escText (drops anything outside [A-Za-z0-9._-]).

## CSP nonce inheritance

The new JS lives inside the existing nonce-bearing
`<script nonce="{{.CSPNonce}}">` block — no new
inline-script source, no CSP relaxation. The
TestDashboard_HasCSPNonce_OnScanScript test pins
this so a template-restructure can't accidentally
move renderScans outside the nonce block.

## Tests (+2 new)

NEW file: `internal/web/handlers/dashboard_test.go`:

  - **TestDashboard_ContainsScanPanel** — pins 10
    HTML/JS markers (panel id, form id, table id,
    body id, /api/v1/scans path, function names).
    A template regression that strips any of them
    fails the test.
  - **TestDashboard_HasCSPNonce_OnScanScript** —
    walks the rendered HTML to confirm the new
    functions stay inside the nonce-bearing
    <script> block.

All pass; lint clean.

## Honest scope

  - **No SSE event for orch transitions.** The
    /api/v1/stream endpoint emits `finding`,
    `run_start`, `run_end` events but no
    `scan_state_change`. A future chunk can add
    that, after which renderScans's poller can be
    replaced (or augmented) with an SSE listener.
  - **Submit form fields are minimal**: input,
    plugin, default_port. The /api/v1/scans request
    shape supports `plugins` as an array but the
    form constrains it to one plugin (matching
    v1.61's single-plugin runner). Multi-plugin UI
    lands when multi-plugin runner does.
  - **No paging**: fixed at top-20 newest-first.
    Adding `?since=...` cursor handling is
    cheap when the operator volume requires it.
  - **No filtering**: state / operator / plugin
    filters are deferred. Dashboard renders all 20
    most recent regardless.
  - **Authorisation**: scan-trigger sits behind the
    existing dashboard auth (none in dev; whatever
    operator wired upstream of /api/v1/). A
    separate "viewer-vs-operator" scope is
    deferred.
