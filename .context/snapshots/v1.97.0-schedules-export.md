# Snapshot — v1.97.0: schedules export (CSV/NDJSON/JSON)

Single-chunk cycle. Operators with 100+ schedules want a
backup file before a DB migration / cluster move / org
transfer. v1.97 adds a one-call export endpoint.

## REST

```
GET /api/v1/schedules/export
GET /api/v1/schedules/export?format=csv
GET /api/v1/schedules/export?format=ndjson
GET /api/v1/schedules/export?format=json    # default
```

  - `csv` — 10-column flat table (id, name, cadence,
    enabled, operator, created_at, last_fired_at,
    audit_retention_days, input, plugins). Cadence
    rendered as `interval=Ns` or `cron=expr (tz)`.
    `Content-Disposition: attachment; filename="elsereno-schedules.csv"`.
  - `ndjson` — one canonical JSON per line. Round-trippable
    via `cat | jq -c '.' | xargs -I {} curl -XPOST -d '{}'`.
    `Content-Disposition: attachment; filename="elsereno-schedules.ndjson"`.
  - `json` — back-compat with /schedules list shape.

400 on unsupported format.

## Honest scope

  - CSV is intentionally lossy (loses nested JSON like
    multiple plugins; the column joins them with "|").
    Operators wanting fidelity should use NDJSON.
  - No server-side restore endpoint — the recipe is:
    `cat backup.ndjson | jq -c '.' | xargs -I{} curl ...`
    Server-side would require disambiguating "create
    new" vs "update existing" semantics; deferred to a
    future cycle (also: backup files come from another
    deployment + the operator wants to choose the
    conflict-resolution rule).
  - INSTALL.md unchanged. Build sizes unchanged.
