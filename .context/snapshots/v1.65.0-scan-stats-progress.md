# Snapshot — v1.65.0: scan_stats_progress SSE event

Single-chunk cycle. Continues the dashboard scan-
orchestration line. v1.63 added scan_state_change so
state transitions land in real time. v1.65 closes
the in-flight visibility gap: the counters in the
dashboard's Targets/Findings cell now tick live as
the worker drains scanner.Run output, instead of
staying at 0 until the job hits the terminal state.

## Why a separate cycle

A 100k-target scan can run for minutes. Without
mid-run progress, the operator sees `running` +
0/0 · 0 findings the entire time, with no signal
that work is actually happening. The state-change
event from v1.63 only fires on the
queued→running and running→completed transitions —
nothing in between. v1.65 adds the third event
kind to fill the gap.

## Architecture

```
defaultScanRunner.Run drains scanner.Run output
    │
    ├── per-event: report(currentStats)
    │       │
    │       ▼
    │   Worker.OnProgress closure
    │       │
    │       ▼
    │   ScanProgressThrottle.Report(jobID, stats)
    │       │   ├── if within window + identical → drop
    │       │   ├── if within window + different → hold (no emit)
    │       │   └── if past window or first → emit
    │       │
    │       ▼
    │   PublishScanProgress(broadcaster, ...)
    │       │
    │       ▼
    │   SSE EventScanProgress
    │       │
    │       ▼
    │   Dashboard JS listener → updates row's
    │   Targets/Findings cell IN PLACE
    │
    ├── on terminal transition (v1.63 BroadcastingStore):
    │   throttle.Forget(jobID) ← keeps the per-job
    │                             map bounded over a
    │                             long-running serve
```

## Chunks

### Chunk 1 — interface + reporter + throttle + UI (`b7f8158`)

`internal/scanorch/worker.go`:

  - **Add ProgressReporter** type `func(Stats)`.
  - **Change JobRunner.Run** signature from
    `Run(ctx, job) (Stats, error)` to
    `Run(ctx, job, report) (Stats, error)`. Breaking
    API change for external runners; the only
    in-tree consumer is defaultScanRunner.
  - **Add Worker.OnProgress** field
    `func(jobID string, stats Stats)`. Worker
    builds a closure from this and passes to
    Runner.Run; nil → no-op reporter so runners
    that call report() don't blow up.

`cmd/elsereno/scan_runner.go`:

  - **Add report parameter** to
    defaultScanRunner.Run.
  - **drainPluginRun gains an emit closure** —
    called after every (finding | error) channel
    event. The closure builds a fresh Stats from
    the shared atomic counters + invokes the
    reporter unconditionally.
  - **No throttling in the runner**. The listener
    side decides cadence.

`internal/web/stream/scan_bridge.go`:

  - **EventScanProgress** event kind.
  - **PublishScanProgress(b, jobID, stats)** —
    nil-safe helper.
  - **ScanProgressThrottle** struct with
    per-job last-emit timestamp + last-emitted
    Stats. Three suppression cases:
      1. Identical snapshot → drop.
      2. Within minInterval window + different
         stats → hold (no emit).
      3. Past window or first call → emit.
    Default 500ms; clamp range [50ms, 5s].
  - **Throttle.Report(jobID, stats)** is the
    closure target wired into Worker.OnProgress.
  - **Throttle.Forget(jobID)** clears per-job
    state — called from BroadcastingStore on
    terminal transitions (so a long-running
    serve doesn't grow the per-job map).
  - **BroadcastingStore.AttachProgressThrottle**
    wires the Forget hook into the existing
    Transition decorator.

`cmd/elsereno/cmd_serve.go`:

  - Build a NewScanProgressThrottle alongside
    the BroadcastingStore. Attach to the store
    so terminal transitions Forget. Wire
    OnProgress to throttle.Report.

`internal/web/handlers/dashboard.go`:

  - JS scan_stats_progress listener updates the
    affected row's Targets/Findings cell **in
    place** (single innerHTML swap) rather than
    triggering a full table re-render. Each event
    is cheap.
  - renderScans() stamps each row with
    `data-scan-id` + the stats cell with
    `data-scan-stats` so the listener can find
    the right cell.

## Throttle correctness

The three suppression cases compose correctly:

  - **Burst of 1000 events in 100ms**: all
    suppressed except the first; the held snapshot
    is the most recent.
  - **Slow trickle (every 600ms)**: each emits
    because each is past the 500ms window.
  - **Stable Stats (drained errors only)**:
    after the first emit, subsequent identical
    snapshots all drop — no spurious "still 33/100"
    events.

The held-but-not-emitted snapshot from case 2 is a
small forfeiture (the operator sees the state at
T=0 and T=window, not the in-between updates) but
acceptable for a 500ms cadence. A future cycle could
add a periodic Flush goroutine; v1.65 ships without
it because the subsequent scan event will emit
naturally as soon as the window passes.

## Honest scope

  - **No periodic Flush** for held-but-not-emitted
    snapshots. The next scan event emits naturally
    as soon as the window passes. A pathological
    case where the runner stops firing exactly at
    a window boundary leaves the held snapshot in
    the map until the terminal transition Forgets
    it — the snapshot just gets dropped. Future
    cycle could add a 1Hz flush ticker.
  - **No per-job rate accounting**. The dashboard
    can't show "currently scanning at 50/sec" — it
    shows the latest counter snapshot. Live
    scan-rate would need a derivative + a separate
    UI element.
  - **No back-pressure**. If the SSE bus is
    saturated (many subscribers, slow clients),
    the broadcaster drops slow subscribers.
    Throttle doesn't know about that. Acceptable;
    the broadcaster's slow-subscriber drop is the
    pressure-release valve.
