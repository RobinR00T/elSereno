# Snapshot — v1.84.0: force-overwrite audit log

Single-chunk cycle. v1.81/v1.83 let an operator skip the
v1.78 optimistic-locking precondition via "Force overwrite".
v1.84 adds a persistent audit trail so a downstream operator
can see who overrode whom + what changed.

## Why

Trust + accountability. Multi-operator deployments are
defenceless against silent overrides — even with the merge-
view UI, an operator who clicks Force overwrite leaves no
record. v1.84 mints a per-event before/after snapshot so
the audit log can answer "who changed this schedule's cron
expression on Tuesday?".

## API

```go
type ScheduleAuditEventType string
const ScheduleAuditEventForceOverwrite = "force_overwrite"
var ValidScheduleAuditEventTypes = []ScheduleAuditEventType{...}

type ScheduleAuditEvent struct {
  ID, ScheduleID, Operator string
  EventType    ScheduleAuditEventType
  OccurredAt   time.Time
  PayloadBefore json.RawMessage
  PayloadAfter  json.RawMessage
}

type ScheduleAuditStore interface {
  Append(ctx, event) (event, error)
  ListBySchedule(ctx, scheduleID) ([]event, error)
}

var ErrScheduleAuditInvalidEventType = ...
```

REST:

```
PUT /api/v1/schedules/{id}
X-Schedule-Force-Overwrite: true
... + usual body ...

→ if X-Schedule-Force-Overwrite=true AND audit store
  non-nil, a force_overwrite audit row is persisted with
  before/after snapshots.

GET /api/v1/schedules/{id}/audit
→ 200 [event, ...] (newest first)
→ 404 if schedule missing
→ 503 if audit store nil
```

## Migration 00011

```sql
CREATE TABLE scan_schedule_audit (
  id TEXT PK, schedule_id TEXT FK CASCADE,
  event_type TEXT CHECK IN ('force_overwrite'),
  operator TEXT, occurred_at TIMESTAMPTZ DEFAULT NOW(),
  payload_before JSONB, payload_after JSONB
);
CREATE INDEX scan_schedule_audit_schedule_at_idx
  ON scan_schedule_audit (schedule_id, occurred_at DESC);
```

CASCADE on delete: deleting a schedule wipes its audit
history. Operators wanting a permanent record disable
instead of delete.

## Implementation

MemoryScheduleAuditStore (sync.RWMutex) for tests +
memory-mode deployments. DBScheduleAuditStore for
--scan-store=db.

updateSchedule wired:
  1. If X-Schedule-Force-Overwrite + audit non-nil →
     store.Get(id) snapshot before.
  2. store.Update as before.
  3. On success, json.Marshal before + after, append.
  4. Append failure → X-Schedule-Audit-Warning response
     header (best-effort; doesn't reverse the update).

Refactored to satisfy gocyclo:
  - parseIfMatchInto (v1.78 header parse).
  - writeUpdateScheduleError (404 / 412 mapping).
  - recordForceOverwriteAudit (the audit branch).

## Honest scope

- No dashboard view of the audit log yet — v1.85 carryover.
- Best-effort audit: Append failures are warning-only.
- No retention policy / pruning. Operators with high
  force-overwrite frequency need to manage retention.
- CASCADE on schedule delete = audit history wiped.
- Only force_overwrite is enumerated. Future cycles can
  expand (delete, set_enabled, etc).
