# Snapshot — v1.80.0: live preview with debounce

Single-chunk cycle. Dashboard-only JS change. v1.77/v1.79
gave operators a "Preview next fire" button; v1.80 makes
the preview live by attaching debounced event listeners to
the cadence fields.

## Why

Operator UX. The button was a manual extra step; with
live updates the operator sees the predicted fire as they
type / paste / toggle. Debouncing prevents mid-typing
parse errors from flashing.

## Implementation

```js
var previewDebounceTimer = null;
function schedulePreviewRefresh() {
  if (previewDebounceTimer) clearTimeout(previewDebounceTimer);
  previewDebounceTimer = setTimeout(function () {
    previewNextFire();
    previewDebounceTimer = null;
  }, 350);
}
["schedule-cadence-mode", "schedule-interval",
 "schedule-cron", "schedule-timezone"].forEach(function (id) {
  var el = document.getElementById(id);
  if (!el) return;
  el.addEventListener("input", schedulePreviewRefresh);
  el.addEventListener("change", schedulePreviewRefresh);
});
```

Same debounce pattern already used by v1.63's
scheduleRefresh helper for SSE-triggered panel refreshes.

The manual "Preview next fire" button (v1.77) is kept as
a force-refresh shortcut.

## Why 350ms

- < 200ms feels twitchy (parse errors flash mid-edit).
- > 500ms feels sluggish.
- 350ms = comfortable typing rhythm + enough time to
  finish a cron expression like `0 9 * * 1-5` before
  the preview attempt.

## Tests

3 dashboard markers: `previewDebounceTimer`,
`function schedulePreviewRefresh`, `schedule-cadence-mode`.
The preview path itself is exercised by v1.77/v1.79
tests — the debounce is timer plumbing.

All build + lint + tests clean.

## Honest scope

- No opt-out. Operators wanting silence leave the form
  alone.
- No request cancellation. An in-flight /preview when
  the debounce fires a new one will complete (its
  result briefly flashes before the next replaces it).
  Visible only on sub-350ms field rewrites.
- No "preview pending…" indicator. Text panel shows
  the previous result mid-debounce.
