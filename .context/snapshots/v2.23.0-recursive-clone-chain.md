# Snapshot — v2.23.0: recursive clone-chain ?depth=N

Single-chunk cycle. v2.10 `/clones` only returns direct
children (one hop). v2.23 walks the chain N levels deep.

## REST

```
GET /api/v1/schedules/{id}/clones?depth=N   # N in [1, 10]
```

  - `depth` default 1 (matches v2.10 behaviour).
  - Clamped server-side to `[CloneDepthMin=1, CloneDepthMax=10]`.
  - `depth=0` or non-integer → 400.

Each returned schedule carries `clone_depth` (BFS distance
from the queried source). Result sorted by depth ASC then
name ASC so dashboards can render generation-grouped trees.

## Store

  - New `ListClonesOfDeep(ctx, sourceID, depth)` on the
    interface.
  - `ListClonesOf` is now a thin wrapper passing depth=1.
  - **Memory**: BFS over the schedule map, stamping
    `CloneDepth` on each result.
  - **PG**: recursive CTE bounded by `WHERE c.clone_depth <
    $2`. Uses the v2.10 partial index for the base case + a
    join in the recursive step. New
    `scanScheduleWithDepth(rows)` helper extends `scanSchedule`
    for the +1-column projection.

## Model

  - `ScanSchedule.CloneDepth int` (JSON omitempty).
    NOT persisted; computed-at-read only.

## Tests (+2)

  - TestListScheduleClones_Deep: 4-schedule chain (root →
    child → grand → great-grand). depth=1 → only child;
    depth=3 → all three descendants with CloneDepth = 1/2/3
    respectively.
  - TestListScheduleClones_DeepBadDepth: depth=0 → 400.

## Honest scope

  - Cycles impossible by construction (source_schedule_id
    is set at clone time + can't be retargeted via Update).
    Walker still tracks visited IDs defensively in case
    data ever gets corrupted manually.
  - INSTALL.md unchanged. Build sizes unchanged.

## Build sizes

Unchanged across all 3 variants.

Adds ~120 LoC + 2 tests.
