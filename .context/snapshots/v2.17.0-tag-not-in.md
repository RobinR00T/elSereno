# Snapshot — v2.17.0: NOT operator on tag filter

Single-chunk cycle. Closes v2.9 carryover (NOT operator).

## REST

```
GET /api/v1/schedules?tag=dev&tag=staging&op=not_in
```

  - `op=not_in` → schedule must carry NONE of the supplied
    tags. Untagged schedules match (empty intersection).
  - 400 expanded to mention the third op
    (`'and', 'or', or 'not_in'`).
  - Single-tag + `op=not_in` bypasses the v2.4 ListByTag
    fast path so semantics stay correct.

## Store

  - New constant `scanorch.TagOpNotIn = "not_in"`.
  - **Memory**: `matchesTags` gains the third case;
    no-overlap test by walking `want` against the
    schedule's tag set.
  - **PG**: predicate becomes
    `NOT (tags && $1::text[])`. The GIN index doesn't
    accelerate negation; planner falls back to seq scan
    + filter (acceptable for typical fleets <10k).

## Tests (+2)

  - TestListSchedules_TagNotIn: multi-tag exclude →
    untagged + non-matching schedules survive.
  - TestListSchedules_TagNotIn_SingleTag: single-tag +
    op=not_in bypasses v2.4 fast path correctly.

## Honest scope

  - GIN can't accelerate NOT predicates. For very large
    fleets, a CTE flipping the question may perform
    better; deferred until someone hits the wall.
  - INSTALL.md unchanged. Build sizes unchanged.

## Build sizes

Unchanged across all 3 variants.

Adds ~60 LoC + 2 tests.
