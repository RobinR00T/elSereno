# Snapshot — v2.29.0: fetchWithETag on /audit + /runs views

Single-chunk cycle. v2.15 introduced fetchWithETag; only
the tag-cloud was wired. v2.29 extends to the audit
history + runs views so opening these panels twice
(common during incident triage) skips body re-render via
304.

## Wired

  - **openAuditView**: `fetchWithETag(/audit)`. 304 →
    no-op (existing table stays). 503 → unchanged
    "audit log unavailable" message via the .catch
    `err.status === 503` branch.
  - **fetchRunsPage** (used by openRunsView +
    loadMoreRuns): same pattern. Cursor pagination URLs
    each get their own ETag entry; "Load more" hits
    a different URL so 304 only fires on identical
    page re-fetches.

## Honest scope

  - No-op on 304 means a stale table stays if the
    operator triggers openAuditView via a state-bumping
    action in another tab — `fetchWithETag` correctly
    sees the new ETag and re-renders.
  - INSTALL.md unchanged. Build sizes unchanged.

## Build sizes

Unchanged across all 3 variants.

Adds ~10 LoC delta in dashboard.go.

## Carryover

None — closes the v2.15 dashboard-ETag carryover.
