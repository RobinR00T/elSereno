# Snapshot — v2.16.0: bulk tag-rename endpoint

Single-chunk cycle. Closes vNext (tag-rename) carryover.
Operators rebranding a tag had to update each schedule
individually; v2.16 lands the bulk endpoint.

## REST

```
POST /api/v1/schedules/tags/rename
Content-Type: application/json
{"from": "prod", "to": "production"}
```

Response:
```json
{
  "schema": "api:v1",
  "data": {
    "from": "prod",
    "to": "production",
    "renamed_count": 17
  }
}
```

  - Both `from` and `to` are required + shape-validated
    (v2.4 rules: 1..32 chars, [a-z0-9_-] only).
  - `from == to` → 400 (ErrTagRenameNoOp).
  - Schedules already carrying `to` get the duplicate
    deduped (canonical tags slice survives).
  - `updated_at` bumped on every mutated row so v1.78
    optimistic-locking clients see the change.

## Store

  - New `Store.RenameTag(ctx, from, to)` method on the
    ScheduleStore interface.
  - Sentinels: `ErrTagRenameRequiresBoth`,
    `ErrTagRenameNoOp`.
  - **Memory**: walks the map, swaps in-place, runs
    canonicaliseTags to dedup + sort.
  - **PG**: single UPDATE with
    `ARRAY_REPLACE(tags, $1, $2)` + unnest +
    `array_agg(DISTINCT t ORDER BY t)` to canonicalise.
    `WHERE $1 = ANY(tags)` skips untouched rows. GIN
    index from v2.4 makes the WHERE clause cheap.

## Tests (+3)

  - TestRenameTag_HappyPath: 'prod' → 'production'
    across 3 schedules; only the one needing the swap
    is reported as renamed; canonical sort + dedup
    verified.
  - TestRenameTag_NoOp: from == to → 400.
  - TestRenameTag_BadShape: malformed `to` → 400.

## funlen refactor

`APIV1` was getting close to the 40-statement cap as
the schedule surface grew. Extracted
`mountScheduleRoutes(mux, deps)` helper — declarative
list of route strings looped over `mux.Handle`. Adding
a new schedule route in v2.17+ is now one line in the
slice instead of a `Handle` call + a statement count
audit.

## Honest scope

  - No bulk-rename audit log entry — the operation
    isn't surfaced as ScheduleAuditEvent because each
    schedule's audit is keyed by schedule_id, not
    "across-fleet operations". Operators tracking
    tag-rename events should keep their own log; the
    `updated_at` bump on mutated rows gives them a
    timestamp anchor.
  - No reverse operation (rename via swap requires two
    calls + an intermediate placeholder).
  - INSTALL.md unchanged. Build sizes unchanged.
