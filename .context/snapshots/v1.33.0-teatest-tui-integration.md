# Snapshot — v1.33.0: teatest TUI integration tests

Single-chunk cycle. Closes the v1.30 + v1.31 carryover that
explicitly deferred "teatest program-level integration tests
for the TUI runner".

## Why a tiny cycle

The component-level tests (53 cases across model_test.go,
update_test.go, filter_test.go, view_test.go, plus the
feed-level tests) pin the public surface of every Model
mutation, every Update branch, and every filter helper. What
they don't cover is the bubbletea program itself: the
WindowSizeMsg dispatch on first render, the keypress wire-up
through tea.Program, the program lifecycle (Init → loop →
Quit → final model). teatest is the upstream-supported
harness for exactly that gap. One chunk lands it without
mixing in unrelated work.

## Chunks

### Chunk 1 — teatest integration tests (`29ecaf4`)

`internal/tui/program_test.go` (≈220 lines) drives the model
through teatest with the production NewModel constructor,
sends synthetic events, asserts on rendered bytes + final
model state. The runner.Run wrapper is bypassed because
teatest manages the program lifecycle itself; the feed
goroutine is replaced by direct tm.Send calls so each test
stays deterministic.

10 cases:

| Test                                              | What it pins |
|---------------------------------------------------|--------------|
| TestProgram_QuitsOnQ                              | q triggers Quitting=true via tea.Quit |
| TestProgram_QuitsOnCtrlC                          | ctrl+c likewise |
| TestProgram_RendersHeaderAndPanes                 | All 4 pane labels visible on first render |
| TestProgram_RendersFindingMsg                     | FindingMsg → protocol name in output |
| TestProgram_RendersAuditMsg                       | AuditMsg → line in audit pane |
| TestProgram_FilterEditCycle                       | Tab×2 → audit pane, /scan + Enter, AuditFilter committed |
| TestProgram_TabCyclesFocus                        | 3× Tab → PaneScan |
| TestProgram_FindingMsg_BumpsTriageCount           | Score 95 → "critical" chip in triage pane |
| TestProgram_TerminalTooSmall                      | 20×5 surfaces friendly message |
| TestProgram_FullSession_FinalOutputIsCleanASCII   | Full session drains without panic/runtime/goroutine leaks |

Configuration constants:

```go
const (
    testTermWidth  = 120  // wide enough for full layout
    testTermHeight = 40   // tall enough for header+4 panes+footer
    waitDuration   = 2 * time.Second   // teatest WaitFor budget
    waitCheckTick  = 10 * time.Millisecond
)
```

The 2s WaitFor budget is generous on slow CI runners but tight
enough that a real regression doesn't hang the suite. Local
runs complete in <100 ms total.
