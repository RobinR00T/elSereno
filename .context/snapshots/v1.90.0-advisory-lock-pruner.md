# Snapshot — v1.90.0: advisory-locked audit pruner

Single-chunk cycle. Closes the v1.87 / v1.89 carryover: when
multiple `serve` instances share a Postgres DB, the audit
pruner can race — each instance issues identical DELETEs at
the tick boundary, idempotent but wasteful + double-counts
the OnPrune callback. v1.90 serialises this with a Postgres
`pg_try_advisory_xact_lock` so only one instance runs the
prune per cutoff.

## Why

- Multi-process serve deployments (HA, blue/green, k8s rolling)
  are the real-world topology v1.87 didn't defend against.
- The DELETE is idempotent, but per-tick log noise (3×
  OnPrune lines for 3 replicas) confuses ops + drowns
  legitimate signals.
- Advisory lock auto-releases on tx commit/rollback (or
  session end on crash) — no stale-lock recovery story.

## Wire-up

`ScheduleAuditStore` stays back-compat. New OPTIONAL
interface:

```go
type AdvisoryLockedAuditStore interface {
    ScheduleAuditStore
    PruneWithLock(ctx, key int64, globalCutoff, overrides) (count int64, acquired bool, err error)
}
```

`AuditPruner` gains:

```go
AdvisoryLockKey int64                // 0 = disabled
OnLockSkipped   func(key int64)      // observability hook
```

`runPrune` dispatches:

  - lock + lock-supported store → PruneWithLock.
  - overrides + non-locking store → PruneWithOverrides.
  - neither → PruneOlderThan.

Stable bigint constant `AuditPrunerLockKey = 4651912875610038017`
(SHA-256 of "elsereno-audit-pruner" truncated) so all serve
instances against the same DB agree on the key without
configuration.

## PG impl

`DBScheduleAuditStore.PruneWithLock`:

  1. Type-assert `s.q` to `txQuerier` (Pool's BeginTx). Fake
     querier in tests → falls back to PruneWithOverrides +
     reports acquired=true.
  2. `BeginTx(ctx, pgx.TxOptions{})`.
  3. `SELECT pg_try_advisory_xact_lock(key)` → acquired bool.
  4. acquired=false → return (0, false, nil) — caller's
     OnLockSkipped fires.
  5. acquired=true → call `pruneAuditUsing(ctx, tx, …)`
     (factored common body shared with PruneWithOverrides),
     `tx.Commit(ctx)`.
  6. `defer tx.Rollback(ctx)` is safe — idempotent after
     Commit.

The lock auto-releases at COMMIT. No leaked locks even on
process crash (PG session ends → lock released).

## cmd_serve

`startAuditPruner` now sets:

```go
AdvisoryLockKey: scanorch.AuditPrunerLockKey,
OnLockSkipped:   func(key) { fprintf(stderr, "...") },
```

Memory mode: MemoryScheduleAuditStore doesn't implement
AdvisoryLockedAuditStore → silent fallback to v1.89
PruneWithOverrides. Zero behavioural change for single-
process or in-memory deployments.

## Tests (+3 unit)

  - `TestAuditPruner_AdvisoryLockKey_FallsBackOnMemory`:
    memory store + AdvisoryLockKey>0 → no skip fires,
    prune runs unguarded.
  - `TestAuditPruner_AdvisoryLockKey_LockNotAcquired`:
    fakeLockingAuditStore with acquired=false → OnLockSkipped
    fires once, OnPrune does NOT.
  - `TestAuditPruner_AdvisoryLockKey_LockAcquired`:
    happy path with acquired=true + pruneCount=42 → OnPrune
    receives 42, OnLockSkipped does NOT fire.

Plus a `fakeLockingAuditStore` test stub exercising the
type-assertion path.

All build (default + offensive + mini) + lint (macOS +
GOOS=linux) + tests + race clean.

## Honest scope

- True multi-process semantics need a real Postgres to
  verify. The unit tests cover the dispatcher logic + the
  fall-back paths; PG-side behaviour relies on
  `pg_try_advisory_xact_lock` documented semantics (which
  are battle-tested).
- The key is one global advisory lock per database; multiple
  audit-pruners against the SAME DB share the lock (correct).
  Pruners against DIFFERENT DBs are independent (also
  correct).
- INSTALL.md unchanged. No CLI flag added (it's a deployment-
  topology concern, not an operator-tunable in the usual
  sense). Operators wanting to opt OUT can run with
  `--scan-store=memory` (no PG → no lock needed).
