# Snapshot — v1.52.0: s7 per-(area, db, byte-address) gating

Single-chunk cycle. Closes the v1.32+ A carryover.
Tightens the s7 write-gate from FC-only (v1.27) to
per-(area, db, byte-address-range) granularity for
FuncWriteVar (0x05). Prevents an allowlisted WriteVar
from being weaponised to mutate ANY address — operators
can now restrict it to "DB42 bytes 100..200" or "M area
only".

## Why

S7-based ICS deployments routinely allow WriteVar for
legitimate engineering ops (parameter tuning, recipe
loads). FC-only gating (v1.27) means that if an
operator authorises WriteVar at all, an attacker who
got that token can write to ANY address — including
safety-critical DBs. Per-address gating closes that gap
without forcing operators to abandon WriteVar entirely.

## Chunks

### Chunk 1 — wire parser + gate (`829b769`)

`internal/protocols/s7/wire/items.go` (new):

  - **WriteItem** struct: Area / DB / ByteAddr /
    Length.
  - **ParseWriteVarItems** walks the param area's item
    count + N × 12-byte S7ANY headers. Computes
    ByteAddr = bit-addr >> 3 (24-bit address field
    treated as bit-aligned). Length scales count by
    transport-size byte (BIT/BYTE=1, WORD/INT=2,
    DWORD/DINT/REAL=4, OCTET STRING=1, unknown=1
    conservative).
  - 4 sentinel errors (ShortPDU, BadFC, ShortItem,
    UnknownSyntax). Gate refuses on any.
  - S7ANY syntax id (0x10) only — DB-symbolic / NCK /
    Drives ES syntaxes are vendor-specific and rare;
    refused.

`offensive/write/s7/gatedproxy.go`:

  - **AllowedWriteItem**{Area, DB, AddrStart, AddrEnd}
    with `Matches(item)` — same area, same DB (for
    DB/DI areas), item byte-range fits within
    [AddrStart, AddrEnd]. M / I / Q / L / V areas
    ignore DB.
  - **AllowlistHash** signature gains items dimension.
    Hash separator 0xF1 (high range, can't collide
    with FC values 0x00..0x29) introduces the per-
    item block. Empty items list yields the v1.27
    hash — pre-v1.52 confirm-tokens keep validating.
  - **SessionMutationLegacy(target, allowed)**
    preserved for any caller that genuinely doesn't
    have items.
  - **WriteGatedHandler.AllowedWriteItems** field;
    empty list = v1.27 behaviour, non-empty activates
    per-item gate.
  - **shouldForward** refactored: extracted
    `fcAllowed`, `writeItemsAllowed`, `itemAllowed`
    helpers. Per-item check fires only for
    FuncWriteVar with non-empty AllowedWriteItems.

## Multi-item refusal semantics

A WriteVar request that targets 3 items refuses the
whole frame if ANY item falls outside the allowlist.
Partial allow would be confusing — the WriteVar is one
transaction. Operator who wants 2 of the 3 items
through configures the allowlist to cover all 3 or
splits into separate requests.

## Backward compat

Empty `AllowedWriteItems` keeps v1.27 behaviour: the
gate evaluates only `AllowedFunction`, and
`AllowlistHash` returns the same bytes as `Session-
MutationLegacy(target, allowed)`. So:

  - Operator who hasn't configured per-item allowlist:
    no behavioural change, pre-v1.52 confirm-tokens
    still validate.
  - Operator who DOES configure per-item allowlist:
    must re-mint confirm-token (the per-item bytes
    fold into the hash).

## Honest scope

  - **Bit-level addressing** (DB42:100.3 = byte 100
    bit 3) is intentionally NOT supported. Operators
    allowlist DBs and byte ranges; bit-position
    guarantees are out of scope.
  - **CLI flag plumbing** (`proxy listen
    --write-item area=DB,db=42,start=100,end=200`)
    is NOT in this chunk. The wire + gate primitives
    are the foundation; flag plumbing in
    cmd_proxy_offensive.go is a future cycle's add.
  - **Other write FCs** (PLCStop, RequestDownload,
    DownloadBlock, …) are NOT constrained by
    AllowedWriteItems — those don't carry an item
    list. They keep FC-level gating via
    AllowedFunction.

## Hash separator allocation

0xF1 — first separator allocation since v1.13's
BACnet block. Reserved range 0xF1..0xFF is for
write-gate dimension separators that can't collide
with protocol-specific function/service codes.
