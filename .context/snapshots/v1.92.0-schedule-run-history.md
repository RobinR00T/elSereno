# Snapshot — v1.92.0: schedule run history

Single-chunk cycle. Operators previously had no way to ask
"show me the last 10 runs of schedule X" — the only signal
was `last_fired_at` (a single timestamp) on the schedule
itself. v1.92 adds a linkage from scheduler-fired jobs back
to their originating schedule + a REST endpoint + a Runs
button in the dashboard.

## Schema (migration 00014)

```sql
ALTER TABLE scan_jobs
  ADD COLUMN triggered_by_schedule_id TEXT NULL
    REFERENCES scan_schedules(id) ON DELETE SET NULL;

CREATE INDEX idx_scan_jobs_triggered_by_schedule
  ON scan_jobs (triggered_by_schedule_id, created_at DESC)
  WHERE triggered_by_schedule_id IS NOT NULL;
```

Partial index keeps the table size cost zero for manual scans
(operator's NULL rows don't enter the index) — only
scheduler-fired rows.

ON DELETE SET NULL preserves the scan_jobs row when its
originating schedule is deleted (matches v1.88 audit
pattern).

## Go

`Job.TriggeredByScheduleID string` (JSON
`triggered_by_schedule_id,omitempty`).

`Store` interface gains 2 methods:

  - `SubmitFromSchedule(ctx, req, op, scheduleID)` — variant
    of Submit that records the linkage. Submit is now a thin
    wrapper passing "".
  - `ListBySchedule(ctx, scheduleID, limit)` — newest-first
    jobs for the schedule; capped at limit (default 50, max
    1000).

Memory + DB store impls.

`Scheduler.fire` now calls SubmitFromSchedule(sched.ID).

`BroadcastingStore` (web/stream) propagates both new methods
through to the inner store with the same broadcast
semantics (Submit publishes; ListBySchedule is pure-read).

## REST

```
GET /api/v1/schedules/{id}/runs
GET /api/v1/schedules/{id}/runs?limit=100
```

Returns `[]scanorch.Job` newest-first. 404 if schedule
unknown. 503 if scan store is nil (memory-only deployments
that disabled persistence). Empty `[]` when schedule has
never fired.

## Dashboard

  - New "Runs" button on each schedule row (next to History).
  - New `#schedule-runs-view` panel with table (When / State

## Honest scope

  - No back-fill of pre-v1.92 jobs. Operator can't see
    historical schedule→job linkage; rows pre-dating the
    migration have triggered_by_schedule_id = NULL.
  - Pagination is single-limit (no cursor). N=50 default is
    plenty for "what fired today" inspection; "give me the
    whole year" use case would justify a cursor in a future
    cycle.
  - INSTALL.md unchanged. Build sizes unchanged.
