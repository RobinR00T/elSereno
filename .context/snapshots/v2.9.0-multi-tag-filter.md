# Snapshot — v2.9.0: multi-tag AND/OR filter

Single-chunk cycle. Closes v2.6 carryover. v2.4 added
single-tag filter (`?tag=critical`); v2.9 extends to
multi-tag with boolean composition.

## REST

```
GET /api/v1/schedules?tag=prod&tag=critical            # AND default
GET /api/v1/schedules?tag=prod&tag=critical&op=and     # explicit AND
GET /api/v1/schedules?tag=prod&tag=critical&op=or      # OR
```

  - `?op=and` (default) → schedule must carry ALL of the
    tags.
  - `?op=or` → schedule must carry AT LEAST ONE of the
    tags.
  - Unknown `?op=` value → 400.
  - Zero `?tag=` → full list (back-compat).
  - Single `?tag=` → ListByTag path (v2.4 fast path).

## Store interface

New method: `ListByTags(ctx, tags []string, op string)`.
v2.4 `ListByTag` becomes a thin wrapper passing
`[]string{tag}` + `TagOpAnd`. New constants
`scanorch.TagOpAnd` + `scanorch.TagOpOr`.

  - **Memory**: `matchesTags` helper builds a set from the
    schedule's tags + walks `want` per op.
  - **PG**:
    - AND → `tags @> $1::text[]` (contains all).
    - OR  → `tags && $1::text[]` (overlaps).
    Both GIN-indexed.

## Tests (+3)

  - INSTALL.md unchanged. Build sizes unchanged.
