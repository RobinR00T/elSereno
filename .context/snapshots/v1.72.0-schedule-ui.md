# Snapshot — v1.72.0: dashboard schedule UI panel

Single-chunk cycle. Operators no longer need curl
to manage scheduled scans. New "Scheduled scans"
panel lives between the scan-jobs table and the
reload-cadence panel. CRUD + Enable/Disable + Delete
via fetch() against /api/v1/schedules.

## Why a separate cycle

v1.70 + v1.71 shipped the backend (model + REST +
DB persistence). The UI was deferred to keep each
backend cycle scope-bounded. v1.72 is the
operator-facing close — the schedule feature is now
fully usable without leaving the dashboard.

## Panel layout

```
Scheduled scans v1.70+ · /api/v1/schedules
  ─ Saved Job templates that fire on an interval...
  [name__] [input________] [plugin(s)____]
  [interval (s) 3600] [Create]   status…

  | Name | Input | Plugins | Interval | State |
    Last fired | Action            |
  | ... | ...   | ...     | 6h       | enabled |
    2026-05-08 13:00 | [Disable] [Delete] |
```

## JS additions

  - `renderSchedules()` — fetches /schedules + populates
    table. 503 from the API surfaces a hint
    ("scan orchestration disabled — start serve with
    --scan-store=memory or =db") instead of an empty
    table.
  - `submitSchedule(ev)` — POST handler with input
    validation (interval clamp Go-side mirrored on
    JS-side as a defence: 60..604800).
  - `toggleSchedule(id, action)` — POST to
    /enable or /disable based on data-attr on the
    button.
  - `deleteSchedule(id)` — DELETE with
    `window.confirm()` prompt (irreversible).
  - `humanInterval(secs)` — renders 60→"1m",
    3600→"1h", 86400→"1d" for the Interval column.
    Falls through to "Ns" for non-canonical values.
  - `scheduleSchedulesPoll(delayMs)` — 30s default
    cadence; 60s on 503/error. Schedules change
    rarely so polling is intentionally slower than
    the scans table.

The plugin-name field reuses the v1.68 `<datalist
id="scan-plugin-options">` — single source of
plugin autocomplete across the whole page. No need
to load the plugin list twice.

## Tests (+11 new dashboard markers)

`TestDashboard_ContainsScanPanel` gained 11 markers
covering panel id, form ids, table ids, JS function
names, and the /api/v1/schedules API path.

A regression that strips any of these (e.g., a
template refactor that drops the panel) fails the
test loudly.

## Honest scope

  - **No edit form.** Schedules are immutable once
    created — operators delete + recreate to change
    a template or interval. Acceptable: schedule
    edits are rare; an edit modal would add
    complexity for limited gain. v1.73+ could add
    a minimal "edit interval" path.
  - **No bulk delete.** One-at-a-time. v1.74+ if
    operator demand surfaces.
  - **No SSE for schedule events.** The polling
    timer is enough — schedule mutations are
    operator-paced, not high-frequency. The next
    fire surfaces as a `scan_state_change` event
    on the scans panel anyway.
