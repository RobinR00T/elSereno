# Snapshot — v1.59.0: scan-job Worker + JobRunner + Pool + cancel

Single-chunk cycle. Continues the dashboard scan-
orchestration feature opened in v1.58. v1.58 landed
the orchestration **shell** (model + state machine +
in-memory store + REST submit/get/list); v1.59 lands
the **execution machinery** that drives queued jobs
through to terminal states.

## Why a separate cycle

The v1.58 shell intentionally stopped at the contract
boundary so the API could get exercised before the
worker arrived. v1.59 is the natural follow-up: the
state-machine API is unchanged, so v1.58 clients keep
working; the new pieces are additive (Worker + Pool +
JobRunner interface + Cancel endpoint).

## What landed

### Worker (orchestration core)

`Worker.Process(ctx, jobID)`:

  1. Claims the job (queued → running). Refuses if
     non-queued — protects against double-claims when
     multiple workers race for the same job.
  2. Calls `Runner.Run(ctx, job)` with panic recovery.
  3. Terminal transition based on outcome:
     - ctx cancelled → StateCancelled
     - runner error → StateFailed (with partial Stats
       preserved so operator sees scan progress)
     - else → StateCompleted (with final Stats)

The terminal transition uses a fresh background
context (5s timeout) so a cancelled parent ctx still
records the cancellation in the store.

### JobRunner interface

```go
type JobRunner interface {
    Run(ctx context.Context, job Job) (Stats, error)
}
type JobRunnerFunc func(ctx context.Context, job Job) (Stats, error)
```

Plus `JobRunnerFunc` adapter for plain functions in
tests / one-offs. The orchestration shell stays
decoupled from `internal/scanner` + `internal/inputs`
concrete types, so:

  - Tests supply a fake runner without dragging the
    plugin registry into scanorch's test graph.
  - Future runners (sandboxed-subprocess / sharded /
    etc.) swap in without touching Worker internals.

### Worker.ProcessAll + Worker.Drain

`ProcessAll(ctx)` — drain every queued job in one
sweep. Skips non-queued jobs. The simple "scheduling
strategy" for ad-hoc workers.

`Drain(ctx, pollInterval)` — continuous polling loop.
The default "background worker" a `serve` binary
spawns. pollInterval bounded [50ms, 1h]; defaults to
1s for out-of-range values (prevents 0-duration CPU
pegging AND silently-dead workers via absurd
intervals).

### Pool — bounded concurrent dispatch

`NewPool(worker, concurrency)` wraps a Worker with a
goroutine pool. Concurrency clamped to [1, 64].
`Submit(ctx, jobID)` dispatches via the next free
goroutine; `Stop()` terminates cleanly.

Closed-pool race fix: Submit fast-paths the closed
check first, so a buffered send can't beat the close
signal. Without this, `Stop` followed by `Submit`
might spuriously succeed before the close drained.

### Cancel endpoint

`POST /api/v1/scans/{id}/cancel` — operator-initiated
cancellation. Routes through `Store.Transition(...,
StateCancelled, ...)`. Returns:

  - 200 + Job envelope on success
  - 404 if no job has the ID
  - 409 if already terminal (the
    ErrInvalidTransition case)

For running jobs, the cancellation is a soft signal —
the dashboard transitions the state but the runner
keeps going until its ctx is cancelled separately.
Tightening this (e.g., a per-job ctx cancellation
hook from the cancel handler) is a future cycle.

## Honest scope

  - **Real runner**: this cycle ships the contract +
    a fake runner for testing. The actual runner that
    parses Job.Input, loads plugins from the registry,
    and calls scanner.Run lives in `cmd/elsereno`
    (where the imports already exist). Wiring that
    runner into a serve binary is the next cycle.
  - **Cancel→runner-stop signal**: the cancel
    endpoint flips state but doesn't reach into the
    running goroutine. The runner will eventually see
    ctx.Done if the operator-side ctx propagates;
    a tighter per-job ctx hook is deferred.
  - **Persistent storage**: still in-memory via
    MemoryStore. v1.60 chunk adds the DB-backed
    Store.
  - **Live progress reporting**: Runner.Run returns
    final Stats. Dashboards that want a live counter
    (refresh every 500ms) need a future runner shape
    (`progress chan<- Stats` argument or callback).
