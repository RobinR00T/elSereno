# Snapshot — v1.36.0: dashboard --input parity (preview endpoint)

Single-chunk cycle. Closes the v1.31 carryover that flagged
"Dashboard `--input` parity with `scan` + `tui`" as deferred.

## Why a tiny cycle

The CLI verbs `scan` (v1.31 chunk 1) and `tui --input KIND`
(v1.31 chunk 1) accept the same 8 input kinds via the
shared `parseInput` dispatcher. The dashboard had no way
to verify an input file (or even know what targets a list:/
nmap: file would resolve to) without dropping back to the
CLI. v1.36 closes that gap with a read-only preview
endpoint.

The cycle delivers a CONSERVATIVE scope: read-only preview,
no scan-running. Triggering a full server-side scan from
the dashboard would need authn-aware scan orchestration +
findings-persistence wiring + per-input rate-limit handling
for provider kinds — all of which is out of scope for a
single chunk. v1.36's output enables the future
"trigger-from-dashboard" feature without committing to its
shape yet.

## Chunks

### Chunk 1 — Preview endpoint (`d8d40e3`)

New package `internal/inputs/preview/`:

  Parse(ctx, opts) → ([]core.Target, error)

  - Dispatcher for 3 "no-credentials, no-network" kinds:
    list:FILE / nmap:FILE / stdin.
  - Provider kinds (shodan, censys, fofa, zoomeye, onyphe,
    internetdb) return `ErrUnsupportedKind` so callers can
    surface a friendly "preview supports list/nmap/stdin
    only" error.
  - Stdin field is io.Reader-injectable; defaults to
    os.Stdin when nil.

New handler `internal/web/handlers/inputs.go`:

  GET /api/v1/inputs/preview?kind=<...>&default_port=<int>

| Status | Trigger |
|--------|---------|
| 200    | Successful parse — returns `{count, targets[], truncated}` |
| 400    | Missing `kind`, unsupported `kind` (provider/bogus), bad `default_port` |
| 404    | list:/nmap: path doesn't exist (`os.IsNotExist`) |
| 500    | Parse failure inside the underlying parser |

Targets sample is capped at 200 entries — operators don't
need 50k JSON objects just to verify a file parses cleanly.
The cap is identical to the TUI's findings-tail cap so the
two surfaces feel symmetric.

Wired into `APIV1` sub-router; `inputPreviewSpecPaths()`
adds the path to the code-sourced OpenAPI spec; running
`elsereno api openapi > docs/openapi.yaml` regenerated the
snapshot.

### Why exclude provider kinds

The dashboard process intentionally runs without provider
credentials — adding shodan / censys / etc. would mean:
1. Surfacing a config field for `--api-creds-file` on the
   `serve` verb.
2. Reading a 0600 YAML at startup, persisting in memory.
3. Per-provider rate-limit tuning to avoid burning the
   operator's quota on UI clicks.

That's a substantial architectural change for a feature
operators can already get from the CLI verb. Preview stays
file-based / stdin-based; provider kinds remain a CLI
concern.
