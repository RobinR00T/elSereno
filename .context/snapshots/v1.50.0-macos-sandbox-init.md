# Snapshot — v1.50.0: macOS sandbox_init(3) cgo-gated

Single-chunk cycle. Closes the longest-standing carryover
item ("macOS sandbox via cgo break") in a way that
preserves the static-Linux invariant.

## Why opt-in cgo

The pure-Go invariant has held since project F0. Breaking
it tree-wide would have downstream costs (cross-compile,
static linking, build pipeline cgo + macOS SDK headers).
The compromise: the cgo wrapper lives in
`sandbox_darwin_cgo.go` with build tag
`offensive && darwin && cgo`. Default release builds keep
`CGO_ENABLED=0`, so:

  - Linux operators: zero change (seccomp-bpf since v1.27).
  - macOS operators using the release tarball: unchanged
    "sandbox: unavailable on darwin" behaviour.
  - macOS operators who want kernel sandbox: opt in via
    `make build-offensive-darwin-sandboxed`. Binary is
    ~23.2 MB and links the same 4 OS frameworks as the
    pure-Go macOS binary (libresolv, CoreFoundation,
    Security, libSystem) — sandbox_init lives in
    libSystem so no extra dependency surface.

## .sb Scheme profiles

Each offensive Profile maps to a hand-rolled .sb Scheme
string. All start with `(deny default)` and explicitly
allow what the subprocess legitimately needs:

| Profile  | Network             | File-write       | process-exec |
|----------|---------------------|------------------|--------------|
| exploit  | `(allow network*)`   | /tmp + /private  | DENY (no pivot back) |
| harvest  | network-outbound only| /tmp only        | DENY |
| dial     | DENY                 | /dev/tty + /tmp  | DENY |

The `(deny process-exec)` is the key shared rule: a
successful RCE on a target via the exploit profile can't
fork a shell back to the operator's host. Combined with
the existing offensive write-gate's triple-confirm fence,
this gives defence-in-depth on macOS that approaches the
Linux seccomp setup.

  - **DarwinProfileSchemesPresent** — every defined
    Profile has a non-empty .sb Scheme with the
    `(version 1)` and `(deny default)` baseline.
  - **DarwinLoadInvalidProfile** — unknown profile
    errors at the input check before touching
    sandbox_init.
  - **DarwinAllProfilesHaveDistinctSchemes** — exploit /

## Build modes recap

| Build target                              | Tag set       | CGO | Use case |
|-------------------------------------------|---------------|-----|----------|
| `make build`                              | (none)        | OFF | default release |
| `make build-offensive`                    | offensive     | OFF | offensive release (no macOS sandbox) |
| `make build-mini`                         | mini          | OFF | device deployments |
| `make build-offensive-darwin-sandboxed`   | offensive     | ON  | macOS sandboxed (v1.50+) |

Sandboxed binary lands at `bin/elsereno-offensive-sandboxed` and is **not** in the release tarballs. Operators who want it run the make target locally (or via their own CI).

## Honest scope notes

  - sandbox_init is **deprecated since macOS 10.7**.
    Apple keeps the API functional through current
    macOS (verified on 14+) but doesn't update the
    docs. We suppress the deprecation warning via
    `#cgo CFLAGS: -Wno-deprecated-declarations`.
  - The .sb Scheme profiles are first-pass. Real-world
    macOS sandbox tuning is iterative: when a kernel-
    side `deny <op>` log fires, the operator widens
    the allow rule. Future cycles can refine the
    profiles based on observed denies.
  - No "kSBXProfile*" constant used — those legacy
    constants are too coarse (NoInternet denies DNS;
    NoWriteExceptTemp doesn't deny exec). Hand-rolling
    the Scheme gives precise control.
