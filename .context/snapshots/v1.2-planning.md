# Snapshot — v1.2 planning (draft, awaiting owner approval)

v1.1 closed **2026-04-21** (see `v1.1-sse-sandbox-opcua-wardial.md`).
This file proposes the shape of v1.2 so the owner can greenlight
scope + ordering without re-deriving it. **No v1.2 code has been
written** — the workspace rule is "no future-phase work without
owner approval" and this is the approval artefact.

## Proposed v1.2 chunks

Ordered by dependency — later chunks assume earlier ones are
landed. Effort is rough developer-days with one person working
the chunk in isolation.

### Chunk 1 — DB-backed audit Writer + findings / triage / runs tables (v1.1 carry-over 4b)  · ~3 days
- `internal/audit.DBWriter` implementing `audit.Writer` over
  pgx. Same chain invariant as `FileWriter`; sequential INSERT
  per ADR-008. `NewMultiWriter(FileWriter, DBWriter)` for the
  typical operator who wants both (file for `tail -f`, DB for
  the dashboard).
- Migration 00003: `findings` + `targets` + `runs` already
  exist from 00001 but have no panels or CLI. This chunk adds
  the dashboard panels reading from DB via a new
  `/api/v1/findings` + `/api/v1/runs` endpoint pair (paginated,
  severity-filterable, cursor-based).
- Dashboard inline template gains three panels (findings,
  triage, runs) wired to the new endpoints. Live feed stays
  broadcast-first; the panels provide the historical tail that
  SSE alone can't replay across reconnects.
- **Unblocks** v1.1 chunk 4b carry-over.

### Chunk 2 — OPC UA write gating  · ~4 days
- `offensive/write/opcua/gatedproxy.go` following the ADR-040
  pattern. The interesting difference vs. modbus/s7/enip:
  OPC UA wraps every payload in a SecureChannel + Session, so
  the allowlist has to match at the service-request layer
  (Write / Call) rather than the TCP-framing layer.
- `internal/protocols/opcua/wire/` needs service-code
  recognition (OpenSecureChannelRequest = 0x1BE, ActivateSession
  = 0x1D3, WriteRequest = 0x2A1, etc.). The full UA binary
  encoding is large; we only need enough of it to recognise +
  route four services (open/activate/write/close).
- Refusal frame for writes not on the allowlist: UA Fault
  response with status `BadUserAccessDenied` (0x80100000), so
  real clients see a parseable refusal.

### Chunk 3 — bacnet/dnp3/iec104/hartip/atg/fox full Handle loops  · ~5 days
- Session primitives from v1.1 chunk 1 stay. Each plugin needs
  a wire-level request dispatch loop that matches service
  codes / function codes against the allowlist, with a
  protocol-native refusal for disallowed writes:
  - bacnet: WritePropertyMultiple / AtomicWriteFile
    (BACnet-APDU abort reason 4).
  - dnp3: Function codes 0x02 (write), 0x05 (direct operate),
    0x06 (dir-op no-ack) → IIN2 bit 2 "FUNC_NOT_SUPP".
  - iec104: I-format ASDU types 45-51 (commands) + 100-105 →
    ACT_CON with COT=44 (unknown type).
  - hartip: burst-mode commands → HART status byte with
    "command not supported".
  - atg: TLS-350 function 10/20 writes → Veeder-Root NAK
    response.
  - fox: "fox a" commands that mutate state → already deny-all
    in v1.1; the full parser for read-only whitelist + write
    gate arrives here.

### Chunk 4 — PSTN / VoIP dial backend  · ~3 days
- `offensive/dial/backend/`: pluggable interface for the
  actual dial action. Two implementations to start:
  - `atmodem`: drive an AT-capable modem over serial (same
    wire-layer the `internal/protocols/atmodem` parser already
    speaks). Number dialled via `ATD<num>;`, call-progress
    read via Hayes result codes.
  - `voip-sip`: `github.com/cloudwebrtc/go-sip` wrapper that
    initiates a SIP INVITE and hangs up after connect/no-answer.
- `elsereno dial batch --backend atmodem|voip-sip` wires each
  allowed number through the chosen backend. Disposition
  becomes `delivered` / `no-answer` / `hangup` / `failed`,
  each recorded in the audit chain.
- PITF: modems in the wild don't always honour `ATH0` — add
  an explicit `+++` escape + DTR-drop fallback in the atmodem
  backend so a stuck line doesn't block the batch.

### Chunk 5 — SLSA `.intoto.jsonl` fix  · ~1 day
- Drop the reusable workflow call; invoke
  `slsa-framework/slsa-github-generator` directly with
  explicit `upload-assets: true` + a retry wrapper around the
  `final` step to sidestep the exit-27 bug. v1.0.1 and v1.1.0
  both shipped with the wrapper gate; v1.2 should land the
  real assets.

### Chunk 6 — Repo-public flip  · ~0.5 days
- Operator task: flip `RobinR00T/elSereno` to public.
- Triggers OpenSSF Scorecard + CodeQL + OSV-scanner + SLSA
  verifier to run on every PR.
- `make release-gate` already requires Scorecard ≥ 8.0; we
  should check current score on a private clone via the
  scorecard docker image before flipping.

### Close — snapshot + STATE + tag v1.2.0 + smoke  · ~0.5 days
Same shape as v1.1 close.

## Total

Roughly **17 developer-days** across six chunks. With the same
ship-daily cadence as v1.0.1 → v1.1.0, that's a 3-4 week
calendar window for the feature work plus the close.

## Questions for the owner

1. **Order**: the proposed order puts the DB Writer first
   because chunk 4b is a v1.1 carry-over; if you'd rather
   ship OPC UA writes first (they're the "most-asked" follow-
   on from operators), chunks 1 and 2 can swap without cost.
2. **SLSA fix**: dropping the reusable workflow means
   understanding a bit of the slsa-generator internals. Worth
   a spike before committing?
3. **Public flip**: do it as chunk 6 (post-v1.2 release) or
   as a separate pre-v1.2 operator action? Pre-flip gives
   Scorecard + CodeQL more runway on the v1.2 PRs; post-flip
   keeps the v1.2 dev cycle private.
4. **PSTN/VoIP backend**: if your threat-model review wants
   to punt real dialling to v1.3, chunk 4 can downgrade to
   "backend interface + mocks only" (saves ~2 days).
5. **SECURITY.md + LEGAL.md review**: v1.2 adds real PSTN /
   VoIP capability — the legal posture needs a line about
   consent + jurisdiction (we have one for writes; should the
   dial wording be stricter?).

## Guardrails

v1.2 must keep every v1.1 invariant:
- Default build stays read-only. Writes require
  `-tags offensive` + triple-confirm + sandbox load.
- Dial guard ≤3 digits hard-blocked.
- Audit chain must stay intact across the DB migration:
  new DBWriter has to bootstrap from the current FileWriter
  tail (one-shot import, chain-preserving).
- No new external Go dependencies without an ADR note citing
  supply-chain impact (ADR-002).
