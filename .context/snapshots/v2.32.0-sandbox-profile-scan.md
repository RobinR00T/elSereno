# Snapshot — v2.32.0: ProfileScan for read-only scan subprocesses

Single-chunk cycle. Builds on the v1.50 sandbox infrastructure
(macOS sandbox_init(3) + Linux seccomp + ADR-042 profile model).
v1.50 shipped 3 profiles (Exploit / Harvest / Dial); v2.32
adds a 4th: ProfileScan for the default-build scan subprocesses.

## Why

The scan path is already "safe" by design — it reads-only,
talks network, doesn't write findings to disk (parent owns
the audit chain). But defence in depth says dropping unused
capabilities is cheap insurance against an in-the-wild bug
in a fingerprint parser. A compromised target response can't
trigger a write or pivot.

## What

New `Profile` constant + per-platform policy:

  - **`sandbox.go`**: `ProfileScan = "scan"` + Valid() switch
    extends the recognised set.
  - **`sandbox_darwin_cgo.go`**: `.sb` Scheme profile —
    `deny default`, allow read+network, **deny file-write\***,
    deny process-exec.
  - **`syscalls_linux.go`**: seccomp extra-deny list mirrors
    ProfileHarvest (truncate/unlink/mknod/chmod/rename/
    symlink/link).
  - **`bpf_argfilter_linux.go`**: openat write-flag arg filter
    (same as Harvest).
  - **`sandbox_other.go`**: no-op degradation path — same as
    every other profile on unsupported platforms.

## Honest scope

  - No callsite plumbing yet. v2.32 lands the profile;
    integrating it at every default-build subprocess entry
    point is a follow-up cycle (operator opts in via
    `--sandbox-profile=scan` once that flag lands).
  - INSTALL.md unchanged. Build sizes unchanged.

## Build sizes

Unchanged across all 3 variants.

Adds ~40 LoC across the sandbox files.

## Carryover

  - **vNext**: `--sandbox-profile=scan` flag on `elsereno scan`.
  - **vNext**: per-protocol policy override (e.g.
    `--sandbox-profile=scan` for default + dial protocols
    pick ProfileDial automatically).
