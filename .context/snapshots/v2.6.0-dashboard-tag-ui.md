# Snapshot — v2.6.0: dashboard tag chips + tag-cloud widget

Single-chunk cycle. Closes v2.4/v2.5 carryover. The v2.4
tags + v2.5 tag-counts endpoints existed but the dashboard
had no UI for them — operators were stuck with curl.

## Dashboard surface

### Schedule form

  - New `tags` input (comma-separated). Plumbed through
    submit + edit-mode load + cancel reset.
  - Server-side canonicalisation handles dedup/sort/shape
    validation; the UI just sends what the operator typed
    (lowercased + trimmed).

### Schedules table

  - New "Tags" column between Name + Input.
  - Each row renders tags as clickable chips. Clicking a
    chip sets the global tag filter; clicking the same
    chip again clears it (toggle).
  - colspan on empty-state row bumped 8 → 9 to match the
    new column.

### Tag-cloud widget

  - Above the table: chips loaded from
    `/api/v1/schedules/tags`, formatted as
    `<tag> (<count>)`.
  - Active chip gets a highlight (bg #99c / fg #fff).
  - "Clear filter" button appears when a filter is active.
  - Empty fleet → "no tags yet".

### State

JS-level `scheduleTagFilter` string (empty = no filter).
`renderSchedules` fetches `/api/v1/schedules?tag=…` when
set + reuses the v2.4 list endpoint's filter semantics.

## JS API

  - `refreshScheduleTagCloud()` — polled in parallel with
    `renderSchedules`.
  - `setScheduleTagFilter(tag)` — toggle handler.
  - `clearScheduleTagFilter()` — clear button handler.

Both exposed on `window.*` so inline `onclick` attributes
resolve.

## Honest scope

  - No multi-select filter (operators click one tag at a
    time). v2.9 carryover adds AND/OR multi-tag semantics.
  - No tag-rename UI yet — operators wanting to rebrand
    a tag must update each schedule manually.
  - No autocomplete on the form input — operators look at
    the tag-cloud chips above to see what tags exist.
  - INSTALL.md unchanged. Build sizes unchanged.

## Build sizes

Unchanged across all 3 variants.

Adds ~150 LoC in dashboard.go (HTML + JS).
