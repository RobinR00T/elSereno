# Snapshot — v1.16.0: CWMP correlation + BACnet refinements + token-generation foundation

Four chunks covering refinement of the gates closed in v1.13
(BACnet) and v1.15 (CWMP), plus the cryptographic foundation
for in-process allow-file reload.

## Chunk 1 — CWMP TransferComplete authorisation cross-reference

`33284c8 feat(v1.16 chunk 1): CWMP TransferComplete
authorisation cross-reference`

Closes the v1.15 chunk-1 observer half by correlating CPE →
ACS TransferComplete envelopes with the prior Download
authorisation that started the transfer. The gate records
`(CommandKey, DownloadURL, AllowlistURL, AllowlistSHA256,
AuthorisedAt)` at Download authorisation time (FIFO-bounded by
`PendingDownloadCap`, default 256) and resolves the entry on
TransferComplete. The resolved cross-reference is exposed via
the new `TransferCompleteFields.Authorisation
*DownloadAuthorisation` field; `Outcome()` classifies the
envelope into `succeeded` / `failed` / `orphan_complete` /
`orphan_fault`.

Default CLI observer enriches its stderr log line with
`outcome=`, `download_url=`, `allowlist_sha256=`, and
`authorised_at=` fields. Orphan rows surface CPE reports for
CommandKeys we never authorised (suspicious — operators
should alert).

Resolution is one-shot — duplicate / replayed
TransferComplete sees `Authorisation=nil` on the second hit.

**9 new tests** (`transfercomplete_test.go` extended +
new `pendingdownload_test.go`):
- happy path: success → Authorisation populated with
  Allowlist URL/SHA-256
- fault path: FaultCode 9010 → Authorisation still populated
- orphan: TC with unknown CommandKey → Authorisation=nil,
  outcome=orphan_complete
- one-shot: second TC with same CommandKey → orphan
- Outcome() matrix across (success, hasAuth) cells
- ExtractDownloadCommandKey: happy / missing / non-Download
  envelope
- PendingDownloadCap eviction (FIFO at cap)
- Duplicate-key replacement preserves cap accounting
- Empty CommandKey skipped

## Chunk 2 — BACnet per-(type, instance) CreateObject scoping

`83a4b69 feat(v1.16 chunk 2): BACnet per-(type, instance)
CreateObject scoping`

Refines the v1.13 chunk-8 per-type CreateObject (svc 10)
allowlist with a parallel per-(type, instance) list. When the
ACS uses the `[1]` objectIdentifier CHOICE form (operator
pre-declares which exact instance the device should create),
the gate matches against `AllowedCreateObjectInstances`; the
v1.13 per-type list remains as fallback so operators can mix
grains.

Match precedence:
1. CHOICE [1] + AllowedCreateObjectInstances has (type, inst)
   → pass.
2. Otherwise consult AllowedCreateObjects per-type list.
3. Otherwise refuse.

Operators wanting strict per-instance scoping leave
`AllowedCreateObjects` empty + populate
`AllowedCreateObjectInstances`; CHOICE [0] (type-only)
requests then refuse because they carry no instance to match.

Wire parser: new `ParseCreateObjectWithInstance` returns
`(objType, instance, hasInstance, ok)`. The 22-bit instance
is extracted from the packed BACnet ObjectIdentifier when
CHOICE [1] is used. The existing `ParseCreateObject` is
retained as a thin wrapper.

Hash ladder: new separator **0xF7** +
`AllowlistHashWithCreateObjectInstances` +
`SessionMutationWithCreateObjectInstances`. Empty
`CreateObjectInstances` → degrades to
`AllowlistHashWithListElements` (preserves every v1.4 →
v1.13-chunk-13 confirm-token).

CLI: `--create-object-instance type=N;instance=M`. YAML
round-trip via `create_object_instances:`.

**9 new tests** (`createobjectinstance_test.go`).

## Chunk 3 — BACnet per-(operation, type, instance) LifeSafetyOperation scoping

`ed98c71 feat(v1.16 chunk 3): BACnet per-(operation, type,
instance) LifeSafetyOperation scoping`

Refines the v1.13 chunk-11 per-operation LSO (svc 27)
allowlist with a parallel per-target list. When the ACS
includes the optional `[3]` objectIdentifier ("silence
LifeSafetyPoint #3" instead of "silence the whole device"),
the gate scopes to specific (op, type, instance) tuples;
device-wide requests (no `[3]`) fall back to the per-operation
list.

**Operationally important** for fire-alarm panel deployments:
"operator may unsilence LifeSafetyPoint #3 only" is much
tighter than "operator may unsilence anything on this device"
— silencing operations on a fire-alarm panel are POTENTIALLY
LETHAL on production buses.

Wire parser: new `ParseLifeSafetyOperationWithTarget` returns
`(op, target, hasTarget, ok)`. `ParseLifeSafetyOperation`
retained as thin wrapper.

Hash ladder: new separator **0xF6** +
`AllowlistHashWithLSOTargets` +
`SessionMutationWithLSOTargets`. Empty `LSOTargets` →
degrades to chunk-2.

CLI: `--lso-target op=N;type=N;instance=N`. YAML round-trip
via `lso_targets:`.

Refactor in `cmd_proxy_offensive.go`: `buildBACnetHandler` had
grown past funlen; extracted `parseBACnetProxyOpts` (returns
the bundled `parsedBACnetProxyOpts` struct) and
`buildBACnetServiceList` so the constructor stays under 60
lines.

**9 new tests** (`lsotarget_test.go`) — gate matrix includes
the "HOSTILE silence attempted on Reset-only target" case.

## Chunk 4 — BACnet token-generation cookie

`c3256da feat(v1.16 chunk 4): BACnet token-generation cookie
(foundation for in-process reload)`

Cryptographic foundation for in-process allow-file reload.
Operators bump `Generation` when editing the allow-file; the
new generation folds into the session hash so a stale confirm-
token (minted at the prior generation) is rejected at
`Authorise()` time.

v1.15 chunk 5 chose the supervisor-restart pattern (SIGHUP →
exit 75) because changing the allowlist in-place would
invalidate the operator's existing token. The fix: pre-mint a
token bound to generation N+1, the proxy at reload time
atomically switches both the allowlist AND the expected
generation.

This chunk ships the **hash-layer foundation only** — the
actual reload signal handler + atomic allowlist swap are
v1.17+ work.

Hash ladder: new separator **0xF5** +
`AllowlistHashWithGeneration` + `SessionMutationWithGeneration`.
`Generation=0` (default) → degrades to chunk-3 hash
(preserves every v1.4 → v1.16-chunk-3 confirm-token).

CLI: `--token-generation N` on `proxy listen --plugin bacnet`
+ `write bacnet dry-run`. YAML round-trip via
`token_generation:`.

**7 new tests** (`tokengeneration_test.go`) covering hash
distinctness across generations + the E2E Authorise matrix
(stale-rejected / fresh-accepted / chunk-3-backwards-compat).

Foundation for BACnet only; cross-protocol parity (sip /
iax2 / pbxhttp / modbus / opcua / cwmp) follows in subsequent
chunks if operators need it.

## Deferred to v1.17+

- **In-process reload signal handler** — chunk 4 ships the
  hash foundation; the actual SIGUSR1 handler (or fsnotify
  watcher), atomic allowlist swap, and control-plane
  confirm-token delivery are still pending. The supervisor-
  restart pattern (v1.15 chunk 5) keeps working as the
  operational option until this lands.
- **Cross-protocol token-generation parity** — only BACnet has
  the `Generation` field after chunk 4. Cross-protocol
  symmetry (sip / iax2 / pbxhttp / modbus / opcua / cwmp)
  lands incrementally if operators ask.
- **CWMP TransferComplete async firmware re-fetch** — chunk 1
  surfaces the allowlist SHA-256 alongside the CPE's report;
  true on-wire SHA-256 verification would need an out-of-band
  fetch + verify (v1.13 chunk 2's `verify-firmware` already
  does this pre-flight). Post-flash async verification is
  v1.17+.
- 12 legacy ICS protocols, TUI, Windows support, OIDC + roles
  remain in the unchanged backlog.
