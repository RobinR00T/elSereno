## Chunks

### Chunk 1 — ProConOS fingerprint (`7842351`)

Best-effort fingerprint plugin for the KW-Software ProConOS
runtime protocol on TCP/20547 (Phoenix Contact ILC + Berghof +
IPC2u + ABB / B&R / Lenze re-skins + a long tail of OEM
rebrands).

**HONEST SCOPE NOTE — needs validation**: public references to
the ProConOS handshake conflict. Two candidate hello-frame
layouts are documented:

  1. `0xCA 0xFE 0x00 0x00` + `0xCE 0xFA 0xDE 0xC0` (older
     Berghof + Lenze deployments).
  2. `01 06 00 10` + `PROCONOS` + 4-byte zero pad (Wireshark
     dissector + metasploit auxiliary scanner).

The plugin implements **variant 2** because that's the form
two independently verifiable lineages agree on. Banner
classifier accepts both shapes via permissive substring
matching.

Until real-PLC test vectors are available, operators should
treat positive identifications as **confidence ≈ 0.7** rather
than the ≈ 0.95 the v1.20-v1.25 fingerprint plugins produce.
Scoring reflects this: protocol_risk=75 (vs codesys/pcworx
80), capability ceiling=60 (vs 75), cve_exposure=7
(KW-Software ICSA family inheritance: ICSA-15-160-01 + 17-
201-01 + 18-296-01).

9 wire tests + 9 plugin tests covering hello shape, banner
substrings (PROCONOS / ProConOS / KW-Software / MultiProg /
KWS-LDR / alt-prefix), prefix echo, silent responder, non-
ProConOS junk rejection, fail-closed proxy, REPL stub.

### Chunk 2 — GE-SRTP service-0x21 follow-up (`bd70cb8`)

Closes the v1.25 deferred GE-SRTP service-0x21 item. The v1.21
chunk-4 plugin shipped the connection-init probe + model-hint
extractor; this chunk adds an opt-in second exchange that asks
the PLC for its long status (service code 0x21).

`internal/protocols/gesrtp/wire/wire.go` (extended):
  - `ServiceLongStatus` byte = 0x21 constant.
  - `BuildReadLongStatus()` returns the canonical 56-byte
    service-request mailbox (matches nmap NSE script
    `gesrtp-info` + metasploit auxiliary scanner).
  - `LongStatusInfo` type with Model + Firmware string fields.
  - `ParseLongStatus(buf)` extracts model hint via the v1.21-
    chunk-4 ExtractModelHint then scans the tail for a
    "V<digit>(\.\d+)+" version pattern (≥4 chars to reject
    "V0"/"V1" noise). Heuristic — response field-layout varies.

`internal/protocols/gesrtp/gesrtp.go` (extended):
  - `Probe` now does the connection-init + service-0x21
    follow-up.
  - On follow-up success the finding note grows a "fw=Vx.y.z"
    suffix: e.g. "SRTP model=PACSystems fw=V12.45.7".
  - On any follow-up failure (timeout, short frame, missing
    markers) falls back to the v1.21-chunk-4 model-hint result
    — no regression for firmwares that don't speak service-0x21.

7 new wire tests + the existing plugin tests pass unchanged.

**HONEST SCOPE NOTE**: this code path has not been validated
against a physical Mark VIe / RX3i / PACSystems device.

### Chunk 3 — wire record-replay into pcworx + mms gates POC (`0242d68`)

Proof-of-concept for the v1.27-chunk-4 record-replay primitive.
Adds an optional `Recorder *replay.Recorder` field to the
pcworx + mms WriteGatedHandlers; when non-nil, Handle() wraps
both client + upstream io.ReadWriter pairs through the
recorder before the io.Copy goroutines start.

Pcworx + mms are the natural starting points because both ship
session-level (no wire parsing) — adding the Wrap calls is a
4-line change. The 7 wire-aware gates (sip, iax2, pbxhttp,
modbus, opcua, bacnet, cwmp) need careful per-handler
verification because each parses the io.ReadWriter internally;
that's a v1.29+ task.

The v1.28 POC ships:
  - `Recorder` field on both gates (defaults to nil).
  - 4-line Handle() wrap-on-non-nil block per gate.
  - End-to-end test (TestHandle_RecordsBytesWhenRecorderSet)
    that drives a real net.Pipe pair through pcworx with a
    recorder attached, then verifies the recording captures
    DirClientToUpstream chunks, the file is 0600, and the
    header carries the right protocol tag.

CLI integration (an `--record FILE` flag on
`elsereno proxy listen`) is a v1.29+ candidate.
