## Why a multi-chunk cycle now starts as just a shell

Dashboard scan orchestration is genuinely a multi-
chunk feature: the orchestration shell + worker
runtime + persistent storage + UI hookup are
naturally separable concerns. Trying to land them
all in one chunk would either ship a 2000-line
monster commit or compromise on the model. The
shell-first approach lets:

  - The state machine + REST contract get reviewed
    against the dashboard's needs before downstream
    chunks encode them.
  - Operators experiment with the API surface
    (curl) before any UI exists.
  - Future chunks (Worker dispatch, DB-backed Store,
    UI) extend the shell without revisiting the
    contract.

## Chunks

### Chunk 1 — orchestration core + REST (`d20ec3d`)

`internal/scanorch/job.go` (new, ~250 lines):

  - **State** type with 5 values: queued, running,
    completed, failed, cancelled.
  - **State.IsTerminal()** classifier for store
    eviction policy (currently no eviction).
  - **Job** struct: ID + State + CreatedAt +
    StartedAt + FinishedAt + Input + Plugins +
    DefaultPort + Stats + Error + Operator. JSON
    tags drive the wire shape.
  - **Stats** struct: TargetsSeen + TargetsScanned
    + FindingsCount.
  - **SubmitRequest**: dashboard's POST payload
    (Input + Plugins + DefaultPort).
  - **Store** interface with 4 methods (Submit /
    Get / List / Transition).
  - **TransitionFields** carries optional Stats +
    Error updates that travel with state changes.
  - **canTransition(from, to)** enforces the state
    machine via a table:
      - queued → running, queued → cancelled
      - running → completed, running → failed,
        running → cancelled
      - terminal → (no edges)
  - **MemoryStore** in-memory implementation:
    sync.RWMutex around map[ID→Job] + insertion-
    order list for List(). Goroutine-safe.
  - **generateID()** returns 16-char crypto/rand
    hex (8-byte budget).
  - 4 sentinels: ErrInputRequired, ErrJobNotFound,
    ErrInvalidTransition, ErrJobAlreadyTerminal.

## Honest scope

  - **Job execution** (queued → running → completed
    via actual scanner dispatch) NOT in this chunk.
    A Worker + JobRunner interface lands in chunk 2.
    The state-machine API is already correct; the
    Worker just plugs into it.
  - **Persistent storage** NOT in this chunk. The
    MemoryStore loses jobs on restart. DB-backed
    Store (mirroring the runs + findings + audit
    repos) lands in chunk 3.
  - **Cancellation flow**: ctx cancellation is
    plumbed but the cancellation→failed transition
    is best-effort. Tightening this is deferred.
  - **Authorisation** is currently just X-Operator
    header reading. Wiring scan-trigger behind a
    separate scope (so a "viewer" operator can
    list but not submit) is deferred.
  - **UI hookup** NOT in this chunk. Dashboard's
    JSX submits via fetch() against the new endpoints
    in a future cycle.
