# Snapshot — v2.12.0: atomic import preflight

Single-chunk cycle. Closes v1.99 carryover. v1.99's import
processed rows one-by-one; if row N failed validation, rows
1..N-1 were already committed. v2.12 adds preflight
validation so operators can ask for all-or-nothing.

## REST

```
POST /api/v1/schedules/import?atomic=true
```

When `atomic=true`:
  1. Decode body.
  2. **Preflight pass** — validate every row WITHOUT touching
     the store:
     - shape (name + template + cadence + tz).
     - tag canonicalisation (1..10 tags, [a-z0-9_-]{1,32}).
     - audit_retention_days >= 0.
     - duplicate names in the same upload (unless
       on_conflict=rename).
  3. Any failure → 400 with `failures[]` per-row error
     list. **No writes occur.**
  4. All pass → apply per v1.99 semantics (with the chosen
     on_conflict).

When `atomic=false` or omitted → existing v1.99 behaviour
(partial-success best-effort).

## Wire shape on preflight failure

```json
{
  "schema": "api:v1",
  "data": {
    "error": "preflight validation failed; no writes performed",
    "failure_count": 2,
    "failures": [
      {"index": 1, "name": "bad", "error": "..."},
      {"index": 4, "name": "alpha", "error": "duplicate name in input batch ..."}
    ]
  }
}
```

## Exports

New exported wrappers for the preflight pass (handlers
can't reach the private validators):
  - `scanorch.ValidateScheduleFieldsForImport`.
  - `scanorch.ValidateAuditRetentionForImport`.
  - `scanorch.CanonicaliseTagsForImport`.

## Honest scope

  - "Atomic" here means validation-atomic. Post-validation
    transient DB errors at Create time can still leave
    partial state (we don't wrap Create calls in a tx —
    the Store interface doesn't expose tx semantics).
    For 99% of operator workflows (typos, malformed
    cadence, duplicate names) this is sufficient.
  - A future `?atomic=tx` mode could wrap the whole pass
    in a DB tx — requires Store-level Begin/Commit
    support similar to the v1.90 advisory-lock pattern.
  - Duplicate-name detection in the batch is only an
    error when `on_conflict != rename`. Rename mode
    handles in-batch duplicates by appending suffixes.
  - INSTALL.md unchanged. Build sizes unchanged.
