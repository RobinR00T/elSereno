# Snapshot — v2.21.0: real PG WithTx implementation

Single-chunk cycle. Closes v2.20 carryover. v2.20 added the
`Store.WithTx` interface but kept DBScheduleStore as
pass-through. v2.21 wires the real BeginTx + Commit/Rollback
path so `?atomic=tx` actually rolls back on PG-backed deploys.

## Impl

  - Type-asserts `s.q` (Querier) to the existing v1.90
    `txQuerier` interface (BeginTx method).
  - Real path: BeginTx + new DBScheduleStore wrapping the
    pgx.Tx as Querier + invoke fn + Commit on success or
    Rollback on fn-error.
  - Fallback: when type-assert fails (test fakes), fn(s)
    pass-through — matches the v2.20 documented semantics.

No new ctor needed — reuses the existing `NewDBScheduleStore`
since pgx.Tx + pgxpool.Pool + pgx.Conn all satisfy the
txQuerier interface naturally.

## Tests (+1)

  - TestDBScheduleStore_WithTx_FakeQuerier_FallsThrough:
    fake querier (no txQuerier) → pass-through; fn-nil
    returns nil; fn-err propagates via errors.Is.

Real PG integration coverage requires a live PG via
`scripts/dev-db.sh`; deferred to whatever cycle adds the
first PG-integration suite.

## Honest scope

  - BeginTx uses default isolation (READ COMMITTED in PG).
    Future cycles can expose options via a new method
    (`WithTxOpts(opts, fn)`).
  - Rollback errors are silently swallowed (best-effort);
    the fn-error is the canonical thing to surface.
  - INSTALL.md unchanged. Build sizes unchanged.

## Build sizes

Unchanged across all 3 variants.

Adds ~30 LoC + 1 test.

## Carryover

  - Live PG integration test for WithTx covering

  - Optional `WithTxOpts(opts pgx.TxOptions, fn)` for
