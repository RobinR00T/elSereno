# Snapshot — v1.87.0: background audit pruner

Single-chunk cycle. v1.86 added the manual
`DELETE /api/v1/schedules/audit` endpoint; v1.87 adds the
daily background pruner so operators don't have to cron
the curl themselves.

## Why

Operational hygiene. The retention story shouldn't depend
on out-of-band cron. A `--audit-retention-days N` flag
gives operators a single point of configuration.

## API

```go
type AuditPruner struct {
  AuditStore      ScheduleAuditStore // required
  RetentionPeriod time.Duration      // clamped to >= 1m
  Interval        time.Duration      // clamped to [1m, 7d];
                                     // default 24h
  Now             func() time.Time   // test seam
  OnPrune  func(deletedCount int64, cutoff time.Time)
  OnError  func(err error)
}

// Sentinels:
var ErrAuditPrunerNoStore = ...
var ErrAuditPrunerNoRetention = ...
```

Run(ctx) blocks until ctx cancelled. Eager first tick on
start (restart doesn't wait a full interval). Errors
surface via OnError; next tick re-attempts.

Tick(ctx) is a test affordance — single-shot prune
without the ticker goroutine (mirrors Scheduler.Tick).

## CLI

```
elsereno serve --audit-retention-days N
  0 (default)  = disabled
  N > 0        = spawn pruner; events older than N days
                 deleted every 24h
```

Wired in cmd_serve.buildScanAndSchedule alongside the
v1.70 scheduler. ctx-cancellation lifecycle; stderr
logging on prune events + errors.

## Clamping (operator footgun guards)

  pruneMinRetention = 1 minute
  pruneMinInterval  = 1 minute
  pruneMaxInterval  = 7 days
  default interval  = 24h

A misconfigured `--audit-retention-days=0.001` would
otherwise wipe new events on the next tick. The clamp
makes "retention < 1 minute" silently equivalent to "1
minute" — operators with sub-minute requirements aren't a
real use case for an audit log.

All build + lint + tests clean.

## Honest scope

- No per-schedule retention (same as v1.86).
- Single-process pruner — multiple `serve` instances
  against a shared DB would race; the DELETE is
  idempotent so the worst case is duplicate OnPrune
  callbacks. No advisory lock yet.
- Eager first tick: trade-off documented.
- No Prometheus metrics — OnPrune is a callback hook
  by design; operator wires their own metrics.
