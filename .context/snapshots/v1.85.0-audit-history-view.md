# Snapshot — v1.85.0: dashboard audit history view

Single-chunk cycle. Dashboard-only JS+HTML. v1.84 added
the /audit endpoint + persistence; v1.85 surfaces it in
the dashboard.

## UI

- Per-schedule "History" button between Enable/Disable
  and Delete in the action column.
- Hidden `<div id="schedule-audit-view">` below the
  schedules table, revealed by openAuditView.
- Table with columns When · Who · Event · Changes. The
  Changes cell renders a `<ul class="audit-diff">` of
  per-field "before → after" lines.

## Helpers

```js
openAuditView(id, displayName) {
  GET /api/v1/schedules/{id}/audit
  503 → "audit log unavailable"
  200 + [] → "no audit events recorded"
  200 + events → render table
}
closeAuditView() // hides panel

computeAuditEventDiff(ev) {
  parse(payload_before) + parse(payload_after)
  → [{field, before, after}, ...] for changed fields
}
```

`computeAuditEventDiff` covers the six editable fields
+ `enabled`. Future event_type values (delete /
set_enabled / etc) benefit without code changes.

## Honest scope

- No pagination. With one event per force-overwrite the
  typical schedule sees ≤10 rows lifetime.
- No timestamp filtering.
- No CSV/JSON download — curl directly for bulk export.
- Separate computeAuditEventDiff (vs v1.81
  computeScheduleDiff) because the input shape differs
  (audit event with before/after vs. operator's pending
  payload).
