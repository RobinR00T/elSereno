# Snapshot — v1.68.0: plugin-list autocomplete

Single-chunk cycle. Tightens the scan-submit form's
plugin field with a native HTML `<datalist>`
populated from `/api/v1/plugins`. Discoverability
upgrade for operators new to the registered-plugin
set; no breaking changes.

## Why

The plugin field was free text. Operators had to
either read INSTALL.md or hit `/api/v1/plugins`
manually to know what was available. With the
datalist, clicking the field shows the full sorted
list of registered plugins with their default ports
+ descriptions. A small UX win that closes a
recurring "what can I type here" question.

## Implementation

```html
<input id="scan-plugin"
       list="scan-plugin-options"
       autocomplete="off" />
<datalist id="scan-plugin-options"></datalist>
```

```js
function loadPluginDatalist() {
  fetch("/api/v1/plugins")
    .then(...)
    .then(function (res) {
      var rows = (res && res.data) || [];
      rows.sort(...).forEach(p => {
        list.innerHTML += '<option value="' + name +
          '" label="' + desc + ' (port ' + p.default_port + ')"></option>';
      });
    });
}
loadPluginDatalist(); // once on page boot
```

Plugin set is build-time stable (registered in
`init()`); a single boot-time load suffices.

## Why not a chip-based multi-select

Native `<datalist>` doesn't tokenize on commas, so
typing `modbus,s` gets no suggestion match. A custom
chip widget would solve this but adds:

  - Per-chip render + remove logic.
  - State management (selected set).
  - Keyboard nav for accessibility.
  - Cross-browser styling.

Out of scope for a single chunk. The datalist
suffices for the discoverability win — the operator
sees what's available and types the first plugin
name with autocomplete; subsequent comma-separated
plugins they type by hand.

## Cross-browser

  - **Chrome / Edge**: shows the value + label
    (description + port) in two columns.
  - **Firefox**: same.
  - **Safari**: shows only the value column; the
    label is ignored. Operator still gets the plugin
    name list.

Acceptable fallback. No browser tested misrenders
or breaks.

## Honest scope

  - **Multi-token autocomplete after a comma**: NOT
    supported. `<datalist>` doesn't tokenize. Custom
    widget is a future cycle's task.
  - **Plugin description truncation**: capped at 80
    chars in the label. Long descriptions are
    truncated; the operator sees the start. Tradeoff
    against dropdown row height.
  - **Loading-state UX**: no spinner / placeholder.
    A failed fetch leaves the datalist empty
    silently. Acceptable: the input still works as
    free text.
  - **make sec pre-existing failures**: govulncheck
    DB picked up GO-2026-4971, GO-2026-4918, etc.
    against go1.26.2 (fixed in 1.26.3). These were
    not present at v1.67 close two days ago; the DB
    has been updated in the meantime. Pre-existing
    code paths only — v1.68 introduces no new
    vulnerable callsites. Operator should upgrade
    the Go toolchain in CI/build to clear.
