# Snapshot — v1.86.0: audit retention pruning

Single-chunk cycle. v1.84 added the force-overwrite
audit log; v1.85 the dashboard view. v1.86 adds the
retention enforcement path so operators can prune old
events before the table grows unbounded.

## API

```go
type ScheduleAuditStore interface {
  // ... v1.84 methods ...
  PruneOlderThan(ctx, cutoff time.Time) (int64, error)
}
```

REST:

```
DELETE /api/v1/schedules/audit?before=<rfc3339>
200 → { "data": { "deleted_count": N, "cutoff": "..." } }
400 on missing/malformed before
503 when audit store nil
```

The endpoint is global (operates across all schedules).
Per-schedule retention is deferred.

## Implementation

Memory store: walks the slice under write lock, keeps
events with OccurredAt >= cutoff.

DB store: single DELETE WHERE occurred_at < $1; returns
tag.RowsAffected() as int64.

## Honest scope

- No automatic background pruner — operator invokes
  the DELETE endpoint manually (cron + curl pattern).
  v1.87+ may add `--audit-retention-days` flag.
- No per-schedule pruning endpoint.
- No "preview" mode — DELETE is irrevocable. Operators
  GET /audit + filter client-side to dry-run.
- Future cutoffs accepted (no warning).
- No cross-store transactional semantics.
