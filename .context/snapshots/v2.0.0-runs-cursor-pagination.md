# Snapshot — v2.0.0: cursor pagination on /runs

Single-chunk cycle. Closes v1.92 carryover. The /runs
endpoint capped at limit=1000 with no cursor — schedules
firing daily for a year produced 365 rows max in one
fetch. v2.0 adds keyset pagination via ?before=<rfc3339>.

## Wire API

**Request**:

```
GET /api/v1/schedules/{id}/runs?limit=N&before=<rfc3339>
```

Both query params optional. Default limit=50, max 1000.
Default before=zero (newest page).

**Response shape change (BREAKING)**:

```diff
- "data": [Job, Job, …]
+ "data": {
+   "items": [Job, Job, …],
+   "next_before": "2026-05-12T03:14:15.726Z"  // omitted on last page
+ }
```

Operators paginate by reading `next_before` from one
response and passing it back on the next call. Empty
`next_before` ≡ "this is the last page". A response with
`items.length < limit` is guaranteed to be the last page
(server omits next_before for partial pages).

## Store interface

New optional method on `scanorch.Store`:

```go
ListByScheduleBefore(ctx, scheduleID string, before time.Time, limit int) ([]Job, error)
```

Existing `ListBySchedule` becomes a thin wrapper passing
zero `before`. Memory + DB impls.

PG variant: when `before` is non-zero, uses
`WHERE triggered_by_schedule_id = $1 AND created_at < $2`
which composes with the v1.92 partial index for fast cursor
scans. Zero `before` uses the existing $1-only query
(saves a parameter binding).

`BroadcastingStore` (web/stream) passes through both
methods.

## Honest scope

  - Response shape is a v2.0-incrementing break. Operators
    polling the dashboard's UI consume via JS already
    updated; external integrations need to update their
    consumers. Major version bump justifies it.
  - Cursor is created_at-only; if two jobs share the same
    created_at to the nanosecond (won't happen in practice
    given microsecond truncation + sequential Submit) the
    cursor could skip rows. Acceptable.
  - INSTALL.md unchanged. Build sizes unchanged.
