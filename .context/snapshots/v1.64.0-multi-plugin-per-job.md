# Snapshot — v1.64.0: multi-plugin per scan Job

Single-chunk cycle. Relaxes the v1.61 single-plugin
contract on scan jobs. The orchestration shell, DB
schema, REST contract, worker, and SSE bridge all
stay unchanged — this is purely a cmd-side runner
change plus a UI form tweak.

## What changed

### Plugins field semantics

| Plugins value | Behaviour |
|---------------|-----------|
| `[]` (empty)  | Run every registered plugin in the build. |
| `["modbus"]`  | Single plugin (back-compat with v1.61). |
| `["modbus","s7","enip"]` | Fan out to each named plugin per target. |

### Per-target dispatch

For each plugin × target, the runner attempts a probe
iff the plugin's DefaultPort matches the target's Port.
A plugin with DefaultPort=0 (banner-style probe-
anywhere) matches every target. This mirrors the
existing `discover --hosts` logic and matches operator
mental model ("modbus probes 502, s7 probes 102").

### Stats accounting

Probe-attempts semantics, not unique-targets:

  - **TargetsSeen**: len(parsed targets). The shape of
    the input.
  - **TargetsScanned**: count of (target, plugin)
    probe attempts. A target probed by 3 plugins
    counts as 3.
  - **FindingsCount**: total findings produced across
    all (target, plugin) combinations.

Operators who want unique-targets-touched can derive
it from Job.Input parsing on their side. The runner
side stays simple — one counter per scanner.Run drain
event, summed across all plugin runs.

### Sentinel changes

| Old (v1.61) | v1.64 |
|-------------|-------|
| ErrRunnerNoPlugin | dropped (empty Plugins is now legal) |
| ErrRunnerTooManyPlugins | dropped (multi-plugin is now legal) |
| ErrRunnerUnknownPlugin | unchanged |
| — | **new** ErrRunnerNoMatchingPlugins |

`ErrRunnerNoMatchingPlugins` fires when a Job's plugin
set produces zero (target, plugin) matches — common
when targets all sit on a port no named plugin
claims. Returning it as a job-failure (vs silently
completing with TargetsScanned=0) gives operators a
clear "your scan did nothing" signal.

## Honest scope

  - **Per-plugin concurrency cap**: each plugin's
    scanner.Run uses MaxConcurrentTargets from the
    runner's `concurrency` knob. A 5-plugin job at
    concurrency=100 has up to 500 in-flight dials
    across the worker. Operators tune via
    `--scan-pool` (worker pool concurrency limits
    parallel jobs; per-plugin concurrency is
    constant). Future cycle could add a per-job
    `concurrency` field.
  - **Plugin order**: resolvePlugins preserves the
    operator-supplied order for named lists; empty
    Plugins inherits core.RegisteredPlugins() order
    (sorted by name). Findings are aggregated, not
    grouped per-plugin in Stats; a future cycle
    could add `findings_by_plugin` map.
  - **No filter for offensive plugins**: if the
    operator runs `serve` with `-tags offensive`,
    blank-plugin-list dispatches every offensive
    plugin too. The default build's registry doesn't
    include the offensive Probe paths (they live
    behind plugin Factory(), not Probe()), so this
    is not an issue in default builds — but operators
    running offensive-tag should be explicit about
    the plugin list.
