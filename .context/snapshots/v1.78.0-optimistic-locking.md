# Snapshot — v1.78.0: optimistic locking on schedule edits

Single-chunk cycle. Multi-operator deployments could lose
edits silently in v1.74-v1.77 — two operators editing the
same schedule from the dashboard would last-write-wins,
with no signal to the loser. v1.78 adds an `updated_at`
column + If-Match precondition so concurrent edits surface
as 412 Precondition Failed.

## Why

The v1.74 honest-scope explicitly noted: "No optimistic
locking. Two concurrent edits to the same schedule race;
the last write wins. Acceptable for typical single-operator
deployments. A future cycle could add an updated_at column
+ If-Match header for multi-operator setups." This is that
future cycle.

## Contract change

```go
type ScanSchedule struct {
    // ... existing fields ...
    UpdatedAt time.Time `json:"updated_at"` // v1.78+
}

type UpdateScheduleRequest struct {
    // ... existing fields ...
    IfMatch *time.Time `json:"-"` // v1.78+, header-driven
}

var ErrSchedulePreconditionFailed = errors.New(...)
```

REST:

```
PUT /api/v1/schedules/{id}
If-Match: 2026-05-10T19:00:00Z
Body: { ...UpdateScheduleRequest body... }

200 → updated schedule
412 → precondition failed (UpdatedAt mismatch)
400 → malformed If-Match header
404 → schedule not found
(missing If-Match header → no precondition check)
```

## cronIsDue / NextFire — unchanged

The optimistic lock only concerns the editable fields
(name, template, cadence, timezone). MarkFired and
SetEnabled don't bump UpdatedAt because they don't touch
those fields — concurrent fires/toggles shouldn't
false-positive the precondition.

## DBScheduleStore.Update

```sql
-- with If-Match:
UPDATE scan_schedules
SET name = $2, ..., updated_at = $10
WHERE id = $1 AND updated_at = $9
RETURNING <scheduleColumns>

-- without If-Match (back-compat):
UPDATE scan_schedules
SET name = $2, ..., updated_at = $9
WHERE id = $1
RETURNING <scheduleColumns>
```

When the conditional UPDATE returns 0 rows, a follow-up
`SELECT 1 FROM scan_schedules WHERE id = $1` disambiguates:
row exists → ErrSchedulePreconditionFailed; missing →
ErrScheduleNotFound. The extra round-trip is only on the
failure path (rare).

## Migration 00010

```sql
ALTER TABLE scan_schedules
    ADD COLUMN updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW();
UPDATE scan_schedules SET updated_at = created_at;
ALTER TABLE scan_schedules ALTER COLUMN updated_at DROP DEFAULT;
```

Backfill from created_at (pre-v1.78 schedules have never
been edited, so updated_at == created_at is the correct
logical value). DEFAULT dropped after backfill so v1.78+
INSERTs must set it explicitly.

## Persistence

- scheduleColumns 12 → 13 (updated_at slotted between
  created_at and last_fired_at).
- INSERT writes 12 → 13 args.
- UPDATE writes the column on both branches.
- scanSchedule reads &s.UpdatedAt in column order.

## Dashboard

- `editingScheduleUpdatedAt` module-scoped variable
  captures the schedule's updated_at at edit-load.
- `submitSchedule` sends `If-Match: <updated_at>` on PUT.
  412 surfaces a clear "schedule was modified by another
  operator — refresh and retry" error inline.
- `cancelEditSchedule` clears the captured stamp.

## Honest scope

  - No RFC 7232 ETag round-trip — custom UpdatedAt
    timestamp, not weak/strong ETag values.
  - No If-Unmodified-Since support.
  - No "merge view" UI on 412 — operator manually
    re-enters edits after refresh.
  - MarkFired / SetEnabled don't bump updated_at
    (intentional).
