# Snapshot — v1.8.0: FOFA + ZoomEye inputs (operator-requested)

Closed **2026-04-23**. Delivers the two attack-surface input
providers the operator requested immediately after v1.7.0
signing. Complements the existing Shodan + Censys clients.

## Chunks

### Chunk 1 — FOFA client (`da41262`)
- `internal/inputs/fofa/client.go` NEW. FOFA needs both
  `email` + `apiKey` (unlike Shodan's single key). `Search`
  base64-encodes the query per FOFA's `qbase64` convention,
  requests `fields=host,ip,port` for a stable row shape, maps
  rows to `core.Target{Address, Port}`. Unparseable IPs or
  out-of-range ports are dropped silently.
- `ErrNoCredentials` + `ErrAPIError` as typed sentinels so
  callers can distinguish auth / quota / bad-query errors
  from transport failures.
- 5 tests — happy path with IPv4/IPv6/invalid-IP/out-of-range-
  port rows, qbase64-decodes-back assertion, ErrAPIError path,
  non-200 path, default-size assertion.

### Chunk 2 — ZoomEye client (`315ad0c`)
- `internal/inputs/zoomeye/client.go` NEW. ZoomEye uses single
  API key delivered via `API-KEY` HTTP header (not query
  param), so credentials don't leak in URL logs. `Search(ctx,
  query, page)` with 1-based paging. JSON envelope has
  top-level `ip` + nested `portinfo.port`; the decoder
  mirrors that shape.
- `ErrNoAPIKey` sentinel.
- 5 tests — happy path + header-auth assertion + default-page +
  non-200 (402 Payment Required is the real-world symptom
  when free-tier credits are exhausted) + empty-matches.

## Scope decision — library-level only

Matches the existing Shodan + Censys pattern: both have
library clients with tests but are NOT wired into the `scan
--input <kind>` dispatch. FOFA + ZoomEye follow the same
precedent.

v1.9+ carry-over: design decision between
  (a) extending `--input` (e.g. `--input fofa:<query>
     --fofa-creds-file <path>`)
  (b) new verb (`elsereno search fofa --query <q>` → pipe to
     `scan --input stdin`)
  (c) vault integration (`elsereno creds store fofa` + auto-
     use from scan)

Option (c) is the most consistent with the rest of the
codebase (matches how the offensive vault works) but has the
widest surface; (a) is the quickest win.

## Release distribution (addendum 2026-04-23, post-publish)

v1.8.0 was **published as the first community release** at
https://github.com/RobinR00T/elSereno/releases/tag/v1.8.0.

Flow that shipped the artefacts (documented in RELEASING.md):

1. `git checkout v1.8.0` (detached HEAD on the tag commit)
2. `goreleaser release --clean --skip=publish,sign,docker,validate`
   → 4 tarballs (darwin/linux × amd64/arm64) + 4 CycloneDX
   SBOMs + checksums.txt under `dist/`.
3. `git checkout main` (back to working branch).
4. `gh release create v1.8.0 dist/* --title "v1.8.0 — first
   community release" --notes-file /tmp/release-notes.md`.

Why local-built: all 8 CI release workflow runs (v1.0.0 →
v1.8.0) had been aborting at ~30s with "The job was not
started because recent account payments have failed or your
spending limit needs to be increased" — GitHub Actions billing
was the blocker, not any code issue. Rather than top up
billing, the project moved to the free tier: `.github/workflows
/*.yml` gated to `workflow_dispatch:` only (triggers preserved
in comments for restoration).

What's intentionally missing vs. a CI-backed release:
- cosign keyless Sigstore signature on checksums.txt (needs
  GitHub Actions OIDC).
- SLSA v1.0 provenance attestations (needs
  `actions/attest-build-provenance@v2`).
- GHCR multi-arch docker image with cosign-signed manifest.

What IS there:
- GPG-signed tag with key ACE3B86BACACE7D6 / Daniel Solís Agea.
- SHA-256 checksums in `checksums.txt`.
- 4 CycloneDX 1.6 SBOMs via syft.
