### systemd unit hardening

Both units share a baseline that requires zero
capabilities:

```
NoNewPrivileges=true
RestrictSUIDSGID=true
PrivateDevices=true
ProtectSystem=strict
ProtectHome=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectKernelLogs=true
ProtectControlGroups=true
ProtectClock=true
ProtectHostname=true
RestrictNamespaces=true
RestrictRealtime=true
LockPersonality=true
MemoryDenyWriteExecute=true
CapabilityBoundingSet=
AmbientCapabilities=
SystemCallFilter=@system-service ~@mount @swap @reboot @debug @cpu-emulation
SystemCallArchitectures=native
```

Operator gets a defence-in-depth posture without any
config — the daemon runs in a sandbox even on hosts
where the operator hasn't tuned anything. Not a
substitute for the in-process seccomp filter (which
applies to harvest + dial subprocess profiles), but a
solid outer wrap.

### preinstall / postinstall behaviour

  - **preinstall**: creates `elsereno` system user +
    group via `useradd --system` (with `adduser`
    fallback for Alpine). Idempotent.
  - **postinstall**: applies the tmpfiles drop-in
    (creates /run/elsereno + /var/lib/elsereno +
    /var/log/elsereno with correct perms), runs
    `systemctl daemon-reload`, prints an operator
    quick-start MOTD. **Does NOT auto-enable** — the
    unit only starts when the operator explicitly says
    `systemctl enable --now`.
  - **preremove**: stops + disables both units.
    Persistent state preserved on `apt remove`; only
    `apt purge` should ever wipe it.

### State preservation on remove

`/var/lib/elsereno`, `/var/log/elsereno`,
`/etc/elsereno` are intentionally NOT in the package's
file list — they're created by the
postinstall+tmpfiles. The package owns the systemd
units and the config sample only. So:

  - `apt install elsereno`     → install + create dirs
  - `apt remove elsereno`      → uninstall, dirs survive
  - `apt purge elsereno`       → also wipes /etc/...
  - reinstall after remove     → resumes cleanly
                                  (audit chain + vault
                                  intact)

### Static linking proof

```
$ file dist/elsereno_v1.48.0-next_linux_amd64
… ELF 64-bit LSB executable, x86-64, version 1 (SYSV),
  statically linked, Go BuildID=…, stripped
$ go tool nm dist/elsereno | grep -E "libc\.so|libpthread\.so"
(empty)
```

No glibc dep means the binary runs on any Linux
distribution with kernel ≥ 2.6.32 (Go's default
minimum). The deb/rpm only specify `libcap2-bin` /
`libcap` as deps for operator convenience (so
`setcap` is available if they want to drop CAP_NET_BIND
later); the binary itself doesn't use them.

## Standing process change (v1.49+)

Per operator instruction, every cycle from here:

  1. Builds + verifies BOTH macOS + Linux artefacts
     (already done by goreleaser; ensure the matrix
     stays full).
  2. Updates INSTALL.md when:
     - any platform-specific feature changes,
     - install/upgrade/uninstall flow changes,
     - a new variant lands or one is retired,
     - a CLI verb gains/loses a flag worth documenting
       at install-doc level.
  3. Per-cycle commit messages explicitly note the
     impact on INSTALL.md (or "INSTALL.md unchanged" if
     no change).
