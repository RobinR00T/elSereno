# Snapshot — v1.91.0: pruner Prometheus metrics

Single-chunk cycle. Closes the v1.90 carryover: pruner
outcomes were observable only via stderr until now. v1.91
adds two Prometheus instruments so operators can graph the
v1.90 advisory-lock coordination from Grafana.

## Metrics

  - `elsereno_audit_pruner_runs_total{result}` — counter
    vector. `result` ∈ {"acquired", "skipped_lock",
    "error"}. Wired through the existing OnPrune /
    OnLockSkipped / OnError callbacks the AuditPruner
    already exposed.
  - `elsereno_audit_pruner_events_deleted_total` — counter,
    cumulative events deleted by the background pruner.
    Only incremented on `acquired` ticks (the v1.90 lock
    ensures no double-count across replicas).

Both registered in `telemetry.NewMetrics` alongside the
existing 5 instruments. Test: `TestAuditPrunerMetricsRegistered`
covers label values + handler smoke.

## cmd_serve wiring

`startAuditPruner` callbacks (OnPrune / OnError /
OnLockSkipped) now bump the relevant counter via
`telemetry.Global()`. stderr logging preserved alongside.

## Refactor

`NewMetrics` exceeded the 60-line funlen limit after the
additions. Extracted `buildCoreInstruments` +
`buildAuditPrunerInstruments` helpers so future cycles can
add instruments without re-tripping the linter.

## Why two instruments + a label, not three

A labelled counter is the canonical Prometheus pattern for
mutually-exclusive outcomes. Operators write one PromQL
query (`sum(rate(...[5m])) by (result)`) instead of three
separate `rate(...)` calls. Cardinality is 3 — bounded +
predictable.
